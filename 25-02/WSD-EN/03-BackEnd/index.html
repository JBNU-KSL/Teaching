1.Back-End Server Structure

A. Back-End Trend?

  • Recently, 'serverless' has become a trend in the back-end field, and it is one of the models of cloud computing, which is an architecture that does not require developers to manage servers directly.
  • If a backend developer expects 10,000 users on a server and buys the right amount of capacity, he or she will pay the same amount whether there are 10,000 or zero users. This situation can cause losses, large or small.
  • Serverless, however, dynamically allocates resources on the server. When there are no users, it waits without allocating resources and only when requests come in, it allocates resources to serve them.
  • Recently, there's been a lot of interest in server-side rendering (SSR)!
  • Server-Side Rendering (SSR) is a method of pre-rendering the content of a web page on the server and then delivering the finished HTML to the client. This approach is beneficial for SEO (search engine optimization) because it allows for faster first-page loading speeds, since the finished page is already visible on the screen before any JavaScript is executed on the client side.
  • SSR specifically improves SEO performance by making it easier for search engines to crawl your web pages. It also improves user experience because the initial screen is displayed quickly. However, it can increase the load on your server and may require more complex setup than client-side rendering (CSR).
  • Major frameworks include Next.js (React), Nuxt.js (Vue.js), and others.

B. Types and structure of different servers

* Servers are categorized by role and function, and the scope of a backend developer's responsibilities depends on which server they are working with.

  • Web Server (Web Server)
    • Receives HTTP requests and serves static content (HTML, CSS, JS).
    • Examples: Apache, Nginx
  • Application Server
    • Executes business logic and generates dynamic content
    • Examples: Node.js, Spring Boot, Django
  • Database Server (DB Server)
    • Data storage/retrieval, transaction management
    • Examples: MySQL, PostgreSQL, MongoDB
  • Cache Server (Cache Server)
    • Temporary storage of frequently requested data → faster response time
    • Examples: Redis, Memcached
  • Proxy Server
    • Relays requests between clients and servers, enhances security, and performs load balancing
  • Mail server, file server, authentication server, etc.
    • Servers that specialize in specific functions (SMTP, FTP, LDAP, etc.)

C. Types of Client Server Architecture

  • N-Tier Architecture: A structure in which servers are divided into multiple tiers, each responsible for different functions. Favorable for scalability and handling complex business logic.
    • 1-Tier Architecture: Client and server run on the same system. Mostly used for single-user applications.
    • 2-Tier Architecture: Traditional client-server architecture where the client and server are separated. The client acts as the front-end and the server as the back-end.
    • 3-Tier Architecture: Separate client, server, and database. The server handles client requests, while the database manages data.
  • Micro Service Architecture (MSA): Divides applications into small, independent services that can be deployed and scaled individually.
  • Serverless Architecture: An event-driven structure where the cloud automatically runs the infrastructure without server management and focuses on code execution.

(1) N-Tier Architecture

a. Client-Server Architecture (2-Tier)

  • Distributionof roles between client and server: Client-Server Architecture refers to a structure in which the client and server interact by performing different roles. The client creates a request from the user, and the server processes it and returns a response.
  • Theserver's function: The server is centrally responsible for data processing, executing business logic or interacting with the database based on requests received from the client. This allows clients to receive the data they need.
  • Communication protocols: The client and server exchange data using communication protocols, primarily HTTP and HTTPS. During this process, requests and responses are exchanged, and the client receives the results and displays them on the screen.

  • Scalability and maintainability: Client-Server Architecture has a modularized structure, which makes it easy to scale and maintain. The server is designed to handle multiple client requests and can be changed independently of the client and server.
  • Therole of the backend: The backend works on the server side and is responsible for data processing, storage, and integration with external services. When a client requests the data it needs, the backend processes it and returns the information to the client.
  • Multi-user support: Centralized servers provide the ability to handle simultaneous access from multiple clients, which allows multiple users to access the same resource.

b. Client-Server Architecture (3-Tier)

  • The 3-Tier Architecture isan extension of the Client-Server Architecture, which means that the application is divided into three independent tiers.
  • Each tier fulfills a specific role, and this separation contributes to making the system more maintainable, scalable, and secure.
  • Scalability: The ability to scale the server or database tiers independently, making it easier to increase performance when traffic increases.
  • Increasedsecurity: Because the database and business logic tiers are separated, the database is never accessed directly through the presentation tier, increasing security.
  • Reusability: The business logic layer and data layer can be reused across multiple client applications.

1) Presentation layer

  • Role: This layer is responsible for all processing related to the user interface (UI) and operates primarily on the client side. It handles the visual part of the screen or application that the user interacts with.
  • Main functions: Takes user input and processes it, forwards requests to the server, and outputs the response from the server to the screen.
  • Examples: A front-end application based on HTML, CSS, and JavaScript that runs in a web browser, or the UI portion of a mobile application.

2) Business logic layer

  • Role: The layer responsible for the core logic of the application, processing client requests and executing data processing rules and business logic. Usually runs on the server side.
  • Key functions: Receives client requests, interacts with the database, applies business rules according to the requests, and returns the results to the presentation layer.
  • Examples: Backend servers of web applications, API servers, application servers, etc. Typically written in programming languages such as Python, Java, C#, Node.js, etc.

3) The Persistence Layer

  • Role: This layer provides an abstracted connection between the application's business logic and data storage (DB, cache, file storage, etc.), bringing together data access code in one place for greater reusability and maintainability. Responsible for managing transaction boundaries, object-relationship mapping (ORM), and exception transformation.
  • Key features: CRUD operation creation/retrieval/modification/deletion abstraction (DAO/Repository), entity ↔ table (or document/key-value) mapping, query building/execution, paging/sorting, transaction management (commit/rollback), lock (optimistic/pessimistic) handling.
  • Example: Repository/DAO/Mapper pattern, Unit of Work, DataSource/Connection Pool (HikariCP, etc.)

4) Database Layer

  • Role: The layer that stores and manages data, where the database that permanently stores data is located.
  • Main functions: Storing or retrieving data as requested by the business logic layer, and managing data securely and efficiently.
  • Examples: Relational or non-relational databases such as MySQL, PostgreSQL, and MongoDB make up this layer.

(2) Micro Service Architecture (MSA)

  • Concept: A microservice architecture is a way to decompose an application into small service units that can be deployed, scaled, and maintained independently. Unlike traditional monolithic architectures, dependencies between services are reduced.
  • Features: Each microservice can have its own database and can be developed, deployed, and scaled independently, focusing on specific business functions. This maximizes the flexibility and scalability of the system.
  • Pros: Scales on a per-service basis, and failures only affect parts of the system, not the entire system. Independent teams can develop individual services, allowing for efficient collaboration.
  • Disadvantages: Communication between services can be complex, and each service requires infrastructure configuration and maintenance for deployment and management.

(3) Serverless Architecture

  • Concept: Serverless architecture is a way for cloud service providers to operate and scale infrastructure without developers managing servers themselves. Developers can focus on business logic and are relieved of the burden of server operations.
  • Key services: AWS Lambda, Google Cloud Functions, and Azure Functions are the main cloud services that support serverless architecture.
  • Benefits: Auto-scaling, cost efficiency, and reduced management complexity are key benefits. Can reduce idle costs by automatically adjusting resources based on application usage.
  • Cons: Server uptime is limited and may not be ideal for handling complex tasks on the backend. It also creates a dependency on the cloud provider.
