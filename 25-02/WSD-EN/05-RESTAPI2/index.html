A. Request

1) Overall Structure of an HTTP Request

<Request Line>                # e.g., POST /users?active=true HTTP/1.1
<Headers...>                  # e.g., Host: api.example.com
                              #     Authorization: Bearer <token>
                              #     Content-Type: application/json; charset=utf-8
                              #     Content-Length: 27
<Empty Line (separates headers and body)>
<Body>                        # e.g., {"name":"Lee","role":"admin"}
  • Specific Example
POST /todos HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer <JWT>
Content-Length: 55

{
  "title": "Do programming assignment",
  "dueDate": "2025-10-05"
}

2) Request Headers

  • Request Identification / Transmission
    HEAD
POST /upload HTTP/1.1
Host: api.example.com
User-Agent: MyUploader/2.0 (macOS 14; x64)
Date: Tue, 30 Sep 2025 12:34:56 GMT
Connection: keep-alive
Transfer-Encoding: chunked
Expect: 100-continue
TE: trailers
Via: 1.1 cache-01.example.net, 1.1 lb-1
Forwarded: for=203.0.113.7;proto=https;by=lb-1
X-Forwarded-For: 203.0.113.7
X-Forwarded-Proto: https
Content-Type: application/octet-stream
    • Host: Required in HTTP/1.1. Differentiates virtual hosts (e.g., Host: api.example.com).
    • User-Agent: Client information (version, platform, etc.). Useful for logging/analytics.
    • Date: Timestamp of message creation (can be in request/response, more common in response).
    • Connection: Connection options for HTTP/1.1 only (e.g., keep-alive). Not used in HTTP/2.
    • Transfer-Encoding: Message framing (e.g., chunked, HTTP/1.1). Must not be used with Content-Length.
    • Expect: 100-continue: Before uploading a large body, asks the server if it's okay to proceed.
    • TE: Transfer coding negotiation (rare in requests).
    • Via, Forwarded / X-Forwarded-For, X-Forwarded-Proto: Information about passing through proxies/load balancers.

  • Content (Body) Description
    • Content-Type: MIME type of the body (+ encoding).
      • application/json; charset=utf-8,
      • multipart/form-data; boundary=----abc,
      • application/x-www-form-urlencoded.
    • Content-Length: Length of the body in bytes (not character count).
    • Content-Encoding: Body compression, etc. (e.g., gzip). Can be used for request bodies, but the server must support it.
    • Content-Language: Language of the body (e.g., ko).
    • Content-MD5 (legacy): Was used for integrity checks, but is not recommended in the standard. Use signing/hashing schemes instead.
    • JSON + Language specified (most common case)
    HEAD
POST /comments HTTP/1.1
Host: api.example.com
Content-Type: application/json; charset=utf-8
Content-Language: ko
Accept: application/json
Content-Length: 87   # example value

BODY
{
  "postId": 1001,
  "author": "Lee",
  "text": "This was really helpful!",
  "tags": ["rest","http"]
}
    • JSON + Request Body Compression (Content-Encoding: gzip)
    HEAD
POST /bulk/users HTTP/1.1

Content-Type: application/json
Content-Encoding: gzip
Content-Language: ko
Accept: application/json
Content-MD5: qUqP5cyxm6YcTAhz05Hph5==   # example(base64) — not recommended by standard
Content-Length: 314   # example value (byte length of the compressed body)

BODY
<gzip-compressed JSON byte stream>

  • Content Negotiation (Client Preferences)
    • Accept: Preferred media type for the response (e.g., application/json).
    • Accept-Language: Preferred language for the response (e.g., ko, en;q=0.8).
    • Accept-Charset: Preferred character set (rarely used, JSON is UTF-8 by default).
    • Accept-Encoding: Preferred compression for the response (e.g., gzip, br).
    • API Query — Prefer JSON, Korean language, allow gzip/br compression
    HEAD
GET /v1/todos?status=active HTTP/1.1
Host: api.example.com
Accept: application/json
Accept-Language: ko, en;q=0.8
Accept-Charset: utf-8
Accept-Encoding: gzip, br
    • Report Download — Prefer CSV, also accept JSON, indicate character set preference
    HEAD
GET /v1/reports/2025-09 HTTP/1.1
Host: api.example.com
Accept: text/csv; q=1.0, application/json; q=0.8
Accept-Language: ko-KR, en-US;q=0.6
Accept-Charset: utf-8, iso-8859-1;q=0.2, *;q=0.1
Accept-Encoding: gzip, br
    • Image Thumbnail — Prefer modern formats (avif/webp), any language is OK
    HEAD
GET /images/thumbnail/12345 HTTP/1.1
Host: cdn.example.com
Accept: image/avif, image/webp, image/*;q=0.8, */*;q=0.5
Accept-Language: * # Language-independent resource, so no restrictions
Accept-Charset: utf-8
Accept-Encoding: gzip, br
    • Document Preview — Prefer PDF, compress if possible (identity if not)
    HEAD
GET /docs/preview/987.pdf HTTP/1.1
Host: files.example.com
Accept: application/pdf, application/octet-stream;q=0.5
Accept-Language: ko, en;q=0.7
Accept-Charset: utf-8
Accept-Encoding: gzip, identity;q=0.1

  • Authentication/Authorization/Session
    • Authorization: Authentication credentials.

      e.g., Bearer <JWT>, Basic <base64>, Digest ..., HMAC signature, etc.

    • Proxy-Authorization: Proxy authentication.
    • Cookie: Transmits session/state (e.g., Cookie: sid=abc; theme=dark). Avoid placing sensitive information in cookies/URLs; use a token header instead.
    • Bearer Token (JWT) Authentication
    HEAD
GET /v1/me HTTP/1.1
Host: api.example.com
Authorization: Bearer <JWT>
Accept: application/json
    • Basic Authentication
    HEAD
GET /admin HTTP/1.1
Host: api.example.com
Authorization: Basic <base64(username:password)>
Accept: text/html
    • Proxy-Authorization
    HEAD
GET <http://intranet.example.com/docs> HTTP/1.1
Host: intranet.example.com
Proxy-Authorization: Basic <base64(proxyUser:proxyPass)>
Accept: text/html
    • Cookie Session (State Maintenance)
    HEAD
GET /cart HTTP/1.1
Host: shop.example.com
Cookie: sid=abc123; theme=dark
Accept: application/json

3) Request Body

  • The request body is a container for the actual data of a resource!
  • "When" to use a body?
    • Generally used with POST/PUT/PATCH.
    • It's possible to send a body with GET/DELETE (not forbidden by the protocol), but it's not recommended as servers often ignore it.
    • HEAD/TRACE conventionally do not have a body.
    • POST — Resource Creation

Content-Type: application/json
Accept: application/json

  "dueDate": "2025-10-05"
}

B. Response

1) Overall Structure of an HTTP Response

<Status Line>                # e.g., HTTP/1.1 200 OK
<Headers...>                 # e.g., Content-Type: application/json; charset=utf-8
                             #     Content-Length: 27
<Empty Line>
<Body>                       # e.g., {"id":101,"title":"..."}
  • Status Line: Protocol Version + Status Code + Reason Phrase
  • Headers: Response metadata (content type, cache, authentication challenges, location, etc.)
  • Body: The actual data (JSON, file, HTML...). May be absent depending on the code/method (e.g., 204 No Content, 304 Not Modified)

2) Status Code Summary

  • 2xx Success: 200 OK, 201 Created, 202 Accepted, 204 No Content, 206 Partial Content
  • 3xx Redirection: 301/302/303/307/308 + Location
  • 4xx Client Error: 400, 401 + WWW-Authenticate, 403, 404, 409, 412, 415, 429 …
  • 5xx Server Error: 500, 502, 503(+ Retry-After), 504 …

3) Frequently Used Response Headers

  • Content/Representation
    • Content-Type, Content-Length, Content-Encoding, Content-Language
  • Caching/Conditional
    • ETag, Last-Modified, Cache-Control, Expires, Vary
  • Navigation
    • Location(creation/redirect), Link(pagination, etc.),Content-Range/Accept-Ranges
  • Authentication/Session
    • WWW-Authenticate(for 401), Set-Cookie, Clear-Site-Data
  • Rate Limiting (Conventional)
    • RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset or X-RateLimit-*
  • CORS
    • Access-Control-Allow-*, Access-Control-Expose-Headers
  • Reliability/Directives
    • Retry-After(for 429/503/202), Preference-Applied(response to Prefer)

14) Best Practices!

  • Content Contract: Always use the correct Content-Type (response format), and charset=utf-8 when necessary.
  • Caching/Conditionals: Whenever possible, issue an ETag or Last-Modified → allows clients to re-validate with If-* headers.
  • Resource Creation: Use 201 Created with an accurate Location header.
  • Consistent Error Format: Use application/problem+json or a team-standard error schema.
  • Rate Limiting: With 429, provide remaining limits/retry timings via headers.
  • CORS: Allow only the necessary minimum origins, expose headers that clients need to read with Expose-Headers.
  • Security: Session cookies should be Secure/HttpOnly/SameSite; sensitive information should only be in the body.

15) Response Format Standardization

15.1 Specification

  • Envelope(recommended): Wrap all successful responses in the common format below.
    • data: The actual payload (can be an object/array/primitive).
    • meta: Additional information (paging, trace ID, processing time, etc.).
    • links: HATEOAS/navigation links (optional).
  • Time/Numbers
    • Time should be UTC ISO-8601 (YYYY-MM-DDTHH:mm:ssZ).
    • Monetary values as integer in the smallest unit (e.g., KRW in won, USD in cents) + specify currency code.
  • Naming: camelCase (recommended) or team standard. Maintain consistency throughout the documentation.
  • Null/Empty: Differentiate between non-existent (null) and empty values (""/[]).
  • Sorting: Specify a default sort order for lists (e.g., createdAt).

  • Include data in all success responses (empty results as data: [] or data: null).

  • For large collections, apply a default limit/perPage and set an upper bound.

16) Error Format Standardization (Problem Details)

16.1 Specification

  • Use the standard media type application/problem+json (RFC 9457).
  • Common fields:
    • type (a URL or "about:blank"), title, status, detail, instance.
  • Extension fields:
    • errors (map of field-specific errors), requestId, hint, docs.

16.2 Validation Error (400) Example

HEAD
HTTP/1.1 400 Bad Request
Content-Type: application/problem+json

BODY
{
  "type": "<https://example.com/problems/validation-error>",
  "title": "Invalid Request",
  "status": 400,
  "detail": "One or more fields are invalid.",
  "errors": {
    "dueDate": ["must be an ISO-8601 date"]
  },
  "requestId": "8d1d8a30-2a2b-4a1a-9e4e-1dbb6a3d7c01",
  "instance": "/todos"
}

16.3 Authentication Error (401) Example

HEAD
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="api", error="invalid_token", error_description="expired"
Content-Type: application/problem+json

BODY
{
  "type": "<https://example.com/problems/auth>",
  "title": "Unauthorized",
  "status": 401,
  "detail": "Access token expired.",
  "requestId": "8d1d8a30-2a2b-4a1a-9e4e-1dbb6a3d7c01"

16.4 Concurrency Conflict (412) Example

HEAD
HTTP/1.1 412 Precondition Failed
ETag: "b13c-77df29"
Content-Type: application/problem+json

BODY
{
  "type": "<https://example.com/problems/precondition-failed>",
  "title": "Precondition Failed",

16.5 Implementation Checklist

  • All non-2xx responses use application/problem+json.
  • Maintain an error code mapping table (HTTP ↔ domain codes).
  • Provide field errors as errors.{field}:[…] array.
  • Include requestId for linking logs/traces.

17) Pagination

17.1 Specification (Choose one, do not mix)

(A) Query Parameter Page-based

  • Request: ?page=2&perPage=50&sort=-createdAt
  • Response: data array + meta.total/page/perPage + links.self/next/prev
  • Pros: Simple/intuitive.
  • Cons: Cost of calculating dynamic total.

Request/Response Example

HEAD
GET /todos?page=2&perPage=50&sort=-createdAt HTTP/1.1
Host: api.example.com
Accept: application/json

BODY
# None


HEAD
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
Link: <https://api.example.com/todos?page=3&perPage=50>; rel="next",
      <https://api.example.com/todos?page=1&perPage=50>; rel="prev"

BODY
{
  "data": [ /* 50 items */ ],
  "meta": {
    "total": 240,
    "page": 2,
    "perPage": 50
  },
  "links": {
    "self": "<https://api.example.com/todos?page=2&perPage=50>",
    "next": "<https://api.example.com/todos?page=3&perPage=50>",
    "prev": "<https://api.example.com/todos?page=1&perPage=50>"
  }
}

(B) Cursor-based (cursor/token)

  • Request: ?limit=50&cursor=<opaqueToken> (no cursor for the first page)
  • Response: meta.nextCursor (if present, there is a next page), meta.hasMore (optional)
  • Pros: Robust for large-scale/real-time changing data, stable scanning.
  • Cons: Higher implementation complexity, difficult to access random pages.

Request/Response Example

HEAD
GET /todos?limit=50&cursor=eyJpZCI6MTAxLCJjcmVhdGVkQXQiOiIyMDI1LTA5LTMwVDEyOjM0OjU2WiJ9 HTTP/1.1
Host: api.example.com
Accept: application/json

BODY
# None

HEAD
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8

BODY
{
  "data": [ /* 50 items */ ],
  "meta": {
    "limit": 50,
    "nextCursor": "eyJpZCI6MTAyLCJjcmVhdGVkQXQiOiIyMDI1LTA5LTMwVDEyOjM1OjEwWiJ9",
    "hasMore": true
  },
  "links": {
    "self": "<https://api.example.com/todos?limit=50&cursor=>...",
    "next": "<https://api.example.com/todos?limit=50&cursor=eyJpZCI6MTAy>..."
  }
}

17.2 Range-Unit based (Alternative)

  • Request Header:
    HEAD
GET /todos HTTP/1.1
Host: api.example.com
Range-Unit: items
Range: items=0-49
  • Response Header: 206 Partial Content + Content-Range: items 0-49/240

17.3 Implementation Checklist

  • Set a default perPage/limit and a maximum upper bound (e.g., default 50, max 200).
  • Define conventions for sorting fields/directions (e.g., sort=-createdAt).
  • For page-based, document if total calculation is expensive, consider deferred calculation or estimates.
  • For cursor-based, make nextCursor an opaque token (do not expose internal schema).
  • Provide both Link header and links object simultaneously (client's choice).
  • Empty result: data: [], meta.total: 0 or hasMore: false.
