A. REST API Overview

1. Network and Web Basics

1-1. What is a network?

  • Network: Refers to a structure in which multiple computers are connected so that they can exchange data with each other. There are wired (cable) and wireless (Wi-Fi, etc.) connections, and all connections are made according to a common protocol.
  • Internet: A very large network of networks from all over the world, the Internet is not a single system, but a collection of many smaller networks connected together using a standard protocol (TCP/IP).
When we access a website, we are actually communicating with a browser (client)that sends a request to the serverand the server sends back a response!

1-2. Client and Server

  • Client:
    • The entity that requests a service.
    • Usually a program that is directly operated by the user (e.g., a web browser or mobile app).
    • The client is only responsible for generating and sending requests to the server,not actually storing data or performing complex computations.
  • Server:
    • An entity that processes requests sent by a client and responds with the results.
    • A web server serves static pages or API responses, and a database server stores, retrieves, and updates data.
    • Servers typically receive requests from multiple clients simultaneously, requiringstrong parallelism and resource management.

  • Real-life analogy:

  • Client = a customer in a restaurant (the entity requesting service)
  • Server = Chef (the entity that receives the request and performs the cooking and processing)
  • The flow can be understood asorder (request) → cooking (processing) → food (response), and just as the customer does not do the cooking process himself, the client only receives the result without knowing the internal processing.

1-3. HTTP Protocol

  • HTTP (HyperText Transfer Protocol): The predominant convention for sending and receiving data on the web. It allows both requests and responses to be made in the form of text messagesand is the basic unit of communication between client and server.
    • Arequestspecifies what action to perform (e.g., get, create)and the target resource.
    • Response contains the result of the processing, along with a status code and, if necessary, data.

  • Example HTTP requests
    GET /students/1 HTTP/1.1
Host: example.com
    • GET: A request method that tells the server to retrieve data. Other methods include POST, PUT, and DELETE.
    • /students/1: The URL path to the server resource, which in this case means we're requesting student data with ID 1.
    • HTTP/1.1: The HTTP version being used. Currently, 1.1 and 2 are widely used, and the newest version is HTTP/3.
    • Host: example.com: Allows you to specify the hostname of the server being requested. You can run multiple domains on the same server, so this must be included!

  • Example HTTP response:
    HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 1,
  "name": "홍길동"
}
    • HTTP/1.1 200 OK: Response version and status code, where 200 OKmeans the request was successfully processed.
    • Content-Type: application/json: Specifies the data type of the response body. Here we return data in JSON format.
    • { "id": 1, "name": "HongGilDong" }: The actual response data, which is the student information requested by the client in the form of a JSON object.

  • As shown above, HTTP has a simple structure of exchanging request and response messages, and REST API communication can be seen as an extension of this basic principle...!

2. What is an API?

  • API (Application Programming Interface): Refers to an application (program) and the promise (interface)that allows the application to interact with each other. In other words, an API is a set of rules and specifications that define how certain functions can be called and used.
  • With APIs, programmers don't need to know the intricate inner workings, but they can leverage functionality bysending requests and receiving responses in a prescribed way. For example, operating systems provide file open/save APIs, and web services provide data retrieval/registration APIs.

2-1. Examples of everyday analogies

  • Cafe ordering analogy:
    • A customer says "Americano, please" in a cafe, and the waiter understands the order based on the menu (rules), makes the coffee, and serves it to the customer.
    • Thecustomer doesn't need to know how the coffee machine works or where the beans come from. They just need to order the way it's written on the menu.
    • In this case, the customer = client, the clerk = server, and the menu = API.
  • APIs work the same way as above! :
    • The client sends a request, following the rules defined in the API documentation.
    • The server processes the request and returns a response in a prescribed format.
    • Requests that deviate from the rules are not processed and may result in an error. (Example: ordering an invalid menu → response like "We don't have that menu")
  • In summary, APIscan act as a formal language and menu board for communication between programs, allowing developers to easily access the functionality they need without having to know the inner workings.

2-2. The Role of Web APIs

  • Access to resources in servers and databases
    • The information in the database is not available to everyone, so it can only be accessed on demand.
    • APIs allow only authorized people to access the server and database.
  • Standardize all requests and responses
    • It doesn't matter if you're using an iPhone or a Galaxy phone, you're using the same API, so you can standardize client requests and server responses with one API.
  • REST APIs
    • REST API (Representational State Transfer) is a way to represent different resources used on the web as HTTP URIs and define actions onthose resources as HTTP Methods.
    • In other words, it is a way to express how to use a resource (defined as an HTTP URI) in a structurally clean way (HTTP Method + Payload).

2-3. What is REST API?

  • REST (Representational State Transfer): A style of web architecture proposed in Roy Fielding's paper in 2000. Not just a protocol, but a set of design rules for how resources are defined and accessed on the web.
  • REST API: An API that closely follows the constraints of REST. The client manipulates a resource by making a request, and the server returns the result in a response.
  • The core constraints of a REST API:
    • Identification of resources (based on URIs): Every resource is represented by a unique URI (Uniform Resource Identifier), e.g. /students/1 → represents a resource called students with ID 1.
    • Manipulating resources through representations: You send and receive representations (JSON, XML, etc.) of resources, not the resources themselves. The same resource can respond in different representations depending on the context.
    • Stateless: The server does not store the client's state. Each request is handled independently, and all necessary information must be contained within the request message.
    • Uniform Interface: Standardized request and response formats using HTTP methods and status codes.
      • GET: Get a resource
      • POST: Create a resource
      • PUT: Modify an entire resource
      • PATCH: Modify a part of a resource
      • DELETE: Delete a resource
    • Layered System: The client does not need to know about the existence of intermediate servers (proxies, load balancers, etc.), and requests should behave the same even if they go through multiple layers.
    • Cacheable: Response results should be cacheable, which reduces server load and improves performance.
  • Features of REST APIs
    • Uniform Interface
      • Available on platforms that follow the HTTP standard protocol
      • Performs operations on resources specified by URIs with a uniform, constrained interface
      • Handles resources through HTTP methods (GET, POST, PUT, DELETE)
    • Stateless
      • The server does not store the client's state information
      • Sends all necessary information in each request
      • The server recognizes and processes each request as a completely separate entity
    • Cacheable
      • Can apply HTTP's caching features
      • Cache can be used to improve response time and reduce server load
      • GET requests are cacheable, but POST requests are not.
    • Layered System
      • The client does not know whether it is connected to the server directly or through an intermediate server
      • Intermediaries such as proxies and gateways can be used in between
      • Can add security, load balancing, encryption, user authentication, etc.
    • Self-descriptiveness
      • Self-descriptive structure that can be understood by looking at REST API messages alone
      • Use JSON to format messages
      • Understand what information is being provided by just looking at the URI
    • Client-Server Architecture
      • Client and server are separate and independent
      • Server provides API, client manages user authentication or context (session, login information) directly
      • Less dependency on each other

  • Advantages of REST APIs:
    • Intuitive and simple, no need to build a separate infrastructure
    • Because they use the HTTP protocol, you don't need to build your own infrastructure to use REST APIs.
    • Can be used on any platform that follows the HTTP standard protocol
    • Allows for clear separation of server and client roles
    • REST API messages clearly indicate what they are intended to do, making intent easy to understand
    • Minimizes issues that can arise from designing different services
    • Allows servers and clients to evolve independently

  • Disadvantages of REST APIs:
    • No standard exists, requiring definition
    • Limited number of HTTP methods (GET, POST, PUT, DELETE, etc.)
    • PUT and DELETE may not be available on older browsers
    • Related tools may be lacking
    • APIs you build require documentation and can be difficult to manage
    • Clients may need multiple requests to retrieve the exact data they need
    • Designing RESTful APIs can be challenging (API versioning, naming, etc.)
    • Limited ways to handle large data transfers
    • Can be difficult to manage due to lack of standards

To summarize, a REST API is an API that defines resources on the web as URIs and interacts with them in a standardized way through HTTP methods, and is the most popular form of web API today.

B. REST API

1. Components of REST API

1-1. Resource

  • The object thatthe API deals with.
    • Example: A noun-like objectsuch as a student, post, or order.
  • A resource is identifiedby a Uniform Resource Identifier (URI).
    • Examples: /students, /students/1, /orders/2024-09-01
  • Requires acollection/individual resource distinction:
    • /posts(collection) ↔ /posts/10(individual).
    • Sub-relationships should be expressed hierarchically , such as /students/1/courses.
  • URIs only represent objects, not actions(use nouns). Lowercase-plural-hyphenated is recommended.
    • Example: /createStudent(avoided) → /students(preferred).
  • Separatelookup conditions, paging, and sortinginto query parameters.
    • Example: /students?major=cs&page=2&size=20&sort=name,asc.
  • The server keeps the state of the resource, and the client sends and receives a representation of the state.

1-2. Method

  • HTTP methodsthat indicate what to do witha resource !
    • GET: Get. Safe andidempotent. No body, just headers-query semantics.
    • POST: Create or add a child resource to a collection . Idempotent. Usually returns 201 Created and a Location header on success.
    • PUT: Full replacement(representative case). Equal. Missing fields may be considered deletions.
    • PATCH: Partial modification. Usually non-equal (can be equal by design). The scope of the modification is specified in the body.
    • DELETE: Delete. Mutable. Resubmitting a request for a resource that has already been deleted should have the same result.
  • Note Methods: HEAD(header only, no body), OPTIONS(check which methods the server supports).
  • Representation status code depending on the method:GET → 200 OK / 404 Not Found if not presentPOST → 201 Created(+ Location)PUT/PATCH → 200 OK or 204 No Content if no bodyDELETE → 204 No Content

1-3. Representation

  • This refers to the format in which the state of the resource is communicated.
  • The body format is mainly JSON, but in some cases, XML/CSV/binary, etc. can be used.
  • Content Negotiation: Client indicates desired format with Accept, server declares actual format with Content-Type. Example: Accept: application/json, Content-Type: application/json.
  • The representation should be self-descriptive. Field semantics are clear and can include any necessary links (e.g., self, next) or metadata (e.g., page info).
  • Errors should preferably be returned in a consistent format. Example: Suggestions for configuring the status, code, message, and details fields.

1-4. REST API request/response structure

  • Request = Method + URI + Headers + Body (optional)
    • Auth-Authorization: Use Authorization header (JWT/Bearer, etc.).
    • Content-Type required when sending body. Client acceptance type specified as Accept.
    • Cache-Conditional requests: Utilize If-None-Match(ETag), If-Modified-Since.
  • Response = Status Code + Headers + Body (optional)
    • Success: 200/201/204 centered. Providing a new resource URI with Locationon creation.
    • Errors: 4xx(client error), 5xx(server error). Clearly describe the cause and fix in the body, if possible.
    • Cache-versioning: use ETags, Cache-Control, path-versioning(/v1) or header-versioning strategies as needed.

3. How REST APIs are designed

  • Actions on resources are expressed as HTTP Methods (POST, GET, PUT, DELETE).
  • Use / (forward slash) to indicate hierarchical relationships
  • Do not use / (slash) in the last character of a URI
  • Avoid using _ (underscore) in URIs. We also recommend using lowercase letters rather than English capitalization.
  • And avoid long words for readability.
  • Use nouns in URIs, not verbs, because verbs express HTTP methods like GET and POST.
  • Do not include file extensions (e.g. .json , .JPGE) in the URI.
  • Self-descriptiveness: RESTful APIs are designed so that it is easy to understand what the purpose of the API is from the name itself.
