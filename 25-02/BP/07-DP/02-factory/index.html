07-02. 팩토리 메서드 패턴 (Factory Method Pattern)

1. 문제 상황
  • 프로그래머가 특정 객체를 생성해야 하는 상황에서, 매번 new 키워드로 직접 인스턴스를 만들어 사용한다고 가정해보자.
    • 단순한 경우에는 문제 없음
    • 하지만 다양한 타입의 객체를 조건에 따라 생성해야 한다면?
      • 예: 버튼 UI 컴포넌트를 OS 환경에 맞게 생성해야 함
      • Windows에서는 WindowsButton 객체, Mac에서는 MacButton 객체를 생성해야 함
  • 이때 객체 생성 로직이 코드 곳곳에 흩어져 있다면 유지보수 어려움 발생!
// 공통 인터페이스 (겉보기엔 좋아 보임)
interface Button {
  render(): void;
}

// 구체 구현들
class WindowsButton implements Button {
  render() { console.log("Rendering Windows button"); }
}
class MacButton implements Button {
  render() { console.log("Rendering Mac button"); }
}

// 1) 화면 A - OS 분기 + 직접 생성
function drawToolbar(os: "Windows" | "Mac") {
  let saveButton: Button;
  if (os === "Windows") {
    saveButton = new WindowsButton();
  } else {
    saveButton = new MacButton();
  }
  saveButton.render();
}

// 2) 화면 B - 또다시 OS 분기 + 직접 생성 (중복)
function drawSidebar(os: "Windows" | "Mac") {
  const buttons: Button[] = [];
  if (os === "Windows") {
    buttons.push(new WindowsButton(), new WindowsButton());
  } else {
    buttons.push(new MacButton(), new MacButton());
  }
  buttons.forEach(b => b.render());
}

// 3) 다이얼로그 - 또 또 OS 분기 + 직접 생성 (더 많은 중복)
function showConfirm(os: "Windows" | "Mac") {
  let ok: Button, cancel: Button;

  if (os === "Windows") {
    ok = new WindowsButton();
    cancel = new WindowsButton();
  } else {
    ok = new MacButton();
    cancel = new MacButton();
  }

  ok.render();
  cancel.render();
}

// 4) 앱 시작부 - 각 화면에서 OS 분기가 반복 호출됨
function main() {
  // 실제로는 프로세스/UA/환경변수 등으로 결정되겠지만 여기선 하드코딩
  const os: "Windows" | "Mac" = "Windows";

  drawToolbar(os);
  drawSidebar(os);
  showConfirm(os);
}

main();

왜 문제인가?

  • 생성 로직이 중복: drawToolbar, drawSidebar, showConfirm 등 여러 곳에서 if (os) new ...가 반복되고 있음.
  • 확장에 취약(OCP 위반): 새로운 OS(예: LinuxButton)가 추가되면 모든 화면 함수를 찾아 OS 분기와 new를 수정해야 함.
    • OCP (Open-Closed Principle): 기존의 코드를 변경하지 않으면서, 기능을 추가할 수 있도록 설계가 되어야 한다는 원칙
  • 버그 유발 가능성 증가: 한곳만 수정하고 다른 곳을 놓치면 OS별 UI가 섞이거나 런타임 에러 발생.
  • 테스트 어려움: 생성 로직이 흩어져 있어 모킹/대체 주입이 힘들고 테스트 코드가 중복됨.

2. 패턴 개요

1) 정의(Definition)

  • 상위 타입(Creator)이 “제품을 어떻게 사용할지”에 대한 알고리즘(예: someOperation)은 갖되, 무엇을 만들지는 모른 채 createProduct()라는 공장 메서드만 선언하고, 구체적인 생성 결정은 하위 클래스(ConcreteCreator)가 createProduct()를 오버라이드하여 담당하게 만드는 객체지향 패턴.
  • createProduct()의 반환형은 제품 인터페이스/추상클래스(Product)이며, 실제 인스턴스화(new)는 콘크리트 크리에이터 내부로 국지화된다는 특징을 지니고 있음.
  • 결과적으로 클라이언트와 상위 로직은 Product 인터페이스에만 의존하고, 구체 제품 클래스/생성 방식과는 분리된다는 특징을 지니고 있음.
  • 구분해야 할 것
    • Simple Factory: 하나의 정적 메서드가 분기문으로 제품을 만드는 “유틸성 공장”에 가깝고, 상속에 의한 다형적 생성 확장 포인트가 없음.
    • Abstract Factory: “서로 호환되는 제품군을 묶음으로 생성”하는 상위 개념. Factory Method는 그 내부에서 각 제품을 만드는 방법으로 자주 쓰임.

2) 목적(Purpose)

  • 생성 책임 분리 & 캡슐화
    • 객체 생성 로직을 상위 알고리즘과 분리해, 생성 변경이 사용 코드에 파급되지 않도록 할 수 있음.
  • 확장에는 열려 있고 변경에는 닫힌 구조(OCP)
    • 새로운 제품(ConcreteProduct)을 추가할 때 기존 상위/클라이언트 코드를 수정하지 않고, 새로운 ConcreteCreator만 추가하여 확장할 수 있음.
  • DIP(의존성 역전) 준수 & 결합도 감소
    • 고수준 모듈(비즈니스 로직)이 저수준 구현(구체 제품)에 직접 의존하지 않고, Product 추상화에 의존.
  • 테스트 용이성 향상
    • 테스트 시 테스트용 Creator/Mock Product를 주입해 동작만 검증할 수 있음 (생성 부작용/무거운 의존성 제거).
  • 제품 선택의 유연성
    • 런타임 구성(설정/DI/환경)에 따라 다른 Creator를 선택해 동적으로 제품을 교체할 수 있음.
  • 재사용 가능한 상위 알고리즘 확보
    • someOperation() 같은 상위 템플릿은 변하지 않고, 생성 훅만 바꿔 다양한 변형을 재사용 가능(Template Method와의 친연성).

3) 특징(Characteristics)

  • 역할 구조
    • Product: 공통 인터페이스(행동 계약). 예: doStuff().
    • ConcreteProductX: 실제 제품 구현.
    • Creator(추상/기본 클래스): 제품 사용 알고리즘 보유, createProduct()라는 훅(hook) 제공.
    • ConcreteCreatorX: 훅을 오버라이드하여 구체 생성 담당.
  • 동적 바인딩에 의한 생성 결정
    • 상위 메서드 흐름(someOperation) 속에서 호출되는 createProduct()가 런타임에 선택된 하위 클래스 구현으로 연결되어, “무엇을 만들지”가 다형적으로 결정된다.
  • new의 위치가 명확
    • new는 오직 ConcreteCreator 내부에 존재. 클라이언트/상위 로직/상위 Creator에는 등장하지 않는다(생성 책임의 단일화).
  • 변형/확장 포인트
    • 파라메트릭 팩터리 메서드: createProduct(type, config)처럼 매개변수로 추가 분기를 줄 수 있으나, 분기가 과도하면 Simple Factory로 퇴화할 수 있음.
    • 기본 구현 제공: Creator가 기본 createProduct()를 제공하고 일부만 재정의하도록 할 수도 있음.
    • 캐싱/풀링/싱글턴 결합: 생성 비용이 큰 제품을 캐시/풀로 돌리거나 싱글턴 제품을 반환하도록 Creator에서 정책화 가능.

3. TypeScript 코드 예제

// 1. Product 인터페이스 (버튼의 공통 인터페이스)
interface Button {
  render(): void
  onClick(): void
}

// 2. 구체적인 Product 클래스들
class WindowsButton implements Button {
  render(): void {
    console.log("윈도우 스타일 버튼 렌더링")
  }
  onClick(): void {
    console.log("윈도우 버튼 클릭 이벤트 처리")
  }
}

class MacButton implements Button {
  render(): void {
    console.log("맥 스타일 버튼 렌더링")
  }
  onClick(): void {
    console.log("맥 버튼 클릭 이벤트 처리")
  }
}

// 3. Creator 추상 클래스 (팩토리 메서드 정의)
abstract class Dialog {
  // 팩토리 메서드
  abstract createButton(): Button

  // 비즈니스 로직
  renderDialog(): void {
    const button = this.createButton()
    button.render()
    button.onClick()
  }
}

// 4. 구체적인 Creator 클래스들
class WindowsDialog extends Dialog {
  createButton(): Button {
    return new WindowsButton()
  }
}

class MacDialog extends Dialog {
  createButton(): Button {
    return new MacButton()
  }
}

// 5. 클라이언트 코드
function clientApp(osType: string) {
  let dialog: Dialog

  if (osType === "Windows") {
    dialog = new WindowsDialog()
  } else {
    dialog = new MacDialog()
  }

  dialog.renderDialog()
}

// 실행 예시
clientApp("Windows")
// 출력: 윈도우 스타일 버튼 렌더링 / 윈도우 버튼 클릭 이벤트 처리

clientApp("Mac")
// 출력: 맥 스타일 버튼 렌더링 / 맥 버튼 클릭 이벤트 처리
  • Product 인터페이스 (Button)
    • 버튼이 공통적으로 가져야 할 동작(render, onClick)을 정의
  • 구체적인 Product 클래스 (WindowsButton, MacButton)
    • 각 OS 환경(Windows, Mac)에 맞는 버튼 동작을 구현
  • Creator 추상 클래스 (Dialog)
    • createButton()이라는 팩토리 메서드를 정의하고, renderDialog()에서 버튼 생성 → 렌더링 → 클릭 처리 과정을 수행.
  • 구체적인 Creator 클래스 (WindowsDialog, MacDialog)
    • OS별로 어떤 버튼을 생성할지 결정합니다.
  • 클라이언트 코드 (clientApp)
    • OS 타입만 지정하면, 적절한 Dialog가 선택되어 버튼이 생성되고 동작합니다.
  • 장점: 객체 생성 코드(new)가 분리되어 유지보수와 확장이 쉬워지고, 클라이언트는 구체적인 구현을 몰라도 일관되게 사용할 수 있음

4. Factory Method Pattern을 사용했을 때의 장점

4-1) 코드 유지보수성 향상

  • 생성 로직 집중화
    • 여러 모듈에 흩어진 new 호출을 팩토리 메서드 한 곳으로 모아 관리할 수 있음.
    • 생성 규칙 변경 시 단일 지점 수정으로 파급효과 최소화 가능!
  • 변경 이유 분리(SRP 충족)
    • 비즈니스 로직과 “어떻게 만들 것인가”가 분리됨 → 각각 독립적으로 변경 가능.
  • 일관된 후처리 적용 쉬움
    • Creator의 템플릿 메서드에서 공통 초기화, 검증, 로깅을 표준화 가능.
// Before: 생성이 군데군데 흩어져 있음
// 주문, 환불, 정산 모듈이 각각 new ConsoleLogger()/new RemoteLogger() 사용 중임

// After: 공통 Creator에서 일괄 관리됨
abstract class LoggerCreator {
  protected abstract createLogger(): Logger
  logInfo(msg: string) {
    const logger = this.createLogger()
    // 공통 포맷/마스킹/트레이싱 등 한 곳에서 처리됨
    logger.write(`[INFO] ${msg}`)
  }
}
Factory method Pattern 적용 결과: 규칙/후처리 변경 시 Creator만 고치면 전 구간에 반영 가능.

4-2) 확장성 증가 (OCP 충족)

  • 새 제품 추가 시 기존 코드 건드림 최소
    • 새로운 ConcreteProduct와 이를 반환하는 ConcreteCreator(또는 레지스트리 등록)만 추가하면 됨!
  • 기능 플러그인화 용이
    • 외부 팀/서드파티가 구현을 독립적으로 배포하고, 시스템은 팩토리로 연결만 하면 됨!
  • 런타임 선택/전환 가능
    • 환경변수, 구성파일, AB 테스트에 따라 동적으로 제품 교체 가능
// 새 Provider 추가해도 기존 클라이언트 영향 거의 없음
class S3Storage implements Storage { /* ... */ }
class GCSStorage implements Storage { /* ... */ }

class StorageCreator {
  create(kind: "s3" | "gcs"): Storage {
    if (kind === "s3") return new S3Storage()
    return new GCSStorage()
  }
}
// → 새로운 AzureStorage 추가 시: 클래스 하나 + 분기/레지스트리 등록만 수행하면 됨

4-3) 의존성 감소(DIP 충족)

  • 구체 클래스 의존 제거
    • 클라이언트는 Button, Payment, Logger 같은 상위 인터페이스에만 의존.
    • 컴파일/배포 단계에서 구체 구현 교체가 쉬움.
  • 테스트 대체 용이
    • 테스트 전용 ConcreteCreator로 페이크/스텁을 쉽게 주입 가능함 → 단위 테스트 간결해짐.
  • 결합도↓ + 재사용성↑
    • 모듈 간 직접 new 참조 제거로 순환 의존/과도한 import 방지할 수 있음.
// 테스트 친화적 구조
class TestDialog extends Dialog {
  protected createButton(): Button { return { render(){}, onClick(){ /* 호출 추적 */ } } }
}
// 클라이언트는 Button 인터페이스만 알면 되므로 테스트에서 손쉽게 가짜 주입됨
추가 이점
  • 공통 정책(리트라이, 서킷브레이커, 메트릭 수집)을 Creator 한 곳에 삽입 가능함
  • 생성 비용이 큰 경우 캐싱/플라이웨이트와 결합해 메모리/성능 최적화 가능함

5. 실무 적용 사례

  • UI 라이브러리
    • 다양한 OS나 테마별 컴포넌트를 생성할 때 활용
  • DB 연결 객체 생성
    • MySQL, PostgreSQL, MongoDB 등 DB 엔진에 맞는 커넥션 객체 생성 시 사용
  • 로깅 시스템
    • 파일 로그, 콘솔 로그, 원격 서버 로그 등 다양한 로그 전략 선택적 생성

6. 정리

  • 팩토리 메서드 패턴은 객체 생성 책임을 서브클래스로 위임하는 패턴!
  • new를 직접 사용하는 대신, 팩토리 메서드를 통해 객체 생성
  • OCP 원칙 준수: 새로운 객체 타입 추가 시 기존 코드 최소 수정
  • 주요 키워드:
    • Product, ConcreteProduct, Creator, ConcreteCreator
