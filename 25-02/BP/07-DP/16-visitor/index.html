07-16. 방문자 (Visitor) 패턴

1. 문제 상황

예를 들어, 파일 시스템을 나타내는 클래스가 있다고 가정

  • File, Folder 클래스가 있음
  • 처음에는 단순히 이름 출력만 필요했음
  • 나중에는 크기 계산, 트리 구조 출력, 압축 기능 같은 새로운 연산을 추가하고 싶어짐

문제점:

  • 연산이 늘어날 때마다 File, Folder 클래스 내부에 메서드를 계속 추가해야 함
  • 클래스가 점점 비대해지고, 단일 책임 원칙(SRP) 위반
// 문제 상황: 클래스 안에 연산이 계속 추가됨
class File {
  constructor(public name: string, public size: number) {}

  printName(): void {
    console.log(this.name)
  }

  // 새로운 연산을 추가할 때마다 File 내부를 수정해야 함
  getSize(): number {
    return this.size
  }

  compress(): void {
    console.log(`${this.name} 파일 압축`)
  }
}
2. 패턴 개요

1) 정의

  • Visitor 패턴은 객체 구조(Object Structure)와 동작(Operation)을 분리하는 행동 패턴으로, 데이터의 형태와 이를 처리하는 알고리즘을 서로 독립적으로 관리할 수 있도록 함.
  • 객체의 구조(데이터 클래스들)는 고정되어 있어 변경 가능성이 낮지만, 여기에 적용되는 새로운 연산은 기존 클래스의 코드를 수정하지 않고 외부에서 확장 가능함. 이는 유지보수성과 확장성을 동시에 확보하기 위한 설계 전략임.
  • 핵심 아이디어:
    → "데이터 구조는 안정적이고 잘 변하지 않으므로 내부를 수정하지 않고 그대로 두되, 자주 바뀌는 연산은 외부 Visitor 클래스로 분리하여 유연하게 관리한다"는 것임.

2) 목적

  • 새로운 연산(행동)을 쉽게 추가: 기존 데이터 클래스에 메서드를 추가하지 않고, 별도의 Visitor 클래스를 정의함으로써 새로운 기능을 독립적으로 확장할 수 있음. 이는 코드의 응집도를 높이고, 불필요한 클래스 수정으로 인한 오류 가능성을 줄임.
  • 개방-폐쇄 원칙(Open-Closed Principle, OCP) 준수: 객체 구조 자체는 변경에 닫혀 있고, 연산 확장은 열려 있음. 즉, 기존 클래스 정의를 안정적으로 유지하면서도, 외부 Visitor를 통해 기능 확장이 가능함.
  • 이중 디스패치(Double Dispatch) 지원: 실행 시점에 실제 Element 객체의 타입과 Visitor 객체의 타입을 모두 고려하여 올바른 연산이 호출됨. 단일 디스패치 언어(예: Java, C++)에서 객체 타입 기반 동적 바인딩을 한 단계 더 확장하여, 런타임에 구체적인 방문자 메서드를 정확히 실행할 수 있게 함.

3) 특징

  • 주요 구성 요소:
    • Element (요소): 방문될 수 있는 객체로, Visitor를 받아들이는 역할을 가짐. 예: File, Folder. 각 Element 클래스는 공통 인터페이스에 따라 accept() 메서드를 반드시 구현해야 함.
    • Visitor (방문자): Element에 대해 수행할 연산을 정의하는 인터페이스. 예: SizeCalculator, Renderer. 모든 구체 Visitor는 Element 타입별로 오버로딩된 메서드를 제공함.
    • accept() 메서드: Element가 Visitor를 받아들여 적절한 Visitor 메서드를 호출하게 하는 진입점. 즉, element.accept(visitor) 호출 시, 이중 디스패치가 발생하여 실제 객체 타입에 맞는 연산이 실행됨.
  • 장점: 새로운 연산을 추가하는 과정이 용이하고, 데이터 구조와 동작을 분리함으로써 관심사의 분리가 명확해짐. 이는 코드의 가독성과 재사용성을 높여줌.
  • 단점: 새로운 데이터 타입(Element)을 추가하기는 어렵다는 제약이 있음. 모든 Visitor 클래스가 해당 Element 타입을 지원하도록 수정되어야 하므로, 구조 확장에는 비용이 큼.
3. TypeScript 코드 예제

1) Visitor 인터페이스 정의

// Visitor 인터페이스
interface Visitor {
  visitFile(file: FileElement): void
  visitFolder(folder: FolderElement): void
}

2) Element 인터페이스 정의

// Element 인터페이스
interface Element {
  accept(visitor: Visitor): void
}

3) 구체적인 Element 클래스

class FileElement implements Element {
  constructor(public name: string, public size: number) {}

  accept(visitor: Visitor): void {
    visitor.visitFile(this) // 이중 디스패치 발생
  }
}

class FolderElement implements Element {
  constructor(public name: string, public children: Element[] = []) {}

  accept(visitor: Visitor): void {
    visitor.visitFolder(this)
  }
}

4) 구체적인 Visitor 클래스

// 파일 크기를 계산하는 Visitor
class SizeCalculator implements Visitor {
  private totalSize = 0

  visitFile(file: FileElement): void {
    this.totalSize += file.size
  }

  visitFolder(folder: FolderElement): void {
    folder.children.forEach(child => child.accept(this))
  }

  getTotalSize(): number {
    return this.totalSize
  }
}

// 이름을 출력하는 Visitor
class NamePrinter implements Visitor {
  visitFile(file: FileElement): void {
    console.log(`파일: ${file.name}`)
  }

  visitFolder(folder: FolderElement): void {
    console.log(`폴더: ${folder.name}`)
    folder.children.forEach(child => child.accept(this))
  }
}

5) 실행 예시

// 파일 시스템 구성
const file1 = new FileElement("a.txt", 10)
const file2 = new FileElement("b.txt", 20)
const folder = new FolderElement("docs", [file1, file2])

// Visitor 사용
const sizeVisitor = new SizeCalculator()
folder.accept(sizeVisitor)
console.log("총 크기:", sizeVisitor.getTotalSize()) // 출력: 30

const nameVisitor = new NamePrinter()
folder.accept(nameVisitor)
// 출력:
// 폴더: docs
// 파일: a.txt
// 파일: b.txt
4. 이 패턴을 사용했을 때의 장점
  • 연산 추가 용이
    → Visitor 클래스만 새로 만들면 됨 (Element 수정 불필요)
  • 코드의 책임 분리
    → 데이터 구조와 연산을 분리하여 SRP 충족
  • 복잡한 연산 캡슐화
    → 크기 계산, 출력, 통계 분석 같은 로직을 Visitor에 집중 가능
5. 실무 적용 사례
  • 웹 개발 (프론트엔드)
    • DOM 트리 탐색 후 Visitor 패턴으로 스타일링, 이벤트 바인딩, 직렬화 처리
  • 백엔드
    • 컴파일러/인터프리터: 추상 구문 트리(AST)에 Visitor를 적용해 의미 분석, 코드 생성, 최적화 수행
  • 데이터 처리
    • JSON/XML 구조를 순회하면서 Visitor로 변환, 검증, 통계 집계 수행
// 예: JSON 데이터에 Visitor 적용
class JsonPrinter implements Visitor {
  visitFile(file: FileElement): void {
    console.log(JSON.stringify({ type: "file", name: file.name, size: file.size }))
  }

  visitFolder(folder: FolderElement): void {
    console.log(JSON.stringify({ type: "folder", name: folder.name }))
    folder.children.forEach(child => child.accept(this))
  }
}
6. 정리
  • Visitor 패턴은 데이터 구조와 연산을 분리하는 행동 패턴
  • 구조(Element)는 그대로 두고, 새로운 연산(Visitor)만 추가 가능
  • 장점: OCP 준수, 연산 확장 용이, 책임 분리
  • 단점: 새로운 Element 타입 추가 시 모든 Visitor 수정 필요
  • 실무 활용: AST 처리, 파일 시스템 탐색, DOM 처리, JSON/XML 변환
