07-01. 싱글톤 패턴 (Singleton Pattern)

1. 패턴 개요

1) 정의

  • 한 클래스의 인스턴스를 오직 하나만 생성하고, 어디서든 그 유일한 인스턴스에 접근할 수 있도록 보장하는 패턴.
    • 유일 인스턴스 보장: 어떤 클래스의 객체를 단 1개만 만들고, 어디서든 그 객체에 접근할 수 있는 전역 진입점 제공.
    • 생성 통제: 생성자를 private으로 막고, static 메서드(예: getInstance())로만 획득하게 함.
    • TS vs Java
      • Java는 멀티스레드 동시성 고려가 중요.
      • JS/TS는 기본이 단일 스레드 이벤트 루프이지만, Worker/프로세스/번들 중복 로드 환경에서는 다중 인스턴스가 생길 수 있어 설계 주의 필요.

2) 목적 (언제 써야하는지?)

  • 프로그램 전역에서 하나만 있어야 하는 것을 공유할 때
    • 예: 로거(Logger), 설정/환경(Config), DB 연결 풀, 전역 캐시, 테마/지역화, 텔레메트리/트레이싱, 이벤트 버스 등.
  • 상태·자원 관리가 중앙집중적이어야 할 때
    • 비용이 큰 리소스(파일 핸들, 소켓, 브라우저 스토리지 래퍼 등)를 중복 생성하지 않도록.

3) 특징 (장점·단점)

  • 장점
    • 생성 비용 절감: 비싼 객체를 한 번만 생성.
    • 전역 접근: 사용이 간단하고 코드량이 적음.
    • 수명 관리 단순화: 앱 라이프사이클과 함께 움직임.
  • 단점·주의
    • 테스트 어려움/결합 증가: 전역 상태라 목(Mock) 교체가 어렵고, 모듈 간 결합이 커짐.
    • 은닉된 의존성: 어디서나 참조 가능 → 의존성 주입(Dependency Injection)과 상충.
    • 초기화 순서/메모리 상주: Eager 방식은 사용하지 않아도 메모리를 차지.
    • 환경 분리 이슈: HMR(핫 리로드), SSR/CSR 이원화, 여러 워커/프로세스 등에서는 “각 컨텍스트마다” 싱글톤이 따로 생길 수 있음.

2. 문제 상황
  • 전역적으로 공유해야 하는 객체가 있음 (예: 데이터베이스 연결 관리 객체)
  • 여러 개의 인스턴스를 만들면 불필요한 자원 낭비가 발생
  • 하지만 전역 변수를 그대로 사용하면 캡슐화가 깨지고 유지보수가 어려워짐

3. TypeScript 코드 예제

A. Eager Initialization

  • 한 번만 미리 만들어두는 직관적이고 심플한 기법
    • 클래스가 처음 로드될 때 인스턴스를 즉시 생성하기 때문에 구현이 간단함.
  • private static readonly 로 구현
    • TypeScript에서는 final이 없으므로 readonly를 사용해 재할당을 방지.
    • 따라서 멀티 스레드 환경이 아니더라도 안전하게 단일 인스턴스를 보장 가능!
  • 메모리 상주 문제
    • 정적(static) 멤버로 관리되기 때문에 실제로 사용하지 않더라도 메모리에 항상 올라가 있음.
    • 따라서 리소스가 큰 객체라면 공간 낭비가 발생할 수 있음.
  • 예외 처리 불가

    인스턴스가 생성되는 시점에 에러 처리를 끼워 넣을 수 없다는 단점이 있음.

// 싱글톤 클래스 정의
class Singleton {
  private static readonly instance: Singleton = new Singleton();

  private constructor() {}

  public static getInstance(): Singleton {
    return Singleton.instance;
  }

  // 테스트용 메서드
  public sayHello(): void {
    console.log("안녕하세요! 저는 싱글톤 인스턴스입니다 👋");
  }
}

// ------------------ 사용 예시 ------------------

// 1. 인스턴스 가져오기
const s1 = Singleton.getInstance();
const s2 = Singleton.getInstance();

// 2. 메서드 호출
s1.sayHello(); // 출력: 안녕하세요! 저는 싱글톤 인스턴스입니다 👋

// 3. 같은 객체인지 확인
console.log(s1 === s2); // true (항상 같은 인스턴스)

B. Static block initialization

  • 기본 개념
    • 정적(static) 블록은 클래스가 로딩될 때 단 한 번 실행되는 특별한 코드 블록.
    • Java에서는 이 블록 안에서 객체를 초기화하면서 예외 처리까지 넣을 수 있으며, TypeScript도 4.4 이상 버전부터 동일하게 static { ... } 블록을 지원함.
    • 따라서 Singleton.instance를 생성할 때 try/catch를 통해 안전하게 초기화할 수 있다는 특징이 있음.
  • 예외 처리 가능
    • Java에서는 static block을 이용해 인스턴스 생성 시 예외를 잡는 것이 가능.
    • TypeScript에는 static block이 없으므로, 대신 정적 초기화 함수나 즉시 실행 함수(IIFE) 를 활용해 예외 처리를 흉내내 비슷하게 구현 가능.
  • 메모리 상주 문제 동일
    • 정적(static) 멤버의 특성상, 객체를 실제로 사용하지 않더라도 메모리에 항상 올라가 있다는 단점은 여전히 존재!
// 싱글톤 클래스 정의
class Singleton { // TS 4.4 이상
  private static instance: Singleton;

  private constructor() {}

  static {
    try {
      Singleton.instance = new Singleton();
      console.log("✅ 싱글톤 인스턴스가 정상적으로 생성되었습니다.");
    } catch (error) {
      throw new Error("싱글톤 객체 생성 오류");
    }
  }

  public static getInstance(): Singleton {
    return Singleton.instance;
  }

  // 테스트용 메서드
  public greet(): void {
    console.log("안녕하세요! 저는 static block으로 생성된 싱글톤입니다 👋");
  }
}

// ------------------ 사용 예시 ------------------

// 1. 인스턴스 가져오기
const s1 = Singleton.getInstance();
const s2 = Singleton.getInstance();

// 2. 메서드 호출
s1.greet(); // 출력: 안녕하세요! 저는 static block으로 생성된 싱글톤입니다 👋

// 3. 같은 객체인지 확인
console.log(s1 === s2); // true (항상 같은 인스턴스)

C. Lazy Initialization

  • 내부에서 생성 시점 관리: 인스턴스가 필요할 때(getInstance()가 처음 호출될 때) 만들어서 반환.
  • 미사용 시 메모리 절약: 사용 전까지는 생성하지 않으므로 eager 방식의 “미사용 고정 메모리 점유” 단점을 줄임.
  • 스레드 세이프 이슈:
    • 브라우저/일반 Node.js 런타임은 단일 스레드 이벤트 루프이므로, Java처럼 멀티 스레드 경쟁으로 인해 “두 번 생성”되는 문제가 보통 발생하지 않음!
    • 다만 Worker Threads / Web Workers처럼 프로세스/스레드를 명시적으로 나눠 상태를 공유하거나(예: SharedArrayBuffer), 번들/모듈이 중복 로드되는 특수 상황(잘못된 번들링, 테스트 격리 등)에서는 별개의 인스턴스가 생길 수 있으므로, 이런 시나리오는 패턴/구성의 문제로 보고 별도 대책이 필요함.
class Singleton {
  private static instance: Singleton | null = null;

  private constructor() {}

  public static getInstance(): Singleton {
    if (Singleton.instance === null) {
      Singleton.instance = new Singleton(); // 최초 1회만 생성
    }
    return Singleton.instance;
  }

  // 예시 메서드
  public say(): void {
    console.log("lazy로 생성된 싱글톤입니다 👋");
  }
}

// 사용 예시
const a = Singleton.getInstance();
const b = Singleton.getInstance();
console.log(a === b); // true
a.say();
  • 다만, 비동기 초기화가 섞이거나(예: 첫 호출 시 외부 리소스 로드) 모듈 중복 로드 가능성을 조금이라도 줄이고 싶다면, 간단한 once 가드를 둘 수 있음.
    class Singleton {
  private static instance: Singleton | null = null;
  private static creating = false;

  private constructor() {}

  public static getInstance(): Singleton {
    if (Singleton.instance) return Singleton.instance;

    if (!Singleton.creating) {
      Singleton.creating = true;
      Singleton.instance = new Singleton();
      Singleton.creating = false;
    }
    return Singleton.instance!;
  }
}
    주의: JS/TS의 일반 실행 환경에서는 위 같은 가드가 필요하지 않은 경우가 대부분임. 멀티 워커/중복 번들 같은 배포 구조에서 인스턴스가 여러 개 생기는 문제는 애플리케이션 구조 차원의 해결(전역 모듈 캐시 재사용, 싱글 엔트리포인트, DI 컨테이너 등)이 더 적합할 수 있음.

D. TS에서의 Singleton

  • TS/ESM에서는 모듈이 한 번 로드되면 캐시되므로, 아래처럼 인스턴스를 모듈 단위로 만들어 export해도 사실상 싱글톤처럼 동작함! (TS만의 특징!)
class Service {
  public ping() { return "pong"; }
}

export const service = new Service(); // 모듈 캐시로 사실상 싱글톤
  • 라이브러리/프레임워크에 따라 테스트 격리, HMR(핫 리로드) 등으로 모듈 캐시가 초기화될 수 있으니, 이런 환경에선 클래식 싱글톤 패턴이 더 명확할 수 있으며, 환경에 따라 적절한 구현 방법이 필요함!

4. Signleton 패턴을 사용했을 때의 장점
  • 자원 관리 효율성
    • DB 연결, 파일 핸들러 등 무거운 자원을 한 번만 생성해 공유 가능
  • 글로벌 접근 보장
    • 프로그램 어디서든 동일한 객체를 사용할 수 있음
  • 객체 생성 제어
    • 무분별한 인스턴스 생성을 방지 → 예측 가능한 구조 유지
  • 코드 일관성 향상
    • 같은 객체를 사용하므로 상태 관리가 일관적으로 유지됨
5. 실무 적용 사례
  • 웹 프론트엔드
    • 전역 상태 관리 (예: Redux store, 앱 설정, 다크 모드 토글 관리)
  • 백엔드(Node.js)
    • 데이터베이스 연결 풀 (DB Connection Pool)
    • 환경 설정(Configuration) 객체
    • 로그 관리(Logger)
  • 게임 개발
    • 게임 내 오디오 매니저, 리소스 관리자

6. 정리
  • 싱글톤 패턴은 인스턴스를 하나만 생성하고 전역적으로 접근 가능하게 하는 패턴!
  • 생성자를 private으로 막고, static 메서드를 통해 접근하도록 구현함
  • 자원 낭비 방지, 일관된 상태 유지, 전역적 접근이 필요한 경우 유용함
  • 실무에서는 로거, DB 연결, 전역 설정 객체 등에서 많이 사용됨
