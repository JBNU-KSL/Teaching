07-09. 데코레이터 (Decorator) 패턴

1. 문제 상황

상황 예시

  • "알림 시스템"을 구현한다고 가정.
  • 기본적으로 메시지를 콘솔에 출력하는 Notifier 클래스가 있다고 가정.
  • 이후 이메일, SMS, 슬랙(Slack) 같은 다양한 채널에 알림을 보낼 수 있도록 확장하고 싶음.

단순 구현의 문제점

// - 새 채널 추가 시: enum/if/생성자/서브클래스/조합 전부 수정 필요
// - 조합 폭발: Email, SMS, Slack, Push가 생기면 2^n 조합 클래스가 늘어남
// - 공통 로깅/포맷팅이 중복됨

enum Channel {
  Base = "Base",
  Email = "Email",
  SMS = "SMS",
  Slack = "Slack",
  Push = "Push", // 🆕 신규 채널 추가했다고 가정
}

class Notifier {
  constructor(
    // ❌ 플래그/옵션 지옥: 조합이 늘수록 생성자가 비대해짐
    private readonly useEmail: boolean = false,
    private readonly useSMS: boolean = false,
    private readonly useSlack: boolean = false,
    private readonly usePush: boolean = false, // 🆕 추가
  ) {}

  // ❌ OCP 위반: 채널이 늘 때마다 메서드 본문을 계속 수정해야 함
  send(message: string): void {
    const formatted = `[${new Date().toISOString()}] ${message}`; // ❌ 중복될 공통 포맷팅

    console.log(`기본 알림: ${formatted}`);

    if (this.useEmail) {
      // ❌ 채널별 분기 + 중복 로직
      console.log(`[TRACK] sending via EMAIL`); // ❌ 중복 추적 로깅
      console.log(`이메일 발송: ${formatted}`);
    }

    if (this.useSMS) {
      console.log(`[TRACK] sending via SMS`);
      console.log(`SMS 발송: ${formatted}`);
    }

    if (this.useSlack) {
      console.log(`[TRACK] sending via SLACK`);
      console.log(`Slack 발송: ${formatted}`);
    }

    if (this.usePush) { // 🆕 추가
      console.log(`[TRACK] sending via PUSH`);
      console.log(`푸시 발송: ${formatted}`);
    }

    // TODO: 채널이 더 생기면 여기 if 추가 (WhatsApp, Teams, KakaoTalk, ...)
    // TODO: 재시도/에러처리/속도제한/템플릿 바인딩도 채널별로 분기 추가.. 지옥 시작
  }
}

class EmailNotifier extends Notifier {
  send(message: string): void {
    super.send(message);
    console.log(`[TRACK] sending via EMAIL`);
    console.log(`이메일 발송: ${message}`);
  }
}

class SMSNotifier extends Notifier {
  send(message: string): void {
    super.send(message);
    console.log(`[TRACK] sending via SMS`);
    console.log(`SMS 발송: ${message}`);
  }
}

class SlackNotifier extends Notifier {
  send(message: string): void {
    super.send(message);
    console.log(`[TRACK] sending via SLACK`);
    console.log(`Slack 발송: ${message}`);
  }
}

class PushNotifier extends Notifier { // 🆕 신규 채널 클래스
  send(message: string): void {
    super.send(message);
    console.log(`[TRACK] sending via PUSH`);
    console.log(`푸시 발송: ${message}`);
  }
}

// ❌ 조합 클래스들: 채널 하나 추가될 때마다 전 조합이 재정의되어야 함
class EmailAndSMSNotifier extends Notifier {
  send(message: string): void {
    super.send(message);
    console.log(`[TRACK] sending via EMAIL`);
    console.log(`이메일 발송: ${message}`);
    console.log(`[TRACK] sending via SMS`);
    console.log(`SMS 발송: ${message}`);
  }
}

    console.log(`[TRACK] sending via SLACK`);
    console.log(`Slack 발송: ${message}`);
  }
}

class SMSAndSlackNotifier extends Notifier {

    console.log(`SMS 발송: ${message}`);
    console.log(`[TRACK] sending via SLACK`);
    console.log(`Slack 발송: ${message}`);
  }
}

// 🆕 Push가 들어오자마자 아래와 같은 조합 클래스가 무한 증식:

    console.log(`[TRACK] sending via SMS`);

    console.log(`[TRACK] sending via EMAIL`);  console.log(`이메일 발송: ${message}`);
    console.log(`[TRACK] sending via SMS`);    console.log(`SMS 발송: ${message}`);
    console.log(`[TRACK] sending via SLACK`);  console.log(`Slack 발송: ${message}`);
    console.log(`[TRACK] sending via PUSH`);   console.log(`푸시 발송: ${message}`);
  }
}

function demo() {
  console.log("\n--- 플래그 기반 조합 (조건문 지옥) ---");

  n1.send("재고 임계치 도달");

  // 🧨 요구사항 변경: Slack 대신 Teams도 추가해달라?

2. 패턴 개요

1) 정의

  • 데코레이터 패턴(Decorator Pattern)은 객체에 새로운 기능을 동적으로 추가할 수 있도록 설계된 구조 패턴으로, 기존 클래스의 코드 변경 없이도 기능을 확장할 수 있는 방법을 제공할 수 있는 패턴!
  • 상속을 사용하는 대신, 기존 객체를 감싸는(wrapping) 방식으로 기능을 확장할 수 있으며, 이때 원래 객체는 내부적으로 그대로 동작하고 외부에서는 새로운 기능이 추가된 것처럼 보임.
  • 핵심 아이디어: "필요한 기능만 블록처럼 조합해서 붙이자"는 것을 기반으로 하여, 불필요한 기능까지 포함된 거대한 클래스 설계 대신, 작은 단위의 기능을 모듈화하여 조립할 수 있다는 장점을 지니고 있음!

2) 목적

  • 기능 확장 시 유연성 확보
    → 컴파일 시점에 새로운 클래스를 미리 정의하지 않고, 런타임에 객체에 기능을 덧붙일 수 있음. 따라서 상황에 따라 기능을 선택적으로 적용할 수 있음.
  • 클래스 폭발 문제 방지
    → 상속만으로 기능을 추가하면 가능한 기능 조합의 수만큼 서브클래스가 필요해짐. 데코레이터는 여러 기능을 독립된 클래스 단위로 나누어 필요할 때 조합하므로 이 문제를 방지함.
  • 단일 책임 원칙(Single Responsibility Principle, SRP) 준수
    → 각 데코레이터는 하나의 특정 기능만 담당하도록 구성하여, 기능이 분리되어 있으므로 코드 수정·재사용·테스트가 쉬워지고, 유지보수성이 높아짐.

3) 특징

  • 구성 요소
    • Component : 공통 인터페이스. 기본 기능의 형태를 정의하며, 클라이언트는 이를 통해 객체를 다룸.
    • ConcreteComponent : 실제 핵심 기능을 수행하는 객체로 데코레이터가 감싸는 대상이 됨.
    • Decorator : Component를 구현하고, 내부에 다른 Component를 포함하여 동작을 위임하도록 하며, 자체적으로는 새로운 기능을 정의하지 않음.
    • ConcreteDecorator : Decorator를 상속하여 구체적인 추가 기능을 구현하는 클래스. 예를 들어, 출력 결과에 새로운 동작을 덧붙일 수 있음.
  • 상속 vs 조합
    • 상속: 정적인 방식으로 기능을 확장할 수 있으나, 클래스 수가 기하급수적으로 늘어나 관리가 어려워짐.
    • 데코레이터: 객체 간 조합(Composition)으로 기능을 확장할 수 있으며, 원하는 기능만 선택적으로 덧붙일 수 있어 유연성이 큼.
  • 런타임 유연성
    • 객체 실행 도중 원하는 만큼 기능을 덧붙일 수 있음. 예를 들어, 하나의 객체에 여러 데코레이터를 순차적으로 적용하여 기능을 겹겹이 쌓을 수 있음.
3. TypeScript 코드 예제

// 1. Component 인터페이스
interface Notifier {
  send(message: string): void;
}

// 2. Concrete Component (기본 기능)
class BaseNotifier implements Notifier {
  send(message: string): void {
    console.log(`기본 알림: ${message}`);
  }
}

// 3. Decorator 추상 클래스
abstract class NotifierDecorator implements Notifier {
  protected wrappee: Notifier;

  constructor(notifier: Notifier) {
    this.wrappee = notifier;
  }

  send(message: string): void {
    this.wrappee.send(message); // 기존 기능 위임
  }
}

// 4. 구체적인 데코레이터들
class EmailNotifier extends NotifierDecorator {
  send(message: string): void {
    super.send(message);
    console.log(`이메일 발송: ${message}`);
  }
}

class SMSNotifier extends NotifierDecorator {
  send(message: string): void {
    super.send(message);
    console.log(`SMS 발송: ${message}`);
  }
}

class SlackNotifier extends NotifierDecorator {
  send(message: string): void {
    super.send(message);
    console.log(`Slack 발송: ${message}`);
  }
}

// ---------------- 사용 예시 ----------------
let notifier: Notifier = new BaseNotifier();

// 기능을 런타임에서 동적으로 조합
notifier = new EmailNotifier(notifier);
notifier = new SMSNotifier(notifier);
notifier = new SlackNotifier(notifier);

notifier.send("긴급 서버 장애 발생!");

출력 결과:

기본 알림: 긴급 서버 장애 발생!
이메일 발송: 긴급 서버 장애 발생!
SMS 발송: 긴급 서버 장애 발생!
Slack 발송: 긴급 서버 장애 발생!
4. 이 패턴을 사용했을 때의 장점
  • 조합 가능성: Email + SMS + Slack처럼 다양한 조합을 런타임에 결정 가능
  • 클래스 폭발 방지: 모든 조합을 위한 서브클래스를 만들 필요 없음
  • OCP(개방-폐쇄 원칙) 충족: 기존 코드 수정 없이 새로운 기능을 데코레이터로 추가 가능
  • 단일 책임 원칙 준수: 각 데코레이터는 자신의 기능만 담당
5. 실무 적용 사례
  • 웹 프론트엔드
    • React에서 HOC(Higher-Order Component) 패턴 → 컴포넌트를 감싸서 기능 추가
    • 예: withAuth, withLogger 등
  • 백엔드
    • Express.js 미들웨어 → 요청 객체(req)를 감싸면서 기능 추가
    • 예: 로깅, 인증, 캐싱
  • 데이터 처리
    • 파일 I/O → 압축, 암호화, 버퍼링 기능을 데코레이터로 계층적으로 추가
  • UI 컴포넌트
    • 스크롤바, 테두리, 그림자 효과를 동적으로 조합

간단한 예시 (Express 미들웨어 유사 구조):

function loggerMiddleware(req: string, next: () => void) {
  console.log("요청 로그:", req);
  next();
}

function authMiddleware(req: string, next: () => void) {
  console.log("사용자 인증 확인:", req);
  next();
}

function handler(req: string) {
  console.log("최종 요청 처리:", req);
}

// 미들웨어 체이닝 (데코레이터와 유사)
loggerMiddleware("GET /users", () => {
  authMiddleware("GET /users", () => {
    handler("GET /users");
  });
});
6. 정리
  • 데코레이터 패턴은 객체에 동적으로 기능을 추가하는 구조 패턴
  • 상속 대신 객체 합성을 사용하여 클래스 폭발 문제를 방지
  • 런타임에 유연하게 기능을 조합할 수 있음
  • 실무에서는 UI 확장, 미들웨어 체인, 데이터 스트림 처리 등에 활용됨
  • 핵심 포인트:
    • "상속 대신 합성"
    • "필요한 기능만 조합"
    • "OCP 준수"
