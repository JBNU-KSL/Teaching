07-06. 어댑터 (Adapter) 패턴

1. 문제 상황

상황 설명

  • 어떤 시스템에서 "새로운 결제 시스템 API"를 도입하려고 가정해보자.
  • 하지만 기존 코드(Client)는 "기존 결제 인터페이스"만 사용하도록 작성되어 있음.

문제 예시

// 기존 결제 시스템 인터페이스 (Client는 이걸 기대함)
interface OldPaymentProcessor {
  pay(amount: number): void;
}

// 새로운 결제 시스템 (서드파티 API)
class NewPaymentGateway {
  makePayment(value: number): void {
    console.log(`${value}원을 새 결제 게이트웨이로 처리함`);
  }
}

// 클라이언트 코드 (구 OldPaymentProcessor만 알고 있음)
function processPayment(processor: OldPaymentProcessor, amount: number) {
  processor.pay(amount);
}

// 현재 상태에서는 NewPaymentGateway를 바로 쓸 수 없음
const newGateway = new NewPaymentGateway();
// ❌ processPayment(newGateway, 1000); → 타입 불일치 에러 발생
2. 패턴 개요

1) 정의

  • 서로 호환되지 않는 인터페이스(interface)를 가진 객체들을 함께 동작시킬 수 있도록 중간 변환기(Wrapper) 역할을 하는 구조 패턴(Structural Pattern). 즉, 클라이언트가 요구하는 인터페이스와 실제 제공되는 인터페이스 사이를 연결해주는 일종의 변환 계층을 제공함.
  • 기존 코드를 직접 수정하지 않고, 새로운 인터페이스를 기존 코드와 연결해주는 다리(Bridge) 역할을 수행. 이때 Adapter는 기존 코드의 안정성을 유지하면서도 확장 가능성을 제공함.
  • 흔히 "콘센트 어댑터"에 비유할 수 있으며, 예를 들어, 110V 전기 제품을 220V 소켓에서 사용할 수 있도록 변환하는 장치처럼, 서로 다른 규격을 가진 두 요소를 매끄럽게 연결하는 것이 목적.

2) 목적

  • 호환성 문제 해결: 기존 클래스와 새로운 클래스 간 인터페이스 불일치를 해소하여, 클라이언트는 변환 과정을 알 필요 없이 통일된 방식으로 객체를 사용할 수 있음.
  • 재사용성 증가: 이미 존재하는 코드를 변경하지 않고, 새로운 요구사항에 맞춰 활용 가능하므로 레거시 시스템이나 외부 라이브러리도 쉽게 재사용할 수 있음.
  • OCP(Open-Closed Principle, 개방-폐쇄 원칙) 준수: 기존 클래스는 수정(폐쇄)하지 않고, 새로운 어댑터 클래스를 정의하여 기능을 확장(개방)함으로써 유지보수성과 확장성을 동시에 확보할 수 있음.

3) 특징

  • 구성 요소
    • Target: 클라이언트가 기대하는 인터페이스. 클라이언트 코드는 오직 Target 인터페이스에만 의존하므로, Adapter가 뒤에서 어떤 변환을 수행하는지는 알 필요 없음.
    • Adaptee: 기존 클래스. 클라이언트가 직접 사용하기에는 인터페이스가 맞지 않는 객체를 의미함.
    • Adapter: Target 인터페이스를 구현하고, 내부적으로 Adaptee의 메서드를 호출해 변환을 담당함. 따라서 클라이언트는 Adaptee의 복잡한 구현 세부사항을 알지 않고도 서비스를 이용할 수 있음.
  • 적용 상황
    • 기존 시스템과 새로운 모듈을 통합해야 하는 경우. 예를 들어, 기존 레거시 API와 신규 서비스 API 간 연결.
    • 서드파티 라이브러리의 인터페이스가 클라이언트 요구와 다를 때. 즉, 라이브러리를 직접 수정하지 않고도 원하는 방식으로 활용 가능함.
    • 레거시 코드와 현대식 코드(예: REST ↔ gRPC, 콜백(callback) ↔ Promise)를 연결해야 할 때 유용함. 이는 서로 다른 프로그래밍 패러다임을 매끄럽게 이어주는 전형적 활용 사례임.
  • 장점
    • 기존 코드를 전혀 수정하지 않음 → 안정성 보장 및 기존 시스템에 대한 신뢰 유지.
    • 서로 다른 인터페이스의 클래스들을 쉽게 연결할 수 있어 시스템 확장 시 유연성이 높음.
    • 점진적인 시스템 교체 가능. 즉, 기존 기능은 그대로 두면서 새로운 기능을 추가해도 클라이언트 코드는 영향을 받지 않음.
3. TypeScript 코드 예제

// 1. Target 인터페이스 (클라이언트 기대 인터페이스)
interface OldPaymentProcessor {
  pay(amount: number): void;
}

// 2. Adaptee (새로운 클래스, 호환 불가)
class NewPaymentGateway {
  makePayment(value: number): void {
    console.log(`${value}원을 새 결제 게이트웨이로 처리함`);
  }
}

// 3. Adapter (Target 인터페이스 구현 + Adaptee를 내부에서 호출)
class PaymentAdapter implements OldPaymentProcessor {
  private gateway: NewPaymentGateway;

  constructor(gateway: NewPaymentGateway) {
    this.gateway = gateway;
  }

  pay(amount: number): void {
    // 내부적으로 새로운 API 호출
    this.gateway.makePayment(amount);
  }
}

// 4. 클라이언트 코드
function processPayment(processor: OldPaymentProcessor, amount: number) {
  processor.pay(amount);
}

// 5. 실행 예시
const newGateway = new NewPaymentGateway();
const adapter = new PaymentAdapter(newGateway);

// 클라이언트는 여전히 OldPaymentProcessor 인터페이스만 사용
processPayment(adapter, 5000);
// 출력: 5000원을 새 결제 게이트웨이로 처리함
4. 이 패턴을 사용했을 때의 장점
  • 코드 유지보수성
    • 기존 클라이언트 코드를 수정할 필요가 없음.
    • 클라이언트는 항상 Target 인터페이스에만 의존하므로, 내부적으로 어떤 Adaptee가 연결되든지 동일한 방식으로 동작.
    • 이로 인해 코드 변경 범위가 최소화되고, 오류 발생 가능성도 줄어듦.
  • 재사용성
    • 레거시 코드와 최신 코드를 동시에 활용할 수 있음.
    • 기존에 작성된 클래스는 인터페이스가 다르더라도 Adapter를 통해 그대로 재사용 가능하므로, 새로운 요구사항에 맞게 과거 자산을 낭비하지 않고 활용할 수 있음.
  • 확장성:
    • 새로운 시스템이나 라이브러리가 추가되더라도 기존 구조를 건드릴 필요가 없음.
    • 단지 새로운 Adapter 클래스만 작성하면 되므로, 시스템 규모가 커져도 변경 비용이 적고 확장이 용이함.
  • 유연성:
    • 하나의 Target 인터페이스에 대해 여러 종류의 Adapter를 구현할 수 있으므로, 다양한 서드파티 API를 동시에 지원할 수 있음.
    • 즉, 클라이언트는 동일한 방식으로 호출하면서도, 내부적으로는 서로 다른 API나 프로토콜을 활용하는 유연한 구조를 가질 수 있음.
5. 실무 적용 사례
  • 프론트엔드
    • 오래된 콜백 기반 API를 Promise 또는 async/await 스타일로 변환할 때
    • 브라우저 API 간 호환 (예: 구형 이벤트 처리기를 현대 이벤트 시스템과 연결)
  • 백엔드
    • 서로 다른 DB 드라이버를 통합하는 공통 인터페이스 제공
    • REST API ↔ gRPC 변환기
// 예: 콜백 기반 API를 Promise로 감싸는 어댑터
function callbackToPromise<T>(fn: (cb: (err: any, result: T) => void) => void): Promise<T> {
  return new Promise((resolve, reject) => {
    fn((err, result) => {
      if (err) reject(err);
      else resolve(result);
    });
  });
}
  • 외부 라이브러리
    • 특정 로깅 라이브러리(winston, pino 등)를 프로젝트 표준 로깅 인터페이스에 맞추는 경우
6. 정리
  • 어댑터 패턴은 서로 다른 인터페이스를 연결해주는 변환기 역할을 하는 패턴!
    • 기존 코드를 수정하지 않고, 새로운 시스템이나 라이브러리를 안전하게 도입 가능
    • Target-Adapter-Adaptee 구조 이해가 핵심
    • 실무에서는 레거시-신규 시스템 통합, API 변환, 서드파티 라이브러리 호환 등에서 자주 사용됨
