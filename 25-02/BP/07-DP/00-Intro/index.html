07-00. 객체지향 디자인패턴?

1. 객체지향 패러다임 (Object-Oriented Paradigm)

1.1 객체지향이란 무엇인가?

객체지향(Object-Oriented)이란 “데이터와 그 데이터를 처리하는 행위를 하나의 단위(객체)”로 묶어 사고하는 프로그래밍 패러다임으로, 프로그래밍 기법 중 하나임.

* 핵심 개념

  • 절차지향(Procedural) 프로그래밍이 “함수의 순서와 흐름” 중심이라면, 객체지향(Object-Oriented) 프로그래밍은 “객체 간의 상호작용” 중심으로 볼 수 있음!
  • 프로그램의 구성 요소를 “객체(Object)” 로 보고, 각 객체가 데이터를 보유하고, 메서드를 통해 서로 메시지를 주고받으며 동작하도록 설계하는 방식!
  • 객체는 현실 세계의 개념을 소프트웨어 안으로 모델링한 것이라 할 수 있음 → 설계 매우 중요!

⚙️ 예시

개념현실 세계 예시코드 표현
객체(Object)자동차Car 클래스의 인스턴스
속성(Attribute)자동차의 색상, 속도color, speed
행동(Method)가속하기, 멈추기accelerate(), stop()
class Car:
    def __init__(self, color):
        self.color = color
        self.speed = 0

    def accelerate(self):
        self.speed += 10

    def stop(self):
        self.speed = 0

→ 이렇게 객체가 자신의 데이터(speed, color)와 행위(accelerate, stop)를 함께 가지고 있는 것이 바로 객체지향의 핵심으로 볼 수 있음!

1.2 객체지향의 4대 특성

객체지향의 4대 특성은 캡슐화, 추상화, 상속, 다형성으로, 이 네 가지 개념이 객체지향 설계의 기초이자 유연한 소프트웨어를 만드는 핵심 원리로 볼 수 있음.

(1) 캡슐화 (Encapsulation)

객체 내부의 속성과 기능을 감추고, 외부에서는 정해진 인터페이스만 사용하도록 하는 것.
  • 데이터를 은닉(data hiding) 하여 외부에서 직접 접근하지 못하게 함.
  • 외부와 내부의 결합도를 낮추고, 내부 구현 변경 시 외부에 미치는 영향을 최소화함.
  • 접근 제어자(private, protected, public)로 구현.

* 장점

  • 데이터 보호
  • 유지보수성 향상
  • 코드 안정성 증가
  • 예시 (Python)
class BankAccount:
    def __init__(self, balance):
        self.__balance = balance  # private 변수

    def deposit(self, amount):
        self.__balance += amount

    def withdraw(self, amount):
        if self.__balance >= amount:
            self.__balance -= amount
        else:
            print("잔액 부족")

    def get_balance(self):
        return self.__balance
외부에서 멤버변수인 __balance에 직접 접근할 수 없고, 반드시 deposit() / withdraw() 메서드를 통해서만 조작 가능함

(2) 추상화 (Abstraction)

복잡한 시스템에서 중요한 부분만 드러내고 불필요한 세부사항은 감추는 방식.
  • 사용자는 “무엇을 할 수 있는가(What)”에 집중하고,
    구현자는 “어떻게 동작하는가(How)”에 집중하는 전략!
  • 인터페이스(interface)나 추상 클래스(abstract class)로 표현 가능.
  • 현실의 복잡한 개념을 단순화하여 모델링할 수 있음.
  • 예시
from abc import ABC, abstractmethod

class Payment(ABC):
    @abstractmethod
    def pay(self, amount):
        pass

class CreditCardPayment(Payment):
    def pay(self, amount):
        print(f"{amount}원을 신용카드로 결제했습니다.")

class PayPalPayment(Payment):
    def pay(self, amount):
        print(f"{amount}원을 PayPal로 결제했습니다.")
사용자 입장에서는 “결제(pay)”라는 추상적 행위만 알고, 구체적인 결제 방식은 각 클래스에서 다르게 정의됨.

(3) 상속 (Inheritance)

기존 클래스의 속성과 기능을 확장하거나 재사용하기 위한 메커니즘.
  • “is-a” 관계를 표현 (예: Dog is an Animal)
  • 코드 중복 제거, 공통 로직 재사용 가능
  • 단일 상속만 지원하는 언어도 있고, 다중 상속을 지원하는 언어도 있음
  • 주의: 과도한 상속 구조는 유지보수 복잡도를 높임

* 예시

class Animal:
    def speak(self):
        print("소리를 냅니다.")

class Dog(Animal):
    def speak(self):
        print("멍멍!")

dog = Dog()
dog.speak()  # 출력: 멍멍!
Dog 클래스는 Animal 클래스를 상속받고, 기능을 재정의(override)하여 구체적인 행동을 구현.

(4) 다형성 (Polymorphism)

하나의 인터페이스나 메서드가 여러 형태로 동작할 수 있는 성질.
  • 상속 관계에서 동일한 메서드 이름이지만, 실제 동작은 객체의 구체적 타입에 따라 달라짐.
  • 동적 바인딩 (Dynamic Binding) 을 통해 런타임에 메서드가 결정됨.
  • 오버라이딩(Overriding)과 오버로딩(Overloading)을 통해 구현.

* 예시

class Animal:
    def speak(self):
        print("소리를 냅니다.")

class Cat(Animal):
    def speak(self):
        print("야옹!")

class Cow(Animal):
    def speak(self):
        print("음메!")

animals = [Cat(), Cow()]
for a in animals:
    a.speak()  # 각 객체에 맞는 동작 수행
a.speak()는 동일한 호출이지만, 실제로는 각 객체의 타입에 따라 다르게 동작하며, 이것이 다형성의 핵심으로 볼 수 있음!

1.3 객체지향 분석 / 설계 / 프로그래밍 (OOA / OOD / OOP)

객체지향 접근법은 단순히 “코딩 스타일”이 아니라,
분석(Analysis) → 설계(Design) → 프로그래밍(Programming) 의 세 단계를 모두 포함하는 개념!

① 객체지향 분석 (Object-Oriented Analysis, OOA)

  • 현실 세계의 문제를 객체와 관계로 식별
  • 각 객체의 책임(responsibility) 과 협력(collaboration) 을 정의
예: “학생은 강의를 수강한다” → 학생 객체, 강의 객체, “수강” 관계로 모델링

② 객체지향 설계 (Object-Oriented Design, OOD)

  • 분석 결과를 코드로 옮길 수 있도록 클래스, 인터페이스, 상호작용 구조를 구체화
  • SOLID 원칙, 디자인 패턴 등을 고려해 설계
예: “학생” 클래스가 “강의” 클래스의 메서드를 호출하도록 설계

③ 객체지향 프로그래밍 (Object-Oriented Programming, OOP)

  • 실제 코드로 구현하는 단계
  • 클래스, 메서드, 인터페이스를 정의하고 시스템 전체를 통합
예: Python, Java, C++ 등에서 실제 코드 작성

1.4 왜 객체지향이 중요한가?

“좋은 객체지향 설계는 변경에 유연한 소프트웨어를 만든다.”
  1. 복잡성 관리
    • 시스템을 작은 객체 단위로 분리 → 이해하기 쉬움
  1. 유지보수성 향상
    • 내부 구현을 바꿔도 외부에 영향이 적음
  1. 재사용성 증대
    • 상속, 조합을 통해 기존 코드 활용
  1. 확장 용이성
    • 새로운 기능 추가 시 기존 구조를 최소한으로 수정

* 주의할 점

  • 객체지향은 만능은 아님..!
  • 지나친 상속이나 과도한 추상화는 오히려 복잡성을 높일 수 있음!
  • 객체 간의 결합도(coupling) 를 낮추고, 응집도(cohesion) 를 높이는 것이 핵심..!

핵심 정리 요약

개념의미장점대표 예시
캡슐화내부 구현을 숨기고, 인터페이스만 공개변경에 유연접근제한자
추상화불필요한 세부사항 제거, 본질적 개념만 표현복잡성 감소인터페이스, 추상클래스
상속기존 클래스 확장 및 재사용코드 재사용부모-자식 클래스
다형성동일한 메시지에 다른 동작유연성 증가메서드 오버라이딩

2. 객체지향과 디자인패턴

2.1 디자인 패턴이란 무엇인가?

디자인 패턴(Design Pattern) 이란, “객체지향 설계에서 반복적으로 등장하는 문제를 해결하기 위한 검증된 설계 해법(Reusable Solution)”을 의미함.
  • 즉, 개발자들이 수십 년 동안 다양한 소프트웨어 설계 경험을 통해 축적한 ‘좋은 설계 방법’의 정형화된 결과물이라 할 수 있음.
  • 패턴은 구현 코드가 아니라 설계의 아이디어이므로, 언어나 기술에 관계없이 적용 가능함.
  • 객체지향의 유연성(Flexibility) 과 재사용성(Reusability) 을 실제로 실현하기 위한 구체적인 실천 도구임.
쉽게 말해, “문제를 해결하는 방법을 매번 새로 생각하지 않고, 이미 검증된 구조를 활용하자!”는 것.

2.2 왜 디자인 패턴을 배워야 하는가?

(1) 재사용 가능한 설계 지식 축적

  • 모든 문제를 처음부터 새롭게 설계할 필요가 없음.
  • 이미 검증된 구조를 참고하면 개발 속도와 품질이 향상됨.

(2) 유지보수성과 확장성 향상

  • 디자인 패턴을 적용하면 객체 간 결합도를 낮추고, 확장 시 기존 코드 변경을 최소화할 수 있음.
  • 즉, 변경에 유연한 시스템을 만들 수 있음.

(3) 의사소통의 표준화

  • “이 부분은 팩토리 패턴으로 구현했어.”
    → 한 문장으로도 구조를 이해할 수 있음.
  • 개발자 간에 공통 언어(Common Vocabulary) 로 작동함.

(4) 객체지향 원리(SOLID)와의 연결

  • 대부분의 디자인 패턴은 SOLID 원칙을 구체적으로 실천한 형태임.
  • 예를 들어, “전략 패턴(Strategy Pattern)”은 개방-폐쇄 원칙(OCP) 과 의존성 역전 원칙(DIP) 을 잘 반영한 패턴임.

2.3 디자인 패턴의 분류

디자인 패턴은 일반적으로 GoF(Gang of Four) 가 정리한 23가지 패턴으로부터 출발함.
  • 이들은 크게 세 가지 범주로 나눌 수 있음.
분류설명대표 패턴
생성(Creational) 패턴객체 생성 방식의 유연성을 높임Singleton, Factory Method, Abstract Factory, Builder, Prototype
구조(Structural) 패턴클래스나 객체를 조합하여 더 큰 구조를 형성Adapter, Bridge, Composite, Decorator, Facade, Proxy, Flyweight
행위(Behavioral) 패턴객체 간의 상호작용과 책임 분배에 중점Strategy, Observer, Command, State, Template Method, Chain of Responsibility

2.4 대표적인 디자인 패턴 예시

(1) 싱글톤 패턴 (Singleton Pattern)

하나의 클래스에 대해 단 하나의 인스턴스만 존재하도록 보장하고, 그 인스턴스에 전역 접근점(Global Access Point) 을 제공하는 패턴.
class Singleton:
    __instance = None

    @staticmethod
    def get_instance():
        if Singleton.__instance is None:
            Singleton.__instance = Singleton()
        return Singleton.__instance

s1 = Singleton.get_instance()
s2 = Singleton.get_instance()
print(s1 is s2)  # True
  • 장점: 공통 자원(로그, 설정 파일 등)에 일관된 접근 제공
  • 단점: 전역 상태 관리 시, 테스트 어려움 / 의존성 증가 가능

(2) 팩토리 메서드 패턴 (Factory Method Pattern)

객체 생성을 서브클래스에 위임함으로써, 객체 생성 코드와 사용 코드를 분리하는 패턴.
from abc import ABC, abstractmethod

class Product(ABC):
    @abstractmethod
    def use(self):
        pass

class ConcreteProductA(Product):
    def use(self):
        print("Product A 사용")

class Creator(ABC):
    @abstractmethod
    def create_product(self):
        pass

class ConcreteCreatorA(Creator):
    def create_product(self):
        return ConcreteProductA()

creator = ConcreteCreatorA()
product = creator.create_product()
product.use()
  • 장점: 코드 수정 없이 새로운 제품 유형 추가 가능 → 개방-폐쇄 원칙(OCP) 준수
  • 활용 예: Django의 ORM, Spring Framework의 Bean Factory 등

(3) 전략 패턴 (Strategy Pattern)

알고리즘 군을 정의하고, 각각을 캡슐화하여 런타임에 알고리즘을 교체할 수 있게 하는 패턴.
class Strategy:
    def execute(self, a, b):
        pass

class Add(Strategy):
    def execute(self, a, b):
        return a + b

class Subtract(Strategy):
    def execute(self, a, b):
        return a - b

class Context:
    def __init__(self, strategy):
        self.strategy = strategy

    def operate(self, a, b):
        return self.strategy.execute(a, b)

context = Context(Add())
print(context.operate(10, 5))  # 15

context.strategy = Subtract()
print(context.operate(10, 5))  # 5
  • 장점: 알고리즘 변경 시 코드 수정 최소화, 다형성 활용
  • 핵심 원리: “행위를 객체로 바꾸어, 변경에 닫히지 않고 확장에는 열려 있게 한다.”

2.5 디자인 패턴의 핵심 포인트

  1. “패턴은 도구이지 규칙이 아니다.”
    • 무조건 패턴을 적용하는 것이 아니라, 상황에 맞게 선택하는 것이 중요함.
    • 과도한 패턴 사용은 오히려 설계를 복잡하게 함.
  1. “변화에 유연한 구조를 만드는 것이 목표다.”
    • 디자인 패턴의 본질은 변경을 쉽게 하는 설계에 있음.
    • 즉, “변경될 부분과 고정된 부분을 분리하는 것” 이 핵심 전략임.
  1. “패턴보다 원칙이 먼저다.”
    • SOLID 원칙을 이해하지 못한 채 패턴을 외우는 것은 의미가 없음.
    • 패턴은 원칙의 구체적 구현일 뿐, 원리를 이해해야 제대로 활용 가능함.
  1. “패턴은 경험을 코드화한 설계 지식이다.”
    • 패턴을 학습함으로써, 경험 많은 개발자들의 사고방식을 빠르게 습득할 수 있음.
    • 즉, 소프트웨어 설계의 ‘언어’를 배운다는 점에서 가치가 큼.
