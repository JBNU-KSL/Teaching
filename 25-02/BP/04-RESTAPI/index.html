04. API와 활용

날짜
A. REST API 개요

1. 네트워크와 웹 기초

1-1. 네트워크란?

  • 네트워크(Network): 여러 대의 컴퓨터가 서로 데이터를 주고받을 수 있도록 연결된 구조를 의미. 연결 방식에는 유선(케이블)과 무선(Wi-Fi 등)이 있으며, 모든 연결은 공통된 규약(프로토콜, Protocol)에 따라 이루어짐.
  • 인터넷(Internet): 전 세계의 네트워크가 모여 형성된 초거대 네트워크로, 인터넷은 단일 시스템이 아니라 수많은 작은 네트워크들이 표준 프로토콜(TCP/IP)을 기반으로 연결되어 형성되어 있음.
우리가 웹사이트를 접속할 때, 사실은 브라우저(클라이언트)가 서버에게 요청(Request)을 보내고, 서버가 응답(Response)을 돌려주는 과정으로 통신이 발생!

1-2. 클라이언트와 서버

  • 클라이언트(Client):
    • 서비스를 요청하는 주체.
    • 일반적으로 사용자가 직접 조작하는 프로그램 (예: 웹 브라우저나 모바일 앱).
    • 클라이언트는 요청을 생성하고 서버로 전송하는 역할만 수행하며, 실제 데이터 저장이나 복잡한 연산은 하지 않음.
  • 서버(Server):
    • 클라이언트가 보낸 요청을 처리하고 그 결과를 응답하는 주체.
    • 웹 서버(Web Server)는 정적 페이지나 API 응답을 제공하고, 데이터베이스 서버(Database Server)는 데이터를 저장·조회·갱신하는 역할을 수행.
    • 서버는 보통 여러 클라이언트로부터 동시에 요청을 받아야 하므로, 강력한 병렬 처리와 자원 관리가 요구됨.

  • 실생활 비유:

  • 클라이언트 = 식당의 손님 (서비스를 요청하는 주체)
  • 서버 = 요리사 (요청을 받아 조리·처리를 수행하는 주체)
  • 주문(요청) → 조리(처리) → 음식(응답)이라는 흐름으로 이해할 수 있으며, 이 과정에서 손님이 요리 과정을 직접 하지 않는 것처럼, 클라이언트는 내부 처리 과정을 알 필요 없이 결과만 전달받음.

1-3. HTTP 프로토콜

  • HTTP (HyperText Transfer Protocol): 웹에서 데이터를 주고받기 위한 대표적인 규칙. 요청과 응답이 모두 문자 메시지 형식으로 이루어지도록 하고, 클라이언트와 서버 간 통신의 기본 단위가 됨.
    • 요청(Request)에는 어떤 동작을 수행할지(예: 조회, 생성)와 대상 자원이 명시됨.
    • 응답(Response)에는 처리 결과와 상태 코드(Status Code), 그리고 필요한 경우 데이터가 함께 포함됨.

  • HTTP 요청 예시:
    GET /students/1 HTTP/1.1
Host: example.com
    • GET: 서버에게 데이터를 조회하겠다는 요청 메서드. 이 외에도 POST, PUT, DELETE 등 다양한 메서드가 존재.
    • /students/1: 서버 자원의 경로(URL Path)로, 여기서는 ID가 1인 학생 데이터를 요청하는 의미.
    • HTTP/1.1: 사용 중인 HTTP 버전. 현재는 1.1과 2가 널리 쓰이며, 최신 버전은 HTTP/3.
    • Host: example.com: 요청 대상 서버의 호스트명을 명시하도록 함. 여러 도메인을 같은 서버에서 운영할 수 있으므로 반드시 포함되어야 함.!

  • HTTP 응답 예시:
    HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 1,
  "name": "홍길동"
}
    • HTTP/1.1 200 OK: 응답 버전과 상태 코드로, 200 OK는 요청이 정상적으로 처리되었음을 의미.
    • Content-Type: application/json: 응답 본문의 데이터 형식을 지정. 여기서는 JSON 형식의 데이터를 반환.
    • { "id": 1, "name": "홍길동" }: 실제 응답 데이터로, 클라이언트가 요청한 학생 정보가 JSON 객체 형태로 전달됨.

  • 위처럼, HTTP는 요청 메시지와 응답 메시지를 교환하는 단순한 구조를 가지며, 이 기본 원리를 확장한 것이 REST API 통신으로 볼 수 있음..!

2. API란 무엇인가?

  • API (Application Programming Interface): 애플리케이션(프로그램)과 애플리케이션이 서로 상호작용할 수 있도록 도와주는 약속(인터페이스)을 의미. 즉, API는 특정 기능을 어떻게 호출하고 사용할 수 있는지를 정의한 규칙과 명세임.
  • 프로그래머는 API를 사용해 복잡한 내부 동작을 몰라도, 정해진 방식대로 요청을 보내고 응답을 받으면 기능을 활용할 수 있음. 예를 들어, 운영체제는 파일 열기/저장 API를 제공하고, 웹 서비스는 데이터 조회/등록 API를 제공.

2-1. 일상적 비유 예시

  • 카페 주문 비유:
    • 손님이 카페에서 "아메리카노 주세요"라고 말하면, 점원은 메뉴판(규칙)에 따라 주문을 이해하고, 커피를 만들어 손님에게 제공함.
    • 손님은 커피 머신이 어떻게 작동하는지, 원두가 어디서 오는지는 알 필요 없음. 메뉴판에 적힌 방식대로 주문만 하면 됨.
    • 이때 손님 = 클라이언트, 점원 = 서버, 메뉴판 = API라고 볼 수 있음.
  • API도 위와 같은 방식으로 동작! :
    • 클라이언트는 API 문서에 정의된 규칙대로 요청을 보냄.
    • 서버는 그 요청을 처리하고, 정해진 형식으로 응답을 반환함.
    • 규칙을 벗어난 요청은 처리되지 않고, 오류가 발생할 수 있음. (예: 잘못된 메뉴 주문 → "그런 메뉴는 없습니다"와 같은 응답)
  • 정리하면, API는 프로그램 간 소통을 위한 공식적인 언어이자 메뉴판 같은 역할을 수행할 수 있으며, 이를 통해 개발자는 내부 동작을 몰라도 필요한 기능을 쉽게 활용할 수 있음.

2-2. 웹 API의 역할

  • 서버와 데이터베이스안의 리소스에 접근
    • 데이터베이스의 정보를 누구나 열람하면 곤란하기에, 필요에 의해서만 열람 가능.
    • API는 접근 권한이 인가된 사람에게만 서버와 데이터베이스에 접근하도록 함.
  • 모든 요청과 응답을 표준화
    • 아이폰을 쓰던 갤럭시 폰을 사용하든 상관없이 동일한 API를 사용하기 때문에 클라이언트의 요청과 서버의 응답을 하나의 API로 표준화 할 수 있음.
  • REST API
    • REST API(REpresentational State Transfer)는 웹상에서 사용되는 여러 리소스를 HTTP URI로 표현하고, 해당 리소스에 대한 행위를 HTTP Method로 정의하는 방식
    • 즉, 리소스(HTTP URI로 정의됨)를 어떻게 하겠다(HTTP Method + Payload)를 구조적으로 깔끔하게 표현하는 방법

2-3. REST API란?

  • REST (Representational State Transfer): 2000년 로이 필딩(Roy Fielding)의 논문에서 제안된 웹 아키텍처 스타일. 단순히 프로토콜이 아니라, 웹에서 자원을 어떻게 정의하고 접근할지에 대한 설계 규칙 집합임.
  • REST API: REST의 제약 조건을 충실히 따르는 API를 의미. 클라이언트는 요청(Request)을 통해 자원(Resource)을 조작하고, 서버는 그 결과를 응답(Response)으로 돌려주는 방식으로 동작함.
  • REST API의 핵심 제약 조건:
    • 자원의 식별(URI 기반): 모든 자원(Resource)은 고유한 URI (Uniform Resource Identifier)로 표현.
      예: /students/1 → ID가 1인 학생이라는 자원을 의미.
    • 표현(Representation)을 통한 자원 조작: 자원 자체가 아니라 자원의 **표현(JSON, XML 등)**을 주고받음. 같은 자원이라도 상황에 따라 다른 표현 형식으로 응답 가능.
    • 무상태성(Stateless): 서버는 클라이언트의 상태를 저장하지 않음. 각 요청은 독립적으로 처리되며, 필요한 모든 정보는 요청 메시지 안에 포함되어야 함.
    • 일관된 인터페이스(Uniform Interface): HTTP 메서드와 상태 코드를 활용해 요청과 응답 형식을 표준화함.
      • GET: 자원 조회
      • POST: 자원 생성
      • PUT: 자원 전체 수정
      • PATCH: 자원 일부 수정
      • DELETE: 자원 삭제
    • 계층적 구조(Layered System): 클라이언트는 중간 서버(프록시, 로드 밸런서 등)의 존재를 알 필요가 없으며, 요청은 여러 계층을 거쳐도 동일하게 동작해야 함.
    • 캐시 가능(Cacheable): 응답 결과는 캐싱할 수 있어야 하며, 이를 통해 서버 부하를 줄이고 성능을 향상시킴.
  • REST API의 특징
    • 균일한 인터페이스 (Uniform Interface)
      • HTTP 표준 프로토콜을 따르는 플랫폼에서 사용 가능
      • URI로 지정한 리소스에 대한 조작을 통일되고 한정적인 인터페이스로 수행
      • HTTP Method(GET, POST, PUT, DELETE)를 통해 리소스를 처리
    • 무상태성 (Stateless)
      • 서버에 클라이언트의 상태 정보를 저장하지 않음
      • 각 요청에 필요한 모든 정보를 담아서 전송
      • 서버는 각각의 요청을 완전히 별개의 것으로 인식하고 처리
    • 캐시 가능 (Cacheable)
      • HTTP의 캐싱 기능을 적용할 수 있음
      • 캐시 사용을 통해 응답시간 향상 및 서버 부하 감소 가능
      • GET 요청은 캐시가 가능하지만, POST 요청은 캐시가 불가능
    • 계층화 시스템 (Layered System)
      • 클라이언트는 서버에 직접 연결되었는지, 중간 서버를 통해 연결되었는지 알 수 없음
      • 중간에 프록시, 게이트웨이 같은 중간 매체를 사용할 수 있음
      • 보안, 로드 밸런싱, 암호화, 사용자 인증 등을 추가할 수 있음
    • 자체 표현 구조 (Self-descriptiveness)
      • REST API 메시지만 보고도 이해할 수 있는 자체 표현 구조
      • JSON을 이용한 메시지 포맷 사용
      • URI만 보고도 어떤 정보를 제공하는지 알 수 있음
    • 클라이언트서버 구조 (Client-Server Architecture)
      • 클라이언트와 서버가 독립적으로 분리되어 있음
      • 서버는 API를 제공하고, 클라이언트는 사용자 인증이나 컨텍스트(세션, 로그인 정보)를 직접 관리
      • 서로 간의 의존성이 줄어듦

  • REST API의 장점:
    • 직관적이고 간단한 구조로, 별도의 인프라 구축이 필요 없음
    • HTTP 프로토콜을 사용하므로 REST API 사용을 위한 인프라를 따로 구축할 필요가 없음
    • HTTP 표준 프로토콜을 따르는 모든 플랫폼에서 사용 가능
    • 서버와 클라이언트의 역할을 명확하게 분리할 수 있음
    • REST API 메시지가 의도하는 바를 명확하게 나타내므로 의도를 쉽게 파악할 수 있음
    • 여러 가지 서비스 디자인에서 생길 수 있는 문제를 최소화함
    • 서버와 클라이언트가 독립적으로 진화할 수 있음

  • REST API의 단점:
    • 표준이 존재하지 않아 정의가 필요함
    • HTTP Method 형태가 제한적임 (GET, POST, PUT, DELETE 등)
    • 구형 브라우저에서 PUT, DELETE를 사용하지 못하는 경우가 있음
    • 관련 도구들이 부족할 수 있음
    • 구축한 API에 대한 문서화가 필요하며, 관리가 어려울 수 있음
    • 클라이언트가 원하는 정확한 데이터를 호출하기 위해서는 여러 번의 요청이 필요할 수 있음
    • RESTful API 설계의 어려움이 있음 (API 버전 관리, 네이밍 등)
    • 대규모 데이터 전송 시 처리 방법 제한적
    • 표준이 없어 관리가 어려울 수 있음

정리하면, REST API는 웹의 자원을 URI로 정의하고, HTTP 메서드를 통해 표준화된 방식으로 자원을 다루는 API라고 할 수 있으며, 이것이 오늘날 가장 널리 쓰이는 웹 API 형식임.

B. REST API

1. REST API의 구성 요소

1-1. 자원(Resource)

  • API가 다루는 대상을 의미.
    • 예: 학생, 게시글, 주문과 같은 명사형 개체
  • 자원은 URI(Uniform Resource Identifier)로 식별함.
    • 예: /students, /students/1, /orders/2024-09-01
  • 컬렉션/개별 자원 구분 필요:
    • /posts(컬렉션) ↔ /posts/10(개별).
    • 하위 관계는 /students/1/courses처럼 계층적으로 표현하도록 함.
  • URI는 행위가 아닌 대상만 표현함(명사 사용). 소문자·복수형·하이픈 권장.
    • 예: /createStudent(지양) → /students(선호).
  • 조회 조건·페이징·정렬은 쿼리 파라미터로 분리함.
    • 예: /students?major=cs&page=2&size=20&sort=name,asc.
  • 자원의 상태(state)는 서버가 보관하고, 클라이언트는 상태의 표현(representation)을 주고받음.

1-2. 행위(Method)

  • 자원에 대해 무엇을 할지를 나타내는 HTTP 메서드!
    • GET: 조회. 안전(safe)·멱등(idempotent). 본문 없이 헤더·쿼리로만 의미 전달.
    • POST: 생성 또는 컬렉션에 하위 자원 추가. 비멱등. 성공 시 보통 201 Created와 Location 헤더 반환.
    • PUT: 전체 교체(대표 케이스). 멱등. 누락 필드는 삭제로 간주될 수 있음.
    • PATCH: 부분 수정. 일반적으로 비멱등(설계에 따라 멱등 가능). 수정 범위를 본문으로 명시.
    • DELETE: 삭제. 멱등. 이미 삭제된 자원에 대해 재요청해도 결과는 동일해야 함.
  • 참고 메서드: HEAD(본문 없이 헤더만), OPTIONS(서버가 지원하는 메서드 확인).
  • 메서드에 따른 대표 상태 코드:GET → 200 OK / 없으면 404 Not FoundPOST → 201 Created(+ Location)PUT/PATCH → 200 OK 또는 본문 없으면 204 No ContentDELETE → 204 No Content

1-3. 표현(Representation)

  • 자원의 상태를 어떤 형식으로 전달할지를 의미.
  • 본문 형식은 주로 JSON 사용하나, 경우에 따라 XML/CSV/바이너리 등 사용 가능.
  • 콘텐츠 협상(Content Negotiation): 클라이언트는 Accept로 원하는 형식 표시, 서버는 Content-Type으로 실제 형식 선언. 예: Accept: application/json, Content-Type: application/json.
  • 표현은 자기 기술적(self-descriptive)이어야 함. 필드 의미가 명확하고, 필요한 링크(예: self, next)나 메타데이터(페이지 정보 등)를 포함할 수 있음.
  • 오류는 일관된 형식으로 반환하는 것이 바람직함. 예: status, code, message, details 필드 구성 제안.

1-4. REST API 요청/응답 구조

  • 요청(Request) = 메서드 + URI + 헤더 + 본문(선택)
    • 인증·인가: Authorization 헤더 사용(JWT/Bearer 등).
    • 본문 전송 시 Content-Type 필수. 클라이언트 수용 형식은 Accept로 명시.
    • 캐시·조건부 요청: If-None-Match(ETag), If-Modified-Since 활용.
  • 응답(Response) = 상태 코드 + 헤더 + 본문(선택)
    • 성공: 200/201/204 중심. 생성 시 Location으로 새 자원 URI 제공.
    • 오류: 4xx(클라이언트 오류), 5xx(서버 오류). 가능하면 원인과 수정 방법을 본문에 명확히 기술.
    • 캐시·버전 관리: ETag, Cache-Control, 필요 시 경로 버전(/v1) 또는 헤더 버전 전략 사용.

3. REST API 설계 방식

  • 리소스에 대한 행위는 HTTP Method(POST, GET, PUT, DELETE)로 표현
  • /(슬래시)는 계층 관계를 나타낼때 사용
  • URI 마지막 문자에 /(슬래시)를 사용하지 않음
  • URI에 _(underscore)는 사용하지 않도록 함. 또한 영어 대문자보다는 소문자 사용을 권장
  • 그리고 가독성을 위해서 긴 단어는 잘 사용하지 않음
  • URI에 동사는 GET, POST와 같은 HTTP Method를 표현하기 때문에 동사가 아니라 명사를 사용 권장.
  • URI에 파일의 확장자(예를들어 .json , .JPGE)를 포함 시키지 않음.
  • self-descriptiveness: RESTful API는 그 자체만으로도 API의 목적이 무엇인지 쉽게 알 수 있게 설계.
