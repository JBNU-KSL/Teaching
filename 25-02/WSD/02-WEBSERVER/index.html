1. Accessing Web Services

A. How can we access web services?

⇒ Access through a web browser!

⇒ how?

⇒ Every computer has its own unique address!

B. How to access a web service

  • Connect to the server of your choice via IP address!
  • IPv4 is anaddress system consisting of 4 blocks of*8 bits, and is currently the most commonly utilized address!
  • The 8 bits can have values from 0 to 255 (2^8 = 256)in decimal.
  • This means that there are 4 blocks of 0-255!
  • 000.000.000.000.000 ← 4 blocks like this!
  • Each device (computer) has a unique address!

  • But, wouldn't it be hard to memorize the addresses of all computers/servers...? ⇒ That's where DNS (Domain Name Server) comes in!
  • DNS allows you to connect to other computers (servers) via a domain name that is easily memorized by humans, instead of having to memorize IP addresses.
  • When a user makes a request with a domain name, the Domain Name Server returns the IP of the server with that domain , and the content of the web server is received through that IP!

  • IP addresses or DNS can be utilized to efficiently distribute IPs.
  • However, no matter how efficiently you distribute IPs, the total number of available IPs is limited to about 4.2 billion!
  • This is where Private IPs (internal networks/private networks)come in!
  • If you create an IP that can only be used internally, it doesn't matter if it's duplicated by external IPs, allowing you to use resources efficiently!

  • There are times when you need to access your Private IP from the outside.
  • If you utilize a device with a private IP as a web server, you need to forward requests from the router to that device.
  • This is called port forwarding!
  • You can forward requests from a specific port on your router to a specific port on your private IP!

D. URLs

  • In hypertext and HTTP, URLs are one of the core concepts of the Web. URLs are the mechanism used by browsers to retrieve resources published on the Web.
  • URL stands for Uniform Resource Locator, which means that a URL is a unique address of a given resource on the web.
  • In theory, each valid URL points to a unique resource, which can be an HTML page, CSS document, image, etc.
  • In practice, there are a few exceptions, the most common being URLs that point to resources that no longer exist or have been moved.
  • The resources represented by URLs and the URLs themselves are handled by the web server, so it's up to the web server owner to carefully manage those resources and their associated URLs, and that's important!
  • In other words, when you use a web browser on your computer, you call the address (URL/ URI) of a specific server, and you connect to the server!

2. How does a web browser work?

A. How does a web browser work?

  • You connect to a server by calling the address (URL/ URI) of a specific server using a web browser on your computer!
  • The server sends a request to the client, which triggers the web browser!
  • The browser requests the resource (Resource) selected by the user from the server and renders the response (Response) received from the server to the browser!
  • The resource can be in the form of an HTML document, PDF, multimedia, etc. and the address of the resource is a URI (Uniform Resource Identifier).

B. How a Web Browser Works

1.User: Enter the URL address of the web page you want to find through your browser.

2.DNS server: retrieves the domain name from the URL address entered by the user, finds the IP address that matches the domain name, and passes it along with the URL information entered by the user.

3.HTTP protocol: The IP address and web page URL information received in this way is used to create an HTTP request message using the HTTP protocol.

4.TCP/IP protocol: This HTTP request message is sent to the corresponding IP computer via the Internet using the TCP protocol, and the HTTP request message that arrives is converted into web page URL information using the HTTP protocol.

5.Web server: It searches for and finds the data corresponding to the converted information, and generates an HTTP response message. This HTTP response message is then sent to the user's computer via the Internet using the TCP protocol, and the arriving HTTP response message is converted into web page data using the HTTP protocol.

6.Web browser: the converted data is output to the web browser for the user to view!

C. Structure of a Web Browser

  • The user interface (UI) layer is pretty much everything you see in a browser, meaning all UI other than the window that displays the requested page. There are UIs like the address bar, back, forward, refresh, bookmarks, and preferences.
  • The browser engine acts as an intermediary between the user interface and the rendering engine. If you hit the refresh button on the user interface layer, the browser engine understands it and performs the refresh command.
  • The rendering engine is responsible for parsing the HTML, CSS, and JavaScript and drawing the page based on the results. Each browser uses a different engine: Chrome, Opera, and Edge use Blink, Firefox uses Gecko, Internet Explorer uses Trident, and Safari uses WebKit.
  • The network layer is the layer used to obtain external resources and send requests to servers using protocols like HTTP or HTTPS.
  • The JavaScript interpreter is responsible for interpreting and executing JavaScript. The most famous engine is Google's V8, which powers Chrome.
  • The UI backend handles the UI, which follows the interface of the operating system the browser is running on. It's easy to see how Alert or Select boxes behave differently on different operating systems.
  • The data store is the layer that the browser itself uses to store data locally, such as on the hard disk, and is used to access and store cookies, local storage, session storage, IndexedDB, Web SQL, filesystems, etc.

D. User Interface

  • The part of the website that most closely interacts with the user, such as the address bar, previous/next buttons, bookmark menu, etc.

E. Browser Engine

  • The engine that controls the behavior between the user interface and the rendering engine.
  • Translates HTML documents and other supported web pages into a visual representation on the user's device, and implements the Document Object Model (DOM) data structure.
  • Also known as the layout engine, it is closely related to the rendering engine, so the browser engine and rendering engine are sometimes referred to together as the browser engine.
  • Each web browser uses a dedicated browser engine
    • Gecko: A browser engine created by the Mozilla Foundation, which powers Firefox.
    • Webkit: A browser engine derived from KHTML, which powers Safari.
    • Blink: A browser engine derived from Webkit, which powers Chrome and Opera.
    • Trident: Microsoft's browser engine, which powers Outlook Express, Microsoft Outlook, etc.
    • There are many other browser engines.

F. Rendering Engine

  • The engine that outputs the requested content to the screen
  • It can display HTML and XML documents and images, parsing HTML and CSS and finally drawing them on the screen. (Plugins or browser extensions can be used to display other types, such as PDFs).

G. Communication

  • Used to make network calls, such as HTTP requests, and is usually a platform-independent interface that runs under the hood of each platform.

H. JavaScript Interpreter

  • Interpreters that interpret and execute JavaScript code
  • JavaScript is an Interpreted Language that parses code by reading it line by line from top to bottom, which is why it needs an interpreter.
  • It's also called a JavaScript engine.
  • Each web browser has its own JavaScript engine.
  • Rhino: This is an open-source engine maintained by the Mozilla Foundation and is characterized by the fact that it is developed in Java.
  • SpiderMonkey: The first JavaScript engine, it supported Netscape Navigator and now supports Firefox.
  • V8: An open source engine developed by Google that powers Google Chrome. << Remember this!
  • JavascriptCore: An engine developed by Apple that was originally developed for the Webkit framework, but now supports Safari and React Native Apps.
  • Chakra : An engine developed by Microsoft, which supports the Edge browser.

I. UI Backend

  • Responsible for drawing the Render Tree analyzed by the rendering engine to the browser.
    • Draws basic widgets such as Select and Input windows.
    • It is a general interface that is not specified by the platform and uses the OS user interface system.
  • User Interface Types
    • Command Line Interface (CLI)
      • An interface that requires only textual commands to be entered and processed (Terminal on mac, Command Prompt on Windows).
    • Batch Interface
      • A non-interactive user interface in which the user specifies all the details of a batch job prior to batch processing and receives the output when all processing is complete.
    • Often used for processing large amounts of data in large systems.
    • GUI (Graphic User Interface)
      • : An interface that allows intuitive input such as clicking icons using a mouse, keyboard, monitor, etc.

J. Web Storage

  • It is possible to separate local storage, which is permanent storage, and session storage, which is temporary storage, to choose according to the application environment.
  • Features of Web Storage
    • It allows the web browser to store data directly, allowing more information to be stored securely on the user's side.
    • It also reduces network traffic costs because the information is stored by the client rather than sent to the server.
    • There is only one web storage per origin.
    • All web pages belonging to the same origin can access the same data because they store the same data.

K. Local Storage

  • Local storage objects store data without an expiration date.
  • Data stored in local storage does not disappear when you close the browser or reboot the computer.
  • It can be stored and retrieved through a collection of Windows global objects called localStorage (window.localStorage).
  • A separate local storage is created for each domain.

    = Data can be shared globally as long as the domain is the same.

L. Session Storage

  • Session storage objects store data for only one session.
  • Data stored in session storage is only retained within the context of the browser being browsed, so the data is lost when the browser tab is closed.
  • It can be stored and retrieved through a collection called sessionStorage in Windows Global Objects (window.sessionStorage).
  • Different browser contexts have different session storage.
  • = Different browsers have different contexts, so they can't share data with each other.
  • Browsing: The act of running a browser program to find information on the internet.
  • Browser context: the environment in which the browser displays a document
  • Each browsing context has the source of the active document and the history of the visited documents.

3. Different types of servers

A. Types of servers

B. Frontend

  • The user interface that the user can directly interact with, such as layouts, menus, design elements, buttons, images, and graphs!
  • In other words, the frontend is the primary interface that users see, so it needs to be easy to use and pleasing to the eye to keep them engaged.
  • That's why front-end developers need to build an intuitive and easy-to-use user interface (UI) with user experience as a core value!
  • Bothpublishers anddevelopers are involved in creating a website or application, but their roles and scope of work are different!!! << Remember!
  • Publishersare responsible for implementing the design aspects of a website or application by utilizing web technologies such as HTML, CSS, and JavaScript based on design. Therefore, they need basic knowledge and skills in design and front-end technologies! They mainly implement the static contentof a website (HTML/CSS) and do not implement dynamic functionality (JavaScript).
  • Front-end developers, on the other hand, are responsible for implementing the user interface (UI) of a website or application. They also implement the communication, including sending and receiving data to and from the server via APIs, and interfacing with the backend to implement dynamic functionality!
  • As a result, front-end developers are more deeply involved in the creation of a website or application than publishers, so the development skills and technical knowledge required of front-end developers are higher than those of publishers.
  • Front-end server
    • Serves static files such as HTML, CSS, and JavaScript to the user's web browser.
    • Processes client requests and delivers the required data to the API server.
    • Present the UI of a web page or application to users
    • Manages the routing of your web application
    • Improve initial page loading speed with front-end-side rendering capabilities
    • In other words, provide files that can be rendered to the user!
  • Key front-end technologies
    • Popular technologies these days: Angular, React, and Vue
    • Angular: An open-source JavaScript framework created by Google and used to develop single-page applications (SPAs). It offers a wide range of features, but its complexity makes it ideal for long-term projects.
    • React: A library for creating dynamic UIs that is highly flexible and can be combined with a variety of tools. Great for fast, modern application development
    • Vue.js: A popular framework with a pre-coded structure that allows for fast development and clear command processing. Ideal for small applications or first-to-market.
  • Trends in the frontend?
    • Progressive web apps (PWAs)are apps that use web technologies to behave like native apps.
    • PWAs use web technologies to behave like native apps.
    • Can be developed faster than native app development
    • Offer native app features like push notifications and more
    • Work offline and can be developed with common web technologies
    • PWAs allow businesses to offer services on mobile and web simultaneously
  • Front-end applications (SPA vs MPA)
    • The latest front-end applications are mainly divided into SPA (Single Page Application) and MPA (Multi page Application)!
    • In the past, web development was mainly based on MPA, but in recent years, it has been developed with SPA! => User convenience
    • However, both MPA and SPA have their own advantages and disadvantages, so you should choose well depending on the nature of the application.
  • Multi Page Application (MPA)
    • Whenever the user navigates to a page, new HTML, CSS, and JavaScript files are fetched from the server .
    • The entire page is refreshed and re-rendered with new data after each page navigation.
    • MPA is beneficial for applications with complex structures or websites that require a lot of functionality.
    • Users can have a unique URL on each page, which is beneficial for search engine optimization (SEO).
    • Common examples: Amazon, eBay, Wikipedia, Naver, etc.
    • Advantages of MPAs
      • 1. Search Engine Optimization (SEO)
        • Unlike SPAs, MPAs are compatible with crawlers, making them ideal for SEO optimization.
        • Since there are multiple pages, you can easily insert keywords on each page.
      • 2. Scalability
        • MPAs are easily scalable for different services or products because they are structurally flexible enough to add an infinite number of new pages.
        • It's also easy to modify and update each page.
      • 3. Availability of different solutions
        • MPA development typically requires a small technology stack.
        • You can also utilize a wide range of ready-made solutions.
      • 4. Provides analytical information
        • MPAs can analyze the market with data from the internet.
        • It can collect important analytics data about web activity, which can help companies build strategies based on it.
    • Cons of MPAs
      • 1. Slow speed
        • Unlike SPAs, MPAs require the page to reload every time a user clicks on a tab.
        • This can slow down performance and speed.
      • 2. Longer development time
        • MPAs require coding both front-end and back-end development from scratch.
        • The separation of front-end and back-end can create additional challenges.
      • 3. Maintenance and security issues
        • MPAs are complex to maintain and update each page on an ongoing basis.
        • Developers must ensure that each page is secure, which can make maintenance cumbersome.

  • Single Page Application (SPA)
    • Receives all data (HTML/CSS/JavaScript) when a user first clicks on a page.
    • Once the web is opened, it is organized in such a way that it receives and acts on partial elements (data) without refreshing the page for each click.
    • Single-page websites provide a comfortable content experience for users and provide a smooth, easy-to-work-with web experience.
    • Examples: Facebook, Gmail, Google Maps, Twitter, etc.

    • Advantages of SPAs
      • 1. Fast speed and performance
        • SPAs load data only once and then dynamically update only the parts that are needed, so pages load quickly.
        • The user experience is smooth and seamless as the entire page is not refreshed when switching pages.
      • 2. Simple development and maintenance
        • SPA has a simple development structure because it handles all functionality on one page.
        • Clear separation of front-end and back-end makes development and maintenance easy.
      • 3. Mobile-Friendly
        • SPAs are optimized for performance on mobile devices and can provide a user experience similar to a native app.
      • 4. Caching efficiency
        • Once loaded, the content is cached in the browser and loads quickly on subsequent page transitions.

    • Cons of SPAs
      • 1. SEO optimization issues
        • Because the content is loaded dynamically, SPAs can make it difficult for search engine crawlers to index the page.
        • Requires additional server-side rendering or framework settings for SEO optimization.
      • 2. Slow first load speed
        • All necessary resources (HTML, CSS, JavaScript) are downloaded at once during initial loading, which can result in slow initial loading speeds.
      • 3. Browser compatibility issues
        • SPAs are developed for modern browser environments, which can cause performance issues on older or non-standard browsers.
      • 4. Complex state management
        • Because SPAs handle a lot of functionality on a single page, state management can be complicated, requiring additional libraries or frameworks to handle it.

  • Static website vs Dynamic website
    • Static website (Static): A website that serves fixed content. HTML and CSS files are pre-written, stored on the server, and delivered as is when a user requests them.
    • Dynamic website (Dynamic): A website where the server processes data in real time and generates pages based on user requests. The content changes dynamically by interfacing with a database.
    • How static websides work
      • A static website uses HTML, CSS, and image files that are pre-stored on the server, and when a user requests them, the server delivers them as is.
      • All users receive the same content, and the page is static.
      • The content is pre-generated and uses fewer resources on the server.
      • Examples: Blogs, about us pages, etc.
      • Pros:
        • Fast loading speed: Fast loading because no data needs to be processed on the server.
        • Low cost: Less server resources are consumed, making it cheaper to run.
        • SEO-friendly: Pages are static, making them easier for search engine crawlers to index.
      • Cons:
        • Difficult to update: You must manually modify the HTML file to change content.
        • Limited functionality: Difficult to provide dynamic user interaction features.
    • How dynamic websites work
      • A dynamic website receives a request from a user, and the server communicates with a database to fetch the necessary data and generate HTML in real time.
      • Different users may receive different data on different pages, and the content changes based on their requests.
      • This means that the user sees a web page that changes based on their request!
      • Pros:
        • Flexible content management: CMS makes it easy for non-developers to manage content.
        • Real-time data delivery: Real-time data can be processed and delivered on demand.
      • Cons:
        • Slow loading speeds: The server has to process the data every time, which makes page loading a bit slower.
        • Complex maintenance: Requires servers, databases, security management, and more, making maintenance complex.
      • Implementing a dynamic website
        • Client-Side Rendering (CSR):
          • CSR is a method of initially receiving an HTML document or static file with no data from the server, and then using JavaScript to request the necessary data directly from the client (usually on the backend).
          • It uses JavaScript to render the page directlyin the browser (JS Engine + UI), and the server plays a minimal role. All logic (data processing, template rendering, routing, etc.) is done on the client side, and the server is responsible for providing data in the form of APIs.
        • Server-Side Rendering (SSR):
          • SSR is a method where theserver delivers finished HTML to the client, complete with data inserted dynamically.
          • The server preprocesses the data requested by the client and serves the completed HTML page.
          • Server Rendering does all the work (data processing, template creation, etc.) on the server before the browser receives a response, so there are no additional data requests or round trips.
          • All requests are processed on the web server, so the client receives the finished HTML directly and displays it in the browser.

C. Backend

  • The backend performs a variety of technical tasks, including server configuration, performance optimization, and database design.
  • It is also responsible for the technology of constantly developing and managing the server, as it must constantly think about how to store and manage data, how to reduce data processing time, and how to cope with an increase in the number of users.
  • Furthermore, we need to process user data received from the frontend, so we organize business logicand design databases to effectively handle data.
  • Technology stack used in the backend
    • Language
      • Java: An object-oriented programming language that appeared in 1995 and is characterized by the need to specify data types. It is used as a standard framework for e-government in Korea and has many references.
      • Python: An object-oriented language with a simple syntax and easy to learn, used in a variety of fields. It grew in popularity in the 2010s and is supported by many libraries and frameworks.
      • JavaScript: An object-oriented language specialized for the web, often used in front-end development. Its easy syntax makes it highly accessible, and its popularity in web development has skyrocketed in recent years.
      • TypeScript: A static type superset of JavaScript released by Microsoft in 2012. It catchestype errors at the compile (transpile) stage, which is advantageous for large/long-term projects, and it translates to modern JS and runs in browsers and Node.js. Widely used on both front-end (React, Vue, Angular) and back-end(NestJS, Express, etc.).
    • Frameworks
      • Java:
        • Spring Boot: A framework for Java-based backend development, ideal for microservices and large-scale application development with quick and easy setup.
        • Dropwizard: A lightweight framework for building microservices in Java that specializes in rapid development and deployment.
      • Python:
        • Django: A full-stack web framework based on Python that allows you to quickly build backend logic, database processing, user authentication, and more.
        • Flask: A lightweight framework that allows you to quickly develop backends with minimal setup, giving you the flexibility to add only the features you need.
        • FastAPI: An up-and-coming Python asynchronous web framework with fast performance and automatic documentation (Swagger, OpenAPI support). It's ideal for building microservices and AI/data APIs, as it heavily utilizes type hints to increase development productivity and reliability.
      • JavaScript:
        • Node.js + Express: A server-side runtime environment based on JavaScript (Node.js) and a framework that makes it easy to use (Express), optimized for asynchronous processing and fast server responses.
        • NestJS: A backend framework based on Node.js that supports modularity and structured code writing, making it ideal for large-scale application development.

  • Recent backend trends?
    • A recent trend in the backend space is "serverless," which is one of the models of cloud computing and is defined as "an architecture where developers do not need to manage servers themselves.
    • If a backend developer expects 10,000 users on a server and buys the right amount of capacity, he or she will pay the same amount whether there are 10,000 or zero users. This situation can cause losses, large or small.
    • Serverless, however, dynamically allocates resources on the server. When there are no users, it waits without allocating resources, and only when requests come in, it allocates resources to serve them.
    • The result is a more efficient use of scarce resources, especially since the server is fully managed by the company providing the cloud, and you don't have to deal with server issues such as backup, security, etc.
    • Recently, there has also been a lot of interest in server-side rendering (SSR)!!!
    • Server-Side Rendering (SSR) is a method of pre-rendering the content of a web page on the server and then delivering the finished HTML to the client. This method is beneficial for SEO (search engine optimization) because it allows for faster first page load times and a complete page to be displayed on the screen before any JavaScript is executed on the client side.
    • SSR specifically improves SEO performance by making it easier for search engines to crawl your web pages. It also improves user experience because the initial screen is displayed quickly. However, it can increase the load on your server and may require more complex setup than client-side rendering (CSR).
    • Major frameworks include Next.js (React), Nuxt.js (Vue.js), and others.

D. Structure of the server

  • Basic server structure
    • Web server: The server that serves web pages to users and processes HTTP requests. Examples include Apache, Nginx, etc.
    • Application server: A server that handles web application logic, receiving requests from clients and interacting with databases. Examples include Tomcat, Node.js, and others based on Java.
    • Database server: A server that stores and manages data and works in conjunction with the application server to serve data. Examples include MySQL, PostgreSQL, and others.
    • File server: A server that stores and shares files, managing and distributing files over a network. Examples include FTP servers.
    • Proxy servers: Acts as an intermediary between clients and other servers, and improves performance through increased security, caching, and more.
  • Modern server structure
    • 1. Front-end server:
      • Serves static files like HTML, CSS, and JavaScript to the client side.
      • Primarily provides the content and interface that users see in the browser.
      • Example: Web servers that serve static files, such as Nginx, Apache, etc.
    • 2. Backend servers:
      • Servers that process requests from clients and interact with databases.
      • It is responsible for business logic and data management, and also serves as an API server.
      • Examples: Application servers such as Express (Node.js), Spring Boot (Java), Django (Python), etc.
    • 3. Serverless:
      • An approach that leaves the management of server infrastructure to the cloud provider and dynamically allocates resources only when needed.
      • Using backend functions provided by the cloud to run applications with minimal server management.
      • Examples: AWS Lambda, Azure Functions, Google Cloud Functions.
