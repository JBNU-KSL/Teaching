1. 웹 서비스에 접근

A. 어떻게 웹 서비스에 접근할 수 있을까?

⇒ 웹 브라우저를 통해 접근!

⇒ how?

⇒ 모든 컴퓨터는 각각 고유한 주소를 갖고 있음!

B. 웹 서비스에 접근하는 방법

  • IP주소를 통해 내가 원하는 서버로 접속하는 방식!
  • IPv4는 *8비트씩 4블럭으로 이루어져 있는 주소체계로 현재 주로 활용되고 있는 주소!
  • 8비트는 10진수로 표현하면 0~255 (2^8 = 256) 까지의 값을 갖고 있음!
  • 즉, 0~255가 4블럭으로 이루어져 있음!
  • 000.000.000.000 ← 이런식으로 4개의 블록이로 이루어져 있음!
  • 각 디바이스 (computer) 마다 고유한 주소를 갖고 있음!

  • 하지만, 모든 컴퓨터/서버의 주소를 외우는 것은 힘들지 않을까…? ⇒ 그래서 나온 것이 DNS (Domain Name Server) !
  • DNS를 통해 IP주소를 일일이 기억하지 않고, 사람이 쉽게 기억할 수 있는 도메인 네임을 통해 다른 컴퓨터(서버)에 접속할 수 있음
  • 사용자가 도메인 네임으로 요청하게 되면, Domain Name Server가 해당 도메인을 가진 서버의 IP를 반환하게 되고, 해당 IP를 통해 웹서버의 콘텐츠를 받아오는 방식!

  • IP주소나 DNS를 활용하여 IP를 효율적으로 배분할 수 있음.
  • 하지만, 아무리 효율적으로 IP를 분배하더라도 이용 가능한 전체 IP개수가 약 42억 개로 제한되어 있음!
  • 이를 위해, Private IP (내부 망/ 사설 망)가 등장!!
  • 내부에서만 사용할 수 있는 IP를 만든다면, 외부의 IP와 중복이 되어도 상관이 없어 자원을 효율적으로 사용할 수 있음!

  • 외부에서 Private IP에 접근이 필요할 때가 있음
  • Private IP를 할당받은 기기를 웹서버로 활용한다면 공유기로 들어온 요청을 해당 기기로 전달해줘야 함
  • 이럴 때 포트포워딩이라는 방법을 사용!
  • 공유기의 특정 포트로 들어온 접속을 Private IP의 특정 포트로 요청을 전달!

C. Port Number

  • 하나의 컴퓨터에는 여러 개의 서버(프로그램)가 설치 될 수 있음.
  • 클라이언트가 컴퓨터에 접속할 때 어떤 서버와 접속하려고 하는지는 구분할 수 있는 방법이 필요하기 때문에 Port(포트) 개념이 활용됨!
  • 즉, Port(포트) 개념은 클라이언트 프로그램이 네트워크 상의 특정 서버 프로그램을 지정하는 방법임!

    1.네트워크 상에서 통신을 할 때 IP를 토대로 해당 서버가 있는 컴퓨터에 접근

    2.그런데 대부분의 경우 하나의 컴퓨터에는 여러 개의 서버가 실행될 수 있다.

    3.컴퓨터에 여러 개의 서버가 실행되고 있다면, 어느 서버에 접속해야 하는지 컴퓨터에게 알려주어야 한다. 이때 사용되는 것이 포트 번호

  • 미리 지정된 포트 번호
    • 0번 ~ 1023번 : Well-known port, 잘 알려진 포트
    • 1024번 ~ 49151번 : registered port, 등록된 포트
    • 49152번 ~ 65535번 : dynamic port, 동적 포트
    • HTTP (80), HTTPS(443)
    • SSH (22), FTP (20), TELNET (23), SMTP (25)
    • https://wiki1.kr/index.php/%ED%8F%AC%ED%8A%B8

D. URL

  • 하이퍼텍스트와 HTTP에서 URL 은 웹의 핵심 개념 중 하나로, URL은 웹에 게시된 리소스를 검색하기 위해 브라우저에서 사용하는 메커니즘
  • URL은 Uniform Resource Locator의 약자로, URL은 웹에서 주어진 고유 리소스 주소
  • 이론적으로 각각의 유효한 URL은 고유한 리소스를 가리키며, 이러한 리소스는 HTML 페이지, CSS 문서, 이미지 등이 될 수 있음.
  • 실제로는 몇 가지 예외가 있으며 가장 일반적인 예외는 더 이상 존재하지 않거나 이동된 리소스를 가리키는 URL
  • URL로 표시되는 리소스와 URL 자체는 웹 서버에서 처리되므로 해당 리소스와 관련 URL을 신중하게 관리하는 것은 웹 서버 소유자에게 달려 있어, 중요하게 다뤄야 함!
  • 즉, 내 컴퓨터에서 웹 브라우저를 사용해, 특정 서버의 주소 (URL/ URI)를 호출하여, 서버에 접속하게 됨!

2. 웹 브라우저 동작 방식

A. 웹 브라우저는 어떻게 동작하는가?

  • 즉, 내 컴퓨터에서 웹 브라우저를 사용해, 특정 서버의 주소 (URL/ URI)를 호출하여, 서버에 접속하게 됨!
  • 서버는 Client에게 를 보내주어 웹브라우저가 동작하게 함!
  • 브라우저는 사용자가 선택한 자원(Resource)을 서버에 요청(Request)하고, 서버로부터 받은 응답(Response)을 브라우저에 렌더링(Rendering) 수행!
  • 여기서 자원은 HTML 문서, PDF, 멀티미디어 등 여러 형태일 수 있으며, 자원의 주소는 URI(Uniform Resource Identifier)로 되어있음!

B. 웹 브라우저 동작 방식

1.사용자: 브라우저를 통해 찾고 싶은 웹 페이지의 URL 주소를 입력

2.DNS 서버: 사용자가 입력한 URL 주소 중 도메인 네임을 검색하고, 도메인 네임에 일치하는 IP 주소를 찾아, 사용자가 입력한 URL 정보와 함께 전달

3.HTTP 프로토콜: 이렇게 전달 받은 IP 주소와 웹 페이지 URL 정보는 HTTP 프로토콜을 사용해 HTTP 요청 메세지를 생성

4.TCP/IP 프로토콜: 이 HTTP 요청 메세지는 TCP 프로토콜 사용해 인터넷을 거쳐 해당 IP 컴퓨터로 전송되고, 도착한 HTTP 요청 메세지는, HTTP 프로토콜을 이용해 웹 페이지 URL 정보로 변환

5.웹 서버: 변환된 정보에 해당하는 데이터를 검색하여 찾아낸 뒤, HTTP 응답 메세지를 생성한다. 이 HTTP 응답 메세지는 다시 TCP 프로토콜을 사용해 인터넷을 거쳐 사용자의 컴퓨터로 전송되고, 도착한 HTTP 응답 메세지는, HTTP 프로토콜을 이용해 웹 페이지 데이터로 변환

6.웹 브라우저: 변환된 데이터가 웹 브라우저에 출력되어 사용자가 볼 수 있음!

C. 웹 브라우저의 구조

  • 사용자 인터페이스(UI) 레이어는 브라우저에서 볼 수 있는 거의 모든 것으로, 요청한 페이지를 보여주는 창 외의 모든 UI를 의미합니다. 주소창, 뒤로가기, 앞으로 가기, 새로고침, 북마크, 환경설정과 같은 UI가 있습니다.
  • 브라우저 엔진은 사용자 인터페이스와 렌더링 엔진 사이에서 중개자 역할을 합니다. 만약 여러분들이 사용자 인터페이스 레이어에 있는 새로고침 버튼을 눌렀다면, 브라우저 엔진은 이를 이해하고 새로고침 명령을 수행합니다.
  • 렌더링 엔진은 HTML과 CSS, JavaScript를 파싱하고 그 결과물을 바탕으로 페이지를 그려내는 역할을 합니다. 각 브라우저는 다양한 엔진을 사용하는데, Chrome과 Opera, Edge는 Blink를, Firefox는 Gecko를, Internet Explorer는 Trident를, Safari는 WebKit을 사용합니다.
  • 네트워크 레이어는 HTTP나 HTTPS 같은 프로토콜을 이용해 외부의 리소스를 얻어오고, 서버에 요청을 보낼 때 사용되는 레이어입니다.
  • JavaScript 인터프리터는 JavaScript를 해석하고 실행하는 역할을 합니다. 가장 유명한 엔진으로 Chrome에 탑재된 구글의 V8이 있습니다.
  • UI 백엔드는 브라우저가 동작하고 있는 운영체제의 인터페이스를 따르는 UI들을 처리합니다. 얼럿(Alert)이나 셀렉트 박스(Select)가 운영체제 별로 다르게 동작하는 것을 쉽게 확인할 수 있죠.
  • 자료 저장소는 브라우저 자체에서 하드디스크와 같이 데이터를 로컬에 저장하기 위한 레이어로, 쿠키나 로컬스토리지, 세션 스토리지, IndexedDB, 웹 SQL, 파일시스템 등에 접근하고 데이터를 저장하는 데 사용됩니다.

D. 사용자 인터페이스

  • 주소 표시줄, 이전/다음 버튼, 북마크 메뉴 등 유저와 가장 밀접하게 맞닿아있는 부분

E. 브라우저 엔진

  • 사용자 인터페이스와 렌더링 엔진 사이의 동작을 제어하는 엔진
  • HTML 문서와 기타 지원의 웹페이지를 사용자의 장치에 시각 표현으로 변환시키며, 문서 객체 모델(DOM) 자료구조를 구현
  • 레이아웃 엔진(Layout Engine)이라고도 불리며, 렌더링 엔진과 밀접하게 연관이 있어 브라우저 엔진과 렌더링 엔진을 묶어 브라우저 엔진이라고 부르기도 함
  • 웹 브라우저마다 전용 브라우저 엔진을 사용
    • Gecko : Mozilla 재단에서 만든 브라우저 엔진으로, 파이어폭스가 탑재
    • Webkit : KHTML에서 파생된 브라우저 엔진으로, 사파리가 탑재
    • Blink : 웹킷(Webkit)에서 파생된 브라우저 엔진으로, 크롬, 오페라가 탑재
    • Trident : 마이크로소프트의 브라우저 엔진으로, 아웃룩 익스프레스, 마이크로소프트 아웃룩 등이 탑재
    • 이외에도 많은 브라우저 엔진들이 존재한다.

F. 렌더링 엔진

  • 요청한 콘텐츠를 화면에 출력하는 엔진
  • HTML과 CSS를 파싱해 최종적으로 화면에 그려주며, HTML 및 XML 문서와 이미지를 표시할 수 있다. (플러그인이나 브라우저 확장 기능을 이용하면 PDF와 같은 다른 유형도 표시할 수 있다.)

G. 통신

  • HTTP 요청과 같은 네트워크 호출에 사용되며, 보통 플랫폼의 독립적인 인터페이스이고 각 플랫폼의 하부에서 실행

H. 자바스크립트 해석기

  • JavaScript 코드를 해석하고 실행하는 해석기
  • JavaScript는 코드를 위에서 아래로 한 줄씩 읽어가는 방식으로 파싱(parsing)하는 언어(Interpreted Language)이기 때문에 해석기가 필요하다.
  • 자바스크립트 엔진(JavaScript Engine)이라고도 불린다.
  • 웹 브라우저마다 전용 자바스크립트 엔진이 탑재되어 있다.
  • Rhino : Mozilla 재단이 운영하는 오픈 소스 엔진으로, Java로 개발되었다는 특징이 있다.
  • SpiderMonkey : 최초의 JavaScript 엔진으로, 넷스케이프 내비게이터를 지원했으며 현재는 파이어폭스를 지원하고 있다.
  • V8 : 구글이 개발한 오픈 소스 엔진으로, 구글 크롬을 지원하고 있다. << 이거 꼭 기억!!
  • JavascriptCore : 애플에서 개발한 엔진으로, Webkit 프레임워크를 위해 개발되었지만 현재는 사파리와 React Native App을 지원하고 있다.
  • Chakra : 마이크로소프트가 개발한 엔진으로, Edge 브라우저를 지원하고 있다.

I. UI 백엔드

  • 렌더링 엔진이 분석한 Render Tree를 브라우저에 그리는 역할
    • Select, Input창과 같은 기본적인 위젯을 그려준다.
    • 플랫폼에서 명시하지 않은 일반적인 인터페이스로, OS 사용자 인터페이스 체계를 사용한다.
  • 사용자 인터페이스 종류
    • 명령어 라인 인터페이스(CLI; Command Line Interface)
      • : 문자로만 명령어를 입력해 처리해야 하는 인터페이스 (mac은 터미널, 윈도우는 명령 프롬프트)
    • 일괄 처리 인터페이스(Batch Interface)
      • : 사용자가 배치 처리에 앞서 배치 작업의 모든 세부 사항을 지정하고, 모든 처리가 완료되면 출력을 수신하는 비대화형 사용자 인터페이스
    • 대규모의 시스템에서 대량의 데이터를 처리하는 데에 자주 사용된다.
    • 그래픽 사용자 인터페이스(GUI; Graphic User Interface)
      • : 마우스, 키보드, 모니터 등을 이용해 아이콘을 클릭하는 등의 직관적인 입력이 가능한 인터페이스

J. 웹 스토리지

  • 영구적인 저장소인 로컬 스토리지와 임시적인 저장소인 세션 스토리지를 따로 두어 응용 환경에 맞는 선택이 가능하다.
  • 웹 스토리지의 특징
    • 웹 브라우저가 직접 데이터를 저장할 수 있게 해주어, 사용자 측에서 좀 더 많은 정보를 안전하게 저장할 수 있게 해준다.
    • 이러한 정보는 서버로 전송되지 않고, 클라이언트가 저장하고 있기 때문에 네트워크 트래픽 비용 또한 줄여준다.
    • 웹 스토리지는 출처(origin)마다 단 하나씩 존재한다.
    • 하나의 오리진에 속하는 모든 웹 페이지는 같은 데이터를 저장하기 때문에 같은 데이터에 접근할 수 있게 된다.

K. 로컬 스토리지

  • 로컬 스토리지 객체는 보관 기한이 없는 데이터를 저장
  • 로컬 스토리지에 저장된 데이터는 브라우저를 닫거나, 컴퓨터를 재부팅해도 사라지지 않는다.
  • Windows 전역 객체의 localStorage라는 컬렉션(window.localStorage)을 통해 저장 및 조회할 수 있다.
  • 도메인마다 별도의 로컬 스토리지가 생성된다.

    = 도메인만 같으면 전역으로 데이터 공유가 가능하다.

L. 세션 스토리지

  • 세션 스토리지 객체는 하나의 세션만을 위한 데이터를 저장한다.
  • 세션 스토리지에 저장된 데이터는 브라우징되고 있는 브라우저 컨택스트 내에서만 데이터가 유지되기 때문에, 브라우저 탭을 닫으면 저장된 데이터가 사라진다.
  • Windows 전역 객체의 sessionStorage라는 컬렉션(window.sessionStorage)을 통해 저장 및 조회할 수 있다.
  • 브라우저 컨택스트가 다르면 서로 다른 세션 스토리지를 가진다.
  • = 브라우저의 컨택스트가 서로 다르므로 서로 데이터 공유가 불가능하다.
  • 브라우징 : 브라우저 프로그램을 실행해서 인터넷에서 정보를 찾는 행위
  • 브라우저 컨택스트 : 브라우저가 문서를 표시하는 환경
  • 각 브라우징 컨택스트는 활성화되고 있는 문서의 출처, 표시했던 문서의 방문 기록을 가지고 있다.

3. 다양한 서버의 종류

A. 서버의 종류

B. 프론트엔드

  • 레이아웃, 메뉴, 디자인 요소, 버튼, 이미지 및 그래프와 같이 사용자가 직접 상호작용할 수 있는 사용자 인터페이스!
  • 즉, 프론트엔드는 사용자가 1차적으로 보는 인터페이스기 때문에 사용하기 쉽고 보기 편해야 사용자들의 참여를 이끌어낼 수 있음.
  • 그렇기 때문에 프론트엔드 개발자는 사용자 경험을 핵심 가치로 삼아 직관적이고 사용하기 쉬운 UI(User Interface)를 구축해야 함!
  • 퍼블리셔와 는 모두 웹사이트나 어플리케이션을 만드는 데에 관여하지만, 역할과 업무 범위에서 차이가 있음!! << 꼭 기억!
  • 퍼블리셔는 디자인을 토대로 HTML, CSS, JavaScript 등의 웹 기술을 활용하여 웹사이트나 애플리케이션의 디자인적 부분을 구현. 따라서, 디자인과 프론트엔드 기술에 대한 기초적인 지식과 능력이 필요함! 주로 웹사이트의 정적인 콘텐츠를 구현(HTML/CSS)하며, 동적인 기능 구현(JavaScript)은 범위에 포함되지 않음.
  • 반면, 프론트엔드 개발자는 웹사이트나 애플리케이션의 사용자 인터페이스(UI)를 구현하는 역할을 수행. 또한 통신을 포함하여, API를 통해 서버와 데이터를 주고 받는 것까지 구현하며, 백엔드와의 연동을 통해 동적인 기능을 구현하는 역할을 수행!
  • 따라서, 프론트엔드 개발자는 퍼블리셔보다 웹사이트나 어플리케이션을 만드는 업무의 더 깊숙한 부분까지 관여하게 되기에, 프론트엔드 개발자에게 요구되는 개발 역량이나 기술 지식들이 퍼블리셔보다 높은 수준을 상회함.
  • 프론트엔드 서버
    • 사용자의 웹 브라우저에 HTML, CSS, JavaScript 등의 정적 파일을 제공
    • 클라이언트 요청을 처리하여 필요한 데이터를 API 서버에 전달
    • 사용자에게 웹 페이지 또는 애플리케이션의 UI를 제공
    • 웹 애플리케이션의 라우팅을 관리
    • 프론트엔드 측 렌더링 기능을 통해 초기 페이지 로딩 속도를 개선
    • 즉, 사용자에게 렌더링 할 수 있는 파일들을 제공!
  • 프론트엔드의 주요 기술
    • 요새 많이 쓰이는 기술: Angular, React, Vue
    • 앵귤러(Angular): 구글이 만든 오픈소스 자바스크립트 프레임워크로, 싱글 페이지 애플리케이션(SPA) 개발에 사용됨. 폭넓은 기능을 제공하지만 복잡도가 높아 장기 프로젝트에 적합
    • 리액트(React): 동적인 UI를 생성하는 라이브러리로, 유연성이 높아 다양한 도구와 조합 가능함. 빠르고 현대적인 애플리케이션 개발에 적합
    • 뷰(Vue.js): 프리코딩 구조를 가진 인기 프레임워크로, 빠른 개발과 명확한 명령 처리 가능. 작은 애플리케이션이나 시장 진입 단계에 적합
  • 프론트엔드의 트렌드?
    • PWA(프로그레시브 웹 앱)는 웹 기술을 사용해 네이티브 앱처럼 동작하는 앱
    • PWA는 웹 기술을 사용해 네이티브 앱처럼 동작 가능
    • 네이티브 앱 개발보다 빠르게 개발 가능
    • 푸시 알림 등 네이티브 앱의 기능도 제공
    • 오프라인에서도 작동하며, 일반 웹 기술로 개발 가능
    • 기업은 PWA를 통해 모바일과 웹에서 동시에 서비스 제공 가능
  • 프론트엔드 어플리케이션 (SPA vs MPA)
    • 최신 프론트엔드 어플리케이션은 크게 SPA (Single Page Application)과 MPA (Multi page Application)으로 나뉘어 짐!
    • 예전에는 MPA를 기반으로 주로 웹을 개발하였지만, 최근에는 SPA로 많이 개발을 하고 있음! => 사용자의 편의성
    • 다만, MPA와 SPA 모두 각자 장단점이 있으니, 어플리케이션의 성격에 따라 잘 선택해서 개발을 진행해야 함.
  • Multi Page Application (MPA)
    • 사용자가 페이지를 이동할 때마다 서버에서 새로운 HTML, CSS, JavaScript 파일을 받아오는 방식.
    • 각 페이지 이동 시마다 전체 페이지가 새로고침되며, 새로운 데이터를 받아와 다시 렌더링됨.
    • MPA는 구조가 복잡한 애플리케이션이나 다양한 기능이 필요한 웹사이트에서 유리함.
    • 사용자는 각각의 페이지에서 고유한 URL을 가질 수 있으며, SEO(Search Engine Optimization)에 유리한 장점이 있음.
    • 대표적인 예: 아마존, 이베이, 위키피디아, 네이버 등
    • MPA의 장점
      • 1. 검색 엔진 최적화 (SEO)
        • MPA는 SPA와 달리 크롤러와 호환되어 SEO 최적화에 적합함.
        • 여러 페이지가 존재하므로, 각 페이지마다 키워드를 쉽게 삽입할 수 있음.
      • 2. 확장성
        • MPA는 구조적으로 새로운 페이지를 무한히 추가할 수 있어 다양한 서비스나 제품에 맞춰 확장이 쉬움.
        • 또한, 각 페이지에 대한 수정 및 업데이트도 용이함.
      • 3. 다양한 솔루션 사용 가능성
        • MPA 개발은 일반적으로 적은 기술 스택을 요구함.
        • 또한, 다양한 범위의 이미 준비된 솔루션들을 활용할 수 있음.
      • 4. 분석 정보 제공
        • MPA는 인터넷에서 얻은 데이터를 통해 시장을 분석할 수 있음.
        • 웹 활동에 대한 중요한 분석 데이터를 수집할 수 있어 기업이 이를 기반으로 전략을 수립하는 데 도움이 됨.
    • MPA의 단점
      • 1. 느린 속도
        • MPA는 SPA와 달리 사용자가 탭을 클릭할 때마다 페이지가 새로고침되어야 함.
        • 이로 인해 성능과 속도가 저하될 수 있음.
      • 2. 개발 시간이 길어짐
        • MPA는 프론트엔드와 백엔드 개발을 처음부터 모두 코딩해야 함.
        • 또한, 프론트엔드와 백엔드의 분리 문제로 인해 추가적인 어려움이 발생할 수 있음.
      • 3. 유지 보수 및 보안 문제
        • MPA는 각 페이지를 지속적으로 업데이트하고 유지하는 것이 복잡함.
        • 개발자는 각 페이지의 보안을 철저히 관리해야 하므로, 유지 보수가 번거로워질 수 있음.

  • Single Page Application (SPA)
    • 사용자가 처음으로 페이지를 클릭할 때, 모든 데이터(HTML/CSS/JavaScript)를 받아오는 방식.
    • 한 번 웹이 열리면, 각 클릭마다 페이지를 새로고침 하지 않고, 부분적인 요소(데이터)를 받아와 동작하는 방식으로 구성.
    • 싱글 페이지 웹사이트는 사용자에게 편안한 콘텐츠 환경을 제공하며, 부드럽고 작업하기 편한 웹 환경을 제공함.
    • 대표적인 예: 페이스북, 지메일, 구글 지도, 트위터 등

    • SPA의 장점
      • 1. 빠른 속도 및 성능
        • SPA는 한 번만 데이터를 로드하고 이후에는 필요한 부분만 동적으로 업데이트하므로 페이지 로딩 속도가 빠름.
        • 페이지 전환 시 전체 페이지를 새로고침하지 않아 사용자 경험이 부드럽고 원활함.
      • 2. 간단한 개발 및 유지 보수
        • SPA는 하나의 페이지에서 모든 기능을 처리하므로 개발 구조가 단순함.
        • 프론트엔드와 백엔드의 명확한 분리가 가능하여 개발과 유지 보수가 용이함.
      • 3. 모바일 친화성
        • SPA는 모바일 기기에서 최적화된 성능을 제공하며, 네이티브 앱과 유사한 사용자 경험을 제공할 수 있음.
      • 4. 캐싱 효율성
        • 한 번 로드된 콘텐츠는 브라우저에 캐시되어 이후 페이지 전환 시 빠르게 로드됨.

    • SPA의 단점
      • 1. SEO 최적화 문제
        • SPA는 콘텐츠가 동적으로 로드되므로 검색 엔진 크롤러가 페이지를 인덱싱하는 데 어려움이 있음.
        • SEO 최적화를 위해 추가적인 서버 사이드 렌더링 또는 프레임워크 설정이 필요함.
      • 2. 첫 로딩 속도 느림
        • 초기 로딩 시 모든 필요한 리소스(HTML, CSS, JavaScript)를 한꺼번에 다운로드하기 때문에 첫 로딩 속도가 느려질 수 있음.
      • 3. 브라우저 호환성 문제
        • SPA는 최신 브라우저 환경에 맞춰 개발되므로, 구형 브라우저나 비표준 브라우저에서 성능 이슈가 발생할 수 있음.
      • 4. 복잡한 상태 관리
        • SPA는 한 페이지에서 많은 기능을 처리하므로 상태 관리가 복잡해질 수 있으며, 이를 해결하기 위해 추가적인 라이브러리나 프레임워크가 필요함.

  • Static website vs Dynamic website
    • 정적 웹사이트 (Static): 고정된 콘텐츠를 제공하는 웹사이트로, HTML, CSS 파일을 미리 작성하여 서버에 저장한 후 사용자가 요청할 때 그대로 전송함.
    • 동적 웹사이트 (Dynamic): 사용자의 요청에 따라 서버가 실시간으로 데이터를 처리하고, 이를 바탕으로 페이지를 생성하여 제공하는 웹사이트. 데이터베이스와 연동되어 콘텐츠가 동적으로 변함.
    • 정적 웹사이드의 동작 방식
      • 정적 웹사이트는 서버에 미리 저장된 HTML, CSS, 이미지 파일을 사용자가 요청하면, 서버가 이를 그대로 전송함.
      • 모든 사용자는 동일한 콘텐츠를 받아보며, 페이지가 고정되어 있음.
      • 콘텐츠가 사전에 생성되어 있고, 서버의 리소스를 적게 사용
      • 예: 블로그, 회사 소개 페이지 등
      • 장점:
        • 빠른 로딩 속도: 서버에서 데이터를 처리할 필요가 없어 로딩이 빠름.
        • 저렴한 비용: 서버 리소스가 적게 소모되어 운영 비용이 저렴함.
        • SEO 친화적: 페이지가 고정되어 있어 검색 엔진 크롤러가 쉽게 인덱싱할 수 있음.
      • 단점:
        • 업데이트 어려움: 콘텐츠 변경 시 HTML 파일을 직접 수정해야 함.
        • 제한된 기능: 동적인 사용자 상호작용 기능을 제공하기 어려움.
    • 동적 웹사이트의 동작 방식
      • 동적 웹사이트는 사용자의 요청을 받으면 서버가 데이터베이스와 통신하여 필요한 데이터를 가져와 실시간으로 HTML을 생성함.
      • 페이지마다 사용자가 다른 데이터를 받아볼 수 있으며, 요청에 따라 콘텐츠가 변동됨.
      • 즉, 사용자는 요청에 따라 달라지는 웹 페이지를 보게 됨!
      • 장점:
        • 유연한 콘텐츠 관리: CMS를 통해 비개발자도 쉽게 콘텐츠를 관리할 수 있음.
        • 실시간 데이터 제공: 사용자 요청에 따라 실시간 데이터를 처리하여 제공 가능.
      • 단점:
        • 느린 로딩 속도: 서버가 매번 데이터를 처리해야 하므로, 페이지 로딩이 다소 느림.
        • 복잡한 유지 보수: 서버, 데이터베이스, 보안 관리 등이 필요하여 유지 보수가 복잡함.
      • 동적 웹사이트의 구현
        • Client-Side Rendering (CSR):
          • CSR은 데이터가 없는 HTML 문서나 Static 파일만을 처음에 서버로부터 받아오고, 이후 JavaScript를 사용해 필요한 데이터를 클라이언트에서 직접 요청(보통 backend에)하여 받아오는 방식임.
          • 자바스크립트를 사용해 브라우저에서 직접 페이지를 렌더링(JS Engine + UI)하며, 서버는 최소한의 역할을 담당함.
            모든 로직(데이터 처리, 템플릿 렌더링, 라우팅 등)은 클라이언트 측에서 이루어지며, 서버는 API 형태로 데이터를 제공하는 역할을 함.
        • Server-Side Rendering (SSR):
          • SSR은서버에서 동적으로 데이터까지 모두 삽입한 완성된 HTML을 클라이언트에게 넘겨주는 방식임.
          • 서버에서 클라이언트가 요청한 데이터를 미리 처리하여, 완성된 HTML 페이지를 제공함.
          • 서버 렌더링은 브라우저가 응답을 받기 전에 모든 작업(데이터 처리, 템플릿 작성 등)을 서버에서 완료하므로, 추가적인 데이터 요청이나 왕복이 발생하지 않음.
          • 모든 요청은 웹 서버에서 처리되므로, 클라이언트는 완성된 HTML을 바로 받아 브라우저에서 표시함.

C. 백엔드

  • 백엔드는 서버 구성, 성능 최적화, 데이터 베이스 설계 등 다양한 기술을 수행.
  • 또한, 데이터를 어떻게 저장하고 관리할 것인지, 데이터 처리 시간은 어떻게 줄일 것인지, 사용자 수가 늘어났을 때에는 어떻게 대처해야 할 것인지에 대해 끊임없이 고민해야 하기 때문에 지속적으로 서버를 개발하고 관리하는 기술도 담당
  • 더 나아가, 프론트엔드에서 받은 사용자들의 데이터들을 처리해야 하기 때문에 비즈니스 로직을 구성한 후 데이터베이스를 설계하여 데이터를 효과적으로 다루는 기술을 수행합니다.
  • 백엔드에서 사용되는 기술 스택
    • Language
      • 자바(Java): 1995년에 등장한 객체지향 프로그래밍 언어로, 자료형을 지정해야 하는 특징이 있음. 국내에서 전자정부 표준 프레임워크로 사용되며, 많은 레퍼런스를 보유함.
      • 파이썬(Python): 문법이 간결하고 쉽게 배울 수 있는 객체지향 언어로, 다양한 분야에 사용됨. 2010년대에 인기가 급상승하며 다양한 라이브러리와 프레임워크가 지원됨.
      • 자바스크립트(JavaScript): 웹에 특화된 객체지향 언어로, 프론트엔드 개발에서 많이 사용됨. 문법이 쉬워 접근성이 높으며, 최근 웹 개발에서 인기가 급상승 중임.
      • 타입스크립트(TypeScript): 마이크로소프트가 2012년에 공개한 JavaScript의 정적 타입 상위 집합. 컴파일(트랜스파일) 단계에서 타입 오류를 사전에 잡아 대규모/장기 프로젝트에 유리하며, 최신 JS로 변환되어 브라우저와 Node.js에서 실행됨. 프론트엔드(React, Vue, Angular)와 백엔드(NestJS, Express 등) 모두에서 널리 사용됨.
    • Frameworks
      • 자바(Java):
        • Spring Boot: 자바 기반의 백엔드 개발을 위한 프레임워크로, 빠르고 간편한 설정을 통해 마이크로서비스와 대규모 애플리케이션 개발에 적합함.
        • Dropwizard: 자바로 마이크로서비스를 구축할 때 사용되는 경량 프레임워크로, 빠른 개발과 배포에 특화되어 있음.
      • 파이썬(Python):
        • Django: 파이썬 기반의 풀스택 웹 프레임워크로, 백엔드 로직, 데이터베이스 처리, 사용자 인증 등을 빠르게 구축할 수 있음.
        • Flask: 최소한의 설정으로 빠르게 백엔드를 개발할 수 있는 경량 프레임워크로, 필요한 기능만 추가하여 유연하게 사용할 수 있음.
        • FastAPI: 최근 각광받는 파이썬 비동기 웹 프레임워크로, 빠른 성능과 자동 문서화(Swagger, OpenAPI 지원)가 장점임. 타입 힌트를 적극 활용하여 개발 생산성과 안정성을 높일 수 있어, 마이크로서비스 및 AI/데이터 API 구축에 적합함.
      • 자바스크립트(JavaScript):
        • Node.js + Express: 자바스크립트를 기반으로 한 서버 사이드 런타임 환경(Node.js)과 이를 쉽게 사용하게 해주는 프레임워크(Express)로, 비동기 처리와 빠른 서버 응답에 최적화되어 있음.
        • NestJS: Node.js 기반의 백엔드 프레임워크로, 모듈화와 구조화된 코드 작성을 지원하여 대규모 애플리케이션 개발에 적합함.

  • 최근 백엔드 트렌드?
    • 최근 백엔드 분야에서는 ‘서버리스’가 트렌드로 자리 잡고 있으며, 이는 클라우딩 컴퓨팅의 모델 중 하나로 ‘개발자가 서버를 직접 관리할 필요가 없는 아키텍처’
    • 만약 백엔드 개발자가 서버의 사용자가 10,000명이 될 것을 예상하고 그에 맞는 용량의 서비스를 구입했다고 가정했을 경우, 실제 사용자가 10,000명이든 0명이든 같은 금액을 내야 함. 이러한 상황은 크든 작든 손실을 일으킬 수 있음
    • 하지만 서버리스는 동적으로 서버의 자원을 할당합니다. 사용자가 없을 때에는 자원을 할당하지 않고 대기하며 요청이 들어올 때에만 자원을 할당하여 요청을 처리.
    • 결과적으로 한정된 자원을 효율적으로 사용할 수 있게 된 것입니다. 특히 해당 서버는 클라우드를 제공하는 기업에서 전적으로 관리하기 때문에 백업, 보안 등 서버에서 발생하는 이슈에 대응할 필요가 없습니다.
    • 최근, 서버 사이드 렌더링 (SSR)에 대해서도 관심이 많아지고 있음!!
    • 서버사이드 렌더링(SSR, Server-Side Rendering)은 웹 페이지의 콘텐츠를 서버에서 미리 렌더링한 후, 완성된 HTML을 클라이언트에게 전달하는 방식임. 이 방식은 클라이언트 측에서 JavaScript가 실행되기 전에 이미 화면에 완성된 페이지가 보이기 때문에, 첫 페이지 로딩 속도가 빠르고 SEO(검색 엔진 최적화)에 유리함.
    • SSR은 특히 검색 엔진이 웹 페이지를 쉽게 크롤링할 수 있게 하여, SEO 성능을 향상시킴. 또한, 초기 화면이 빨리 표시되므로 사용자 경험이 개선됨. 단, 서버의 부하가 증가할 수 있으며, 클라이언트 사이드 렌더링(CSR)보다 복잡한 설정이 필요할 수 있음.
    • 주요 프레임워크로는 Next.js(React), Nuxt.js(Vue.js) 등이 있음.

D. 서버의 구조

  • 기본적인 서버 구조
    • 웹 서버: 웹 페이지를 사용자에게 제공하는 서버로, HTTP 요청을 처리함. 대표적으로 Apache, Nginx 등이 있음.
    • 애플리케이션 서버: 웹 애플리케이션 로직을 처리하는 서버로, 클라이언트의 요청을 받아 데이터베이스와 상호작용함. Java 기반의 Tomcat, Node.js 등이 예시임.
    • 데이터베이스 서버: 데이터를 저장하고 관리하는 서버로, 애플리케이션 서버와 연동되어 데이터를 제공함. MySQL, PostgreSQL 등이 있음.
    • 파일 서버: 파일을 저장하고 공유하는 서버로, 네트워크를 통해 파일을 관리하고 배포함. FTP 서버 등이 이에 해당됨.
    • 프록시 서버: 클라이언트와 다른 서버 사이에서 중계 역할을 하며, 보안 강화, 캐싱 등을 통해 성능을 향상시킴.
  • 현대적인 서버 구조
    • 1. 프론트엔드 서버:
      • 클라이언트 측에 HTML, CSS, JavaScript와 같은 정적 파일을 제공함.
      • 주로 사용자가 브라우저에서 보는 콘텐츠와 인터페이스를 제공함.
      • 예시: Nginx, Apache 등 정적 파일을 제공하는 웹 서버.
    • 2. 백엔드 서버:
      • 클라이언트의 요청을 처리하고, 데이터베이스와 상호작용하는 서버임.
      • 비즈니스 로직과 데이터 관리를 담당하며, API 서버로도 활용됨.
      • 예시: Express(Node.js), Spring Boot(Java), Django(Python) 등의 애플리케이션 서버.
    • 3. 서버리스(Serverless):
      • 서버 인프라 관리를 클라우드 제공 업체에 맡기고, 필요한 시점에만 자원을 동적으로 할당받는 방식임.
      • 클라우드에서 제공하는 백엔드 기능을 사용하여, 서버 관리를 최소화하면서 애플리케이션을 운영함.
      • 예시: AWS Lambda, Azure Functions, Google Cloud Functions.
