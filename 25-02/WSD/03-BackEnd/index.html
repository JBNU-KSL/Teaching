3. Back-End Architecture

날짜
1.백엔드 서버의 구조

A. 백엔드 트렌드?

  • 최근 백엔드 분야에서는 ‘서버리스’가 트렌드로 자리 잡고 있으며, 이는 클라우딩 컴퓨팅의 모델 중 하나로 ‘개발자가 서버를 직접 관리할 필요가 없는 아키텍처’
  • 만약 백엔드 개발자가 서버의 사용자가 10,000명이 될 것을 예상하고 그에 맞는 용량의 서비스를 구입했다고 가정했을 경우, 실제 사용자가 10,000명이든 0명이든 같은 금액을 내야 함. 이러한 상황은 크든 작든 손실을 일으킬 수 있음
  • 하지만 서버리스는 동적으로 서버의 자원을 할당. 사용자가 없을 때에는 자원을 할당하지 않고 대기하며 요청이 들어올 때에만 자원을 할당하여 요청을 처리.
  • 최근, 서버 사이드 렌더링 (SSR)에 대해서도 관심이 많아지고 있음!!
  • 서버사이드 렌더링(SSR, Server-Side Rendering)은 웹 페이지의 콘텐츠를 서버에서 미리 렌더링한 후, 완성된 HTML을 클라이언트에게 전달하는 방식임. 이 방식은 클라이언트 측에서 JavaScript가 실행되기 전에 이미 화면에 완성된 페이지가 보이기 때문에, 첫 페이지 로딩 속도가 빠르고 SEO(검색 엔진 최적화)에 유리함.
  • SSR은 특히 검색 엔진이 웹 페이지를 쉽게 크롤링할 수 있게 하여, SEO 성능을 향상시킴. 또한, 초기 화면이 빨리 표시되므로 사용자 경험이 개선됨. 단, 서버의 부하가 증가할 수 있으며, 클라이언트 사이드 렌더링(CSR)보다 복잡한 설정이 필요할 수 있음.
  • 주요 프레임워크로는 Next.js(React), Nuxt.js(Vue.js) 등이 있음.

B. 다양한 서버의 종류 및 구조

* 서버는 역할과 기능에 따라 구분되며, 백엔드 개발자가 어떤 서버와 협업하느냐에 따라 책임 범위가 달라짐.

  • 웹 서버 (Web Server)
    • HTTP 요청을 받아 정적 콘텐츠(HTML, CSS, JS) 제공
    • 예: Apache, Nginx
  • 애플리케이션 서버 (Application Server)
    • 비즈니스 로직 실행 및 동적 콘텐츠 생성
    • 예: Node.js, Spring Boot, Django
  • 데이터베이스 서버 (DB Server)
    • 데이터 저장/검색, 트랜잭션 관리
    • 예: MySQL, PostgreSQL, MongoDB
  • 캐시 서버 (Cache Server)
    • 자주 요청되는 데이터 임시 저장 → 응답 속도 향상
    • 예: Redis, Memcached
  • 프록시 서버 (Proxy Server)
    • 클라이언트와 서버 사이에서 요청을 중계, 보안 강화 및 로드밸런싱 수행
  • 메일 서버, 파일 서버, 인증 서버 등
    • 특정 기능에 특화된 서버들 (SMTP, FTP, LDAP 등)

C. Client Server Architecture의 유형

  • N-Tier Architecture: 서버가 여러 계층으로 나뉘어 각각 다른 기능을 담당하는 구조. 확장성과 복잡한 비즈니스 로직 처리에 유리함.
    • 1-Tier Architecture: 클라이언트와 서버가 동일한 시스템에서 실행되는 구조. 대부분 단일 사용자 애플리케이션에 사용됨.
    • 2-Tier Architecture: 클라이언트와 서버가 분리되어 있는 전통적인 클라이언트-서버 구조. 클라이언트는 프론트엔드, 서버는 백엔드 역할을 수행함.
    • 3-Tier Architecture: 클라이언트, 서버, 데이터베이스가 각각 분리된 구조. 서버는 클라이언트 요청을 처리하고, 데이터베이스는 데이터를 관리하는 역할을 함.
  • Micro Service Architecture (MSA): 애플리케이션을 작은 독립 서비스 단위로 나눠 개별 배포·확장이 가능한 구조.
  • Serverless Architecture: 서버 관리 없이 클라우드가 자동으로 인프라를 운영하며, 코드 실행에만 집중하는 이벤트 기반 구조.

(1) N-Tier Architecture

a. Client-Server Architecture (2-Tier)

  • 클라이언트와 서버의 역할 분담: Client-Server Architecture는 클라이언트와 서버가 각기 다른 역할을 수행하며 상호작용하는 구조를 의미함. 클라이언트는 사용자의 요청을 생성하고, 서버는 이를 처리하여 응답을 반환함.
  • 서버의 기능: 서버는 중앙에서 데이터 처리를 담당하며, 클라이언트로부터 수신된 요청에 따라 비즈니스 로직을 실행하거나 데이터베이스와 상호작용함. 이를 통해 클라이언트는 필요한 데이터를 받을 수 있음.
  • 통신 프로토콜: 클라이언트와 서버는 주로 HTTP, HTTPS와 같은 통신 프로토콜을 사용하여 데이터를 교환함. 이 과정에서 요청(Request)과 응답(Response)이 교환되며, 클라이언트는 결과를 받아 화면에 출력함.

  • 확장성과 유지보수성: Client-Server Architecture는 모듈화된 구조를 가지고 있어 확장성과 유지보수에 용이함. 서버는 다수의 클라이언트 요청을 처리할 수 있도록 설계되며, 클라이언트와 서버의 독립적인 변경이 가능함.
  • 백엔드의 역할: 백엔드는 서버 측에서 작동하며 데이터 처리, 저장, 외부 서비스 연동 등을 담당함. 클라이언트가 필요한 데이터를 요청하면 백엔드에서 처리 후 클라이언트에게 필요한 정보를 반환함.
  • 다중 사용자 지원: 중앙 집중식 서버는 여러 클라이언트의 동시 접속을 처리할 수 있는 능력을 제공하며, 이를 통해 다수의 사용자가 동일한 자원에 접근할 수 있음.

b. Client-Server Architecture (3-Tier)

  • 3-Tier Architecture는 클라이언트-서버 아키텍처의 확장된 형태로, 애플리케이션을 세 가지 독립적인 계층으로 나누어 관리하는 구조를 의미
  • 각각의 계층은 특정한 역할을 수행하며, 이러한 분리는 시스템의 유지보수성, 확장성, 보안성을 높이는 데 기여.
  • 확장성: 서버 또는 데이터베이스 계층을 독립적으로 확장할 수 있어 트래픽이 증가할 경우 성능을 높이기 용이
  • 보안 강화: 데이터베이스와 비즈니스 로직 계층이 분리되어 있기 때문에, 프레젠테이션 계층을 통해 직접적으로 데이터베이스에 접근하는 일이 없어 보안이 강화됨.
  • 재사용성: 비즈니스 로직 계층과 데이터 계층은 여러 클라이언트 애플리케이션에서 재사용될 수 있음.

1) 프레젠테이션 계층

  • 역할: 사용자 인터페이스(UI)와 관련된 모든 처리를 담당하는 계층으로, 주로 클라이언트 측에서 동작함. 사용자가 상호작용하는 화면이나 애플리케이션의 시각적인 부분을 처리함.
  • 주요 기능: 사용자 입력을 받아 이를 처리하고, 서버로 요청을 전달하며, 서버로부터 받은 응답을 화면에 출력하는 역할을 수행함.
  • 예시: 웹 브라우저에서 동작하는 HTML, CSS, JavaScript 기반의 프론트엔드 애플리케이션 또는 모바일 애플리케이션의 UI 부분.

2) 비지니스 로직 계층

  • 역할: 애플리케이션의 핵심 로직을 담당하는 계층으로, 클라이언트 요청을 처리하고 데이터 처리 규칙과 비즈니스 로직을 실행함. 주로 서버 측에서 실행됨.
  • 주요 기능: 클라이언트 요청을 받아 데이터베이스와 상호작용하며, 요청에 따라 비즈니스 규칙을 적용한 후, 결과를 프레젠테이션 계층으로 반환함.
  • 예시: 웹 애플리케이션의 백엔드 서버, API 서버, 애플리케이션 서버 등이 여기에 속함. 보통 Python, Java, C#, Node.js 등의 프로그래밍 언어로 작성됨.

3) 퍼시스턴스 (영속성) 레이어

  • 역할: 애플리케이션의 비즈니스 로직과 데이터 저장소(DB, 캐시, 파일 스토리지 등) 사이를 추상화하여 연결하는 계층으로, 데이터 접근 코드를 한곳에 모아 재사용성과 유지보수성을 높임. 트랜잭션 경계를 관리하고, 객체–관계 매핑(ORM)과 예외 변환을 담당.
  • 주요 기능: CRUD 작업 생성/조회/수정/삭제 추상화(DAO/Repository), 엔티티 ↔ 테이블(또는 문서/키-값) 매핑, 쿼리 빌딩/실행, 페이징/정렬, 트랜잭션 관리(커밋/롤백), 락(낙관/비관) 처리
  • 예시: Repository/DAO/Mapper 패턴, Unit of Work, DataSource/Connection Pool(HikariCP 등)

4) 데이터베이스 계층

  • 역할: 데이터를 저장하고 관리하는 계층으로, 영구적으로 데이터를 저장하는 데이터베이스가 위치하는 부분임.
  • 주요 기능: 비즈니스 로직 계층의 요청에 따라 데이터를 저장하거나 조회하며, 안전하고 효율적으로 데이터를 관리함.
  • 예시: MySQL, PostgreSQL, MongoDB와 같은 관계형 또는 비관계형 데이터베이스가 이 계층을 구성함.

(2) Micro Service Architecture (MSA)

  • 개념: 마이크로서비스 아키텍처는 애플리케이션을 작은 서비스 단위로 분리하여 각각 독립적으로 배포, 확장, 유지보수가 가능하도록 하는 방식임. 전통적인 모놀리식 아키텍처와 달리, 서비스 간 의존성이 낮아짐.
  • 특징: 각 마이크로서비스는 자체 데이터베이스를 가질 수 있으며, 특정 비즈니스 기능에 집중하여 독립적으로 개발, 배포, 확장 가능함. 이를 통해 시스템의 유연성과 확장성을 극대화할 수 있음.
  • 장점: 서비스 단위별 확장이 가능하며, 장애 발생 시 전체 시스템이 아닌 부분적으로만 영향이 미침. 독립적인 팀이 개별 서비스를 개발할 수 있어 협업이 효율적임.
  • 단점: 서비스 간 통신이 복잡해질 수 있으며, 서비스마다 배포와 관리를 위한 인프라 구성 및 유지가 필요함.

(3) Serverless Architecture

  • 개념: 서버리스 아키텍처는 개발자가 직접 서버를 관리하지 않고 클라우드 서비스 제공자가 인프라를 운영하고 확장하는 방식임. 개발자는 비즈니스 로직에만 집중할 수 있으며, 서버 운영 부담을 덜 수 있음.
  • 주요 서비스: AWS Lambda, Google Cloud Functions, Azure Functions 등이 서버리스 아키텍처를 지원하는 주요 클라우드 서비스임.
  • 장점: 자동 확장, 비용 효율성, 관리 복잡성 감소 등이 주요 장점임. 애플리케이션의 사용량에 따라 자동으로 리소스를 조정함으로써 유휴 상태에서 비용을 절감할 수 있음.
  • 단점: 서버 실행 시간이 제한적이며, 백엔드에서 복잡한 작업을 처리하는 데는 적합하지 않을 수 있음. 또한, 클라우드 제공자에 대한 종속성이 생길 수 있음.
