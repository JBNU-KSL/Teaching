05. REST API 예시

날짜
A. 요청 (Request)

1) HTTP 요청의 전체 형태

<요청라인>                # 예: POST /users?active=true HTTP/1.1
<헤더들...>              # 예: Host: api.example.com
                        #     Authorization: Bearer <token>
                        #     Content-Type: application/json; charset=utf-8
                        #     Content-Length: 27
<빈 줄(헤더와 바디 구분)>
<바디>                   # 예: {"name":"Lee","role":"admin"}
  • 구체적인 예시
POST /todos HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer <JWT>
Content-Length: 55


{
  "title": "프로그래밍 과제 하기",
  "dueDate": "2025-10-05"
}

2) 요청 헤더 (Headers)

  • 요청 식별/ 전송 관련
    HEAD
POST /upload HTTP/1.1
Host: api.example.com
User-Agent: MyUploader/2.0 (macOS 14; x64)
Date: Tue, 30 Sep 2025 12:34:56 GMT
Connection: keep-alive
Transfer-Encoding: chunked
Expect: 100-continue
TE: trailers
Via: 1.1 cache-01.example.net, 1.1 lb-1
Forwarded: for=203.0.113.7;proto=https;by=lb-1
X-Forwarded-For: 203.0.113.7
X-Forwarded-Proto: https
Content-Type: application/octet-stream
    • Host: HTTP/1.1에서 필수. 가상호스팅 구분(예: Host: api.example.com).
    • User-Agent: 클라이언트 정보(버전, 플랫폼 등). 로깅/분석에 유용.
    • Date: 메시지 생성 시각(요청/응답 둘 다 가능, 응답에서 더 흔함).
    • Connection: HTTP/1.1 전용 연결 옵션(예: keep-alive). HTTP/2에서는 사용 안 함.
    • Transfer-Encoding: 메시지 프레이밍(예: chunked, HTTP/1.1). Content-Length와 병용 금지.
    • Expect: 100-continue: 큰 바디 업로드 전, 서버가 받아도 되는지 확인 요청.
    • TE: 전송 코딩 협상(요청에서 드뭄).
    • Via, Forwarded / X-Forwarded-For, X-Forwarded-Proto: 프록시·로드밸런서 경유 정보.

  • 콘텐츠(바디) 기술
    • Content-Type: 바디의 MIME 타입(+ 인코딩).
      • application/json; charset=utf-8,
      • multipart/form-data; boundary=----abc,
      • application/x-www-form-urlencoded.
    • Content-Length: 바디의 바이트 길이(문자 수 아님).
    • Content-Encoding: 바디 압축 등(예: gzip). 요청 바디에도 사용 가능하지만 서버가 지원해야 함.
    • Content-Language: 바디의 언어(예: ko).
    • Content-MD5(과거 관용): 무결성 체크로 쓰였으나 표준상 권장되지 않음. 대신 서명/해시 스킴을 사용.
    • JSON + 언어 지정 (가장 흔한 케이스)
    HEAD
POST /comments HTTP/1.1
Host: api.example.com
Content-Type: application/json; charset=utf-8
Content-Language: ko
Accept: application/json
Content-Length: 87   # 예시 값


BODY
{
  "postId": 1001,
  "author": "Lee",
  "text": "정말 유용했어요!",
  "tags": ["rest","http"]
}
    • JSON + 요청 바디 압축 (Content-Encoding: gzip)
    HEAD
POST /bulk/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Content-Encoding: gzip
Content-Language: ko
Accept: application/json
Content-MD5: qUqP5cyxm6YcTAhz05Hph5==   # 예시(base64) — 표준에선 비권장
Content-Length: 314   # 예시 값 (압축된 바디의 바이트 길이)


BODY
<gzip으로 압축된 JSON 바이트 스트림>
    • multipart/form-data (파일 업로드 + 텍스트)
    HEAD
POST /avatars HTTP/1.1
Host: api.example.com
Content-Type: multipart/form-data; boundary=----Boundary123
Accept: application/json
Content-Length: 642   # 예시 값


BODY
------Boundary123
Content-Disposition: form-data; name="userId"

42
------Boundary123
Content-Disposition: form-data; name="avatar"; filename="me.png"
Content-Type: image/png

<PNG 파일 바이너리>
------Boundary123--
    • application/x-www-form-urlencoded (로그인 폼 전송)
    HEAD
POST /login HTTP/1.1
Host: auth.example.com
Content-Type: application/x-www-form-urlencoded
Accept: application/json
Content-Length: 35   # 예시 값


BODY
username=lee&password=s3cr3t

  • 콘텐츠 협상(클라이언트 선호)
    • Accept: 응답 미디어 타입 선호(예: application/json).
    • Accept-Language: 응답 언어 선호(예: ko, en;q=0.8).
    • Accept-Charset: 문자셋 선호(거의 사용 안 함, JSON은 기본 UTF-8).
    • Accept-Encoding: 응답 압축 선호(예: gzip, br).
    • API 조회 — JSON 우선, 한국어 선호, gzip/br 압축 허용
    HEAD
GET /v1/todos?status=active HTTP/1.1
Host: api.example.com
Accept: application/json
Accept-Language: ko, en;q=0.8
Accept-Charset: utf-8
Accept-Encoding: gzip, br
    • 리포트 다운로드 — CSV 우선, JSON도 수용, 문자셋 선호 표시
    HEAD
GET /v1/reports/2025-09 HTTP/1.1
Host: api.example.com
Accept: text/csv; q=1.0, application/json; q=0.8
Accept-Language: ko-KR, en-US;q=0.6
Accept-Charset: utf-8, iso-8859-1;q=0.2, *;q=0.1
Accept-Encoding: gzip, br
    • 이미지 썸네일 — 최신 포맷 우선(avif/webp), 어떤 언어든 OK
    HEAD
GET /images/thumbnail/12345 HTTP/1.1
Host: cdn.example.com
Accept: image/avif, image/webp, image/*;q=0.8, */*;q=0.5
Accept-Language: *        # 언어 비의존 리소스라 제한 없음
Accept-Charset: utf-8
Accept-Encoding: gzip, br
    • 문서 미리보기 — PDF 우선, 압축은 가능하면 하고(없으면 identity)
    HEAD
GET /docs/preview/987.pdf HTTP/1.1
Host: files.example.com
Accept: application/pdf, application/octet-stream;q=0.5
Accept-Language: ko, en;q=0.7
Accept-Charset: utf-8
Accept-Encoding: gzip, identity;q=0.1

  • 인증/권한/세션
    • Authorization: 인증 정보.

      예: Bearer <JWT>, Basic <base64>, Digest ..., HMAC 서명 등.

    • Proxy-Authorization: 프록시 인증.
    • Cookie: 세션·상태 전달(예: Cookie: sid=abc; theme=dark). 민감정보는 가급적 쿠키/URL에 두지 말고 토큰 헤더 사용.
    • Bearer 토큰(JWT) 인증
    HEAD
GET /v1/me HTTP/1.1
Host: api.example.com
Authorization: Bearer <JWT>
Accept: application/json
    • Basic 인증
    HEAD
GET /admin HTTP/1.1
Host: api.example.com
Authorization: Basic <base64(username:password)>
Accept: text/html
    • Proxy-Authorization
    HEAD
GET http://intranet.example.com/docs HTTP/1.1
Host: intranet.example.com
Proxy-Authorization: Basic <base64(proxyUser:proxyPass)>
Accept: text/html
    • Cookie 세션 (상태 유지)
    HEAD
GET /cart HTTP/1.1
Host: shop.example.com
Cookie: sid=abc123; theme=dark
Accept: application/json

3) 요청 바디 (Body)

  • 요청 바디는 리소스의 실제 데이터를 담는 container!

  • 바디를 “언제” 쓰나?
    • 일반적으로 POST/PUT/PATCH에서 사용.
    • GET/DELETE에 바디를 보낼 수는 있지만(프로토콜상 금지 X) 서버가 무시하는 경우가 많아 권장되지 않음.
    • HEAD/TRACE는 바디를 갖지 않는 것이 관례.
    • POST-리소스 생성
      HEAD
POST /todos HTTP/1.1
Host: api.example.com
Content-Type: application/json
Accept: application/json


BODY
{
  "title": "프로그래밍 과제 하기",
  "dueDate": "2025-10-05"
}
    • PUT — 전체 교체(스냅샷 업서트)
      HEAD
PUT /profiles/42 HTTP/1.1
Host: api.example.com
Content-Type: application/json
Accept: application/json


BODY
{
  "id": 42,
  "displayName": "K. Lee",
  "lang": "ko"
}
    • PATCH — 부분 변경
      HEAD
PATCH /profiles/42 HTTP/1.1
Host: api.example.com
Content-Type: application/merge-patch+json
Accept: application/json


BODY
{
  "lang": "en"
}
    • GET + Body (권장 X)
      HEAD
GET /search HTTP/1.1
Host: api.example.com
Content-Type: application/json
Accept: application/json


BODY
{
  "q": "rest api",           # ❌ 대체로 무시됨
  "page": 2
}


==>


HEAD
GET /search?q=rest+api&page=2 HTTP/1.1
Host: api.example.com
Accept: application/json


BODY
# 없음
    • DELETE + Body (권장 X)
      HEAD
DELETE /todos/101 HTTP/1.1
Host: api.example.com
Content-Type: application/json
Accept: application/json


BODY
{
  "reason": "duplicate"      # ❌ 서버가 무시할 수 있음
}


==>

HEAD
DELETE /todos/101?reason=duplicate HTTP/1.1
Host: api.example.com
Accept: application/json


BODY
# 없음

  • 대표적인 바디 포맷
    • application/json (가장 흔함)
      • 예: {"name":"Lee","role":"admin"}
      • JSON은 기본 UTF-8(헤더에 charset=utf-8을 함께 보내는 것도 실무에선 흔함).
      HEAD
POST /users HTTP/1.1
Host: api.example.com
Content-Type: application/json; charset=utf-8
Accept: application/json


BODY
{
  "name": "Lee",
  "role": "admin"
}
    • application/x-www-form-urlencoded
      • HTML 폼 전송 기본. key=value&key2=value2 형태(퍼센트 인코딩).
      HEAD
POST /login HTTP/1.1
Host: auth.example.com
Content-Type: application/x-www-form-urlencoded
Accept: application/json


BODY
username=lee&password=s3cr3t
    • multipart/form-data
      • 파일 업로드나 혼합 데이터(텍스트 + 파일). 각 파트마다 자체 헤더 보유.
      • 파트 헤더 예: Content-Disposition: form-data; name="avatar"; filename="me.png", Content-Type: image/png
      HEAD
POST /avatars HTTP/1.1
Host: api.example.com
Content-Type: multipart/form-data; boundary=----Boundary123
Accept: application/json


BODY
------Boundary123
Content-Disposition: form-data; name="userId"

42
------Boundary123
Content-Disposition: form-data; name="avatar"; filename="me.png"
Content-Type: image/png
    • application/octet-stream
      • 바이너리 스트림(예: 대용량 업로드). 바디 자체가 파일.
        HEAD
PUT /storage/raw/backup.tar HTTP/1.1
Host: files.example.com
Content-Type: application/octet-stream
Authorization: Bearer <JWT>


BODY
<tar 바이너리 바이트 스트림>
    • 기타: application/xml, text/plain, application/merge-patch+json, application/json-patch+json, application/x-ndjson(JSON Lines) 등.

B. 응답 (Response)

1) HTTP 응답의 전체 형태

<상태라인>                # 예: HTTP/1.1 200 OK
<헤더들...>              # 예: Content-Type: application/json; charset=utf-8
                        #     Content-Length: 27
<빈 줄>
<바디>                   # 예: {"id":101,"title":"..."}
  • 상태라인: 프로토콜 버전 + 상태코드 + 이유구문
  • 헤더: 응답 메타데이터(콘텐츠 타입, 캐시, 인증요구, 위치 등)
  • 바디: 실제 데이터(JSON, 파일, HTML…). 코드/메서드에 따라 없을 수 있음(예: 204 No Content, 304 Not Modified)

2) 상태코드 일람

  • 2xx 성공: 200 OK, 201 Created, 202 Accepted, 204 No Content, 206 Partial Content
  • 3xx 리다이렉션: 301/302/303/307/308 + Location
  • 4xx 클라이언트 오류: 400, 401 + WWW-Authenticate, 403, 404, 409, 412, 415, 429 …
  • 5xx 서버 오류: 500, 502, 503(+ Retry-After), 504 …

3) 자주 쓰는 응답 헤더

  • 콘텐츠/표현
    • Content-Type, Content-Length, Content-Encoding, Content-Language
  • 캐시/조건부
    • ETag, Last-Modified, Cache-Control, Expires, Vary
  • 네비게이션
    • Location(생성/리다이렉트), Link(페이지네이션 등), Content-Range/Accept-Ranges
  • 인증/세션
    • WWW-Authenticate(401), Set-Cookie, Clear-Site-Data
  • 레이트 리밋(관용)
    • RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset 또는 X-RateLimit-*
  • CORS
    • Access-Control-Allow-*, Access-Control-Expose-Headers
  • 신뢰성/지시
    • Retry-After(429/503/202), Preference-Applied(Prefer 응답)

4) 성공 응답 예시

4-1) 200 OK (JSON)

HEAD
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
ETag: "e7a9-abc123"
Cache-Control: max-age=60
Vary: Accept, Accept-Encoding

BODY
{
  "id": 101,
  "title": "프로그래밍 과제 하기",
  "dueDate": "2025-10-05",
  "done": false
}

4-2) 201 Created (+ Location)

HEAD
HTTP/1.1 201 Created
Content-Type: application/json; charset=utf-8
Location: <https://api.example.com/todos/101>

BODY
{
  "id": 101,
  "title": "프로그래밍 과제 하기",
  "dueDate": "2025-10-05"
}

4-3) 202 Accepted (비동기 작업 수락)

HEAD
HTTP/1.1 202 Accepted
Content-Type: application/json; charset=utf-8
Location: <https://api.example.com/jobs/7f9a>
Retry-After: 10
Preference-Applied: respond-async

BODY
{ "jobId": "7f9a", "status": "queued" }

4-4) 204 No Content (바디 없음)

HEAD
HTTP/1.1 204 No Content

BODY

7) 조건부/캐시 관련 응답

7-1) 304 Not Modified (바디 없음)

HEAD
HTTP/1.1 304 Not Modified
ETag: "e7a9-abc123"

BODY

8) 요청 포맷/콘텐츠 문제

8-1) 400 Bad Request (스키마 검증 실패)

HEAD
HTTP/1.1 400 Bad Request
Content-Type: application/problem+json

BODY
{
  "type": "<https://example.com/problems/validation-error>",
  "title": "Invalid Request",
  "detail": "Field 'dueDate' must be ISO-8601 date.",
  "errors": { "dueDate": ["invalid-format"] }
}

8-2) 406 Not Acceptable (협상 실패)

HEAD
HTTP/1.1 406 Not Acceptable
Content-Type: application/json; charset=utf-8

BODY
{ "error": "not_acceptable", "supported": ["application/json", "text/csv"] }

8-3) 415 Unsupported Media Type (요청 Content-Type 미지원)

HEAD

14) Best Practice!

  • 콘텐츠 계약: 항상 올바른 Content-Type(응답 포맷), 필요 시 charset=utf-8.
  • 캐시/조건부: 가능한 ETag 또는 Last-Modified 발행 → 클라이언트가 If-*로 재검증.
  • 리소스 생성: 201 Created와 정확한 Location 헤더.
  • 에러 포맷 일관성: application/problem+json 또는 팀 표준 에러 스키마 사용.
  • 레이트 리밋: 429와 함께 남은 한도/재시도 타이밍을 헤더로 제공.
  • CORS: 필요한 최소 출처만 허용, Expose-Headers로 클라이언트가 읽어야 할 헤더를 공개.
  • 보안: 세션 쿠키는 Secure/HttpOnly/SameSite; 민감 정보는 바디에만.

15) 응답 포맷 표준화 (Response Format)

15.1 규약

  • Envelope(권장): 모든 성공 응답을 아래 공통 포맷으로 래핑
    • data: 실제 페이로드(객체/배열/원시값 모두 가능)
    • meta: 부가 정보(페이징, 추적 ID, 처리시간 등)
    • links: HATEOAS/네비게이션 링크(선택)
  • 시간/숫자
    • 시간은 UTC ISO-8601 (YYYY-MM-DDTHH:mm:ssZ)
    • 금액은 정수 최소 단위(예: KRW 원 단위, USD 센트 단위) + 통화코드 명시
  • 네이밍: camelCase(권장) 또는 팀 표준. 문서 전반에서 일관 유지
  • 널/비어있음: 존재하지 않음(null)과 빈값(""/[]) 구분
  • 정렬: 목록 기본 정렬을 명시(예: createdAt)

15.2 단건/목록 성공 예시

(A) 단건

HEAD
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
ETag: "e7a9-abc123"

BODY
{
  "data": {
    "id": 101,
    "title": "프로그래밍 과제 하기",
    "dueDate": "2025-10-05",
    "done": false
  },
  "meta": {
    "requestId": "8d1d8a30-2a2b-4a1a-9e4e-1dbb6a3d7c01",
    "servedAt": "2025-09-30T12:34:56Z"
  },
  "links": {
    "self": "<https://api.example.com/todos/101>"
  }
}

(B) 목록

HEAD
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8

BODY
{
  "data": [
    { "id": 101, "title": "A", "done": false },
    { "id": 102, "title": "B", "done": true  }
  ],
  "meta": {
    "total": 240,
    "page": 2,
    "perPage": 50,
    "requestId": "8d1d8a30-2a2b-4a1a-9e4e-1dbb6a3d7c01"
  },
  "links": {
    "self": "<https://api.example.com/todos?page=2&perPage=50>",
    "next": "<https://api.example.com/todos?page=3&perPage=50>",
    "prev": "<https://api.example.com/todos?page=1&perPage=50>"
  }
}

15.3 구현 체크리스트

  • 모든 성공 응답에 data 포함 (빈 결과는 data: [] 또는 data: null)
  • 트레이싱을 위해 meta.requestId(또는 헤더 X-Request-ID) 제공
  • 정렬/필터/페이징 규칙을 meta/links에 반영
  • 큰 컬렉션은 기본 limit/perPage 적용 및 상한 설정

16) 에러 포맷 표준화 (Problem Details)

16.1 규약

  • 표준 미디어 타입 application/problem+json 사용 (RFC 9457)
  • 공통 필드:
    • type(URL 또는 "about:blank"), title, status, detail, instance
  • 확장 필드:
    • errors(필드별 오류 맵), requestId, hint, docs

16.2 유효성 오류(400) 예시

HEAD
HTTP/1.1 400 Bad Request
Content-Type: application/problem+json

BODY
{
  "type": "<https://example.com/problems/validation-error>",
  "title": "Invalid Request",
  "status": 400,
  "detail": "One or more fields are invalid.",
  "errors": {
    "dueDate": ["must be ISO-8601 date"]
  },
  "requestId": "8d1d8a30-2a2b-4a1a-9e4e-1dbb6a3d7c01",
  "instance": "/todos"
}

16.3 인증 오류(401) 예시

HEAD
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="api", error="invalid_token", error_description="expired"
Content-Type: application/problem+json

BODY
{
  "type": "<https://example.com/problems/auth>",
  "title": "Unauthorized",
  "status": 401,
  "detail": "Access token expired.",
  "requestId": "8d1d8a30-2a2b-4a1a-9e4e-1dbb6a3d7c01"
}

16.4 동시성 충돌(412) 예시

HEAD
HTTP/1.1 412 Precondition Failed
ETag: "b13c-77df29"
Content-Type: application/problem+json

BODY
{
  "type": "<https://example.com/problems/precondition-failed>",
  "title": "Precondition Failed",
  "status": 412,
  "detail": "ETag mismatch.",
  "hint": "Refetch the resource and retry with the latest ETag."
}

16.5 구현 체크리스트

  • 모든 비-2xx 응답은 application/problem+json
  • 에러 코드 매핑표(HTTP ↔ 도메인 코드) 유지
  • 필드 에러는 errors.{field}:[…] 배열로 제공
  • 로그/추적 연계를 위해 requestId 포함

17) 페이지네이션 (Pagination)

17.1 규약(둘 중 하나 선택, 혼용 금지)

(A) 쿼리 파라미터 페이지형

  • 요청: ?page=2&perPage=50&sort=-createdAt
  • 응답: data 배열 + meta.total/page/perPage + links.self/next/prev
  • 장점: 단순/직관적.
  • 단점: 변동(total) 계산 비용.

요청/응답 예시

HEAD
GET /todos?page=2&perPage=50&sort=-createdAt HTTP/1.1
Host: api.example.com
Accept: application/json

BODY
# 없음

HEAD
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
Link: <https://api.example.com/todos?page=3&perPage=50>; rel="next",
      <https://api.example.com/todos?page=1&perPage=50>; rel="prev"

BODY
{
  "data": [ /* 50 items */ ],
  "meta": {
    "total": 240,
    "page": 2,
    "perPage": 50
  },
  "links": {
    "self": "<https://api.example.com/todos?page=2&perPage=50>",
    "next": "<https://api.example.com/todos?page=3&perPage=50>",
    "prev": "<https://api.example.com/todos?page=1&perPage=50>"
  }
}

(B) 커서형(cursor/token)

  • 요청: ?limit=50&cursor=<opaqueToken> (첫 페이지는 cursor 없음)
  • 응답: meta.nextCursor(있으면 다음 페이지 있음), meta.hasMore(선택)
  • 장점: 대규모/실시간 변동 데이터에 강함, 안정적 스캔.
  • 단점: 구현 복잡도↑, 임의 페이지 접근 어려움.

요청/응답 예시

HEAD
GET /todos?limit=50&cursor=eyJpZCI6MTAxLCJjcmVhdGVkQXQiOiIyMDI1LTA5LTMwVDEyOjM0OjU2WiJ9 HTTP/1.1
Host: api.example.com
Accept: application/json

BODY
# 없음
HEAD
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8

BODY
{
  "data": [ /* 50 items */ ],
  "meta": {
    "limit": 50,
    "nextCursor": "eyJpZCI6MTAyLCJjcmVhdGVkQXQiOiIyMDI1LTA5LTMwVDEyOjM1OjEwWiJ9",
    "hasMore": true
  },
  "links": {
    "self": "<https://api.example.com/todos?limit=50&cursor=>...",
    "next": "<https://api.example.com/todos?limit=50&cursor=eyJpZCI6MTAy>..."
  }
}

17.2 Range-Unit 기반(대안)

  • 요청 헤더:
    HEAD
GET /todos HTTP/1.1
Host: api.example.com
Range-Unit: items
Range: items=0-49
  • 응답 헤더: 206 Partial Content + Content-Range: items 0-49/240

17.3 구현 체크리스트

  • 기본 perPage/limit와 최대 상한 설정(예: 기본 50, 최대 200)
  • 정렬 필드/방향 규약(예: sort=-createdAt)
  • 페이지형은 total 계산 비용이 큰 경우 지연 계산 또는 추정치 문서화
  • 커서형은 nextCursor를 불투명 토큰으로(내부 스키마 노출 금지)
  • Link 헤더와 links 객체를 동시에 제공(클라이언트 선택권)
  • 빈 결과: data: [], meta.total: 0 또는 hasMore: false
