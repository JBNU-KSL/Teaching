Part 1 · Foundation

자료구조와 ADT: 왜 분리해서 보나?

Python 3.10의 built-in types 분류(수치형·시퀀스형·매핑형·집합형)를 기준으로 보면, 자료구조 선택은 단순 문법이 아니라 연산 의미를 선택하는 과정입니다. 또한 데이터 모델 관점에서 변수는 객체를 담는 상자가 아니라 이름 바인딩이므로, 같은 코드라도 참조 공유 여부에 따라 동작이 크게 달라집니다.

$$ \text{ADT} = \langle \mathcal{S}, \mathcal{O}, \mathcal{I} \rangle $$ \(\mathcal{S}\): 상태 공간, \(\mathcal{O}\): 연산 집합, \(\mathcal{I}\): 불변식

문제를 ADT로 추상화한 뒤, 구현 구조를 선택하고 정확성/복잡도/재현성을 검증합니다.

ADT (추상 자료형)

“어떻게 하는가”를 정의합니다. 예: Stack을 list로 구현할지 deque로 구현할지 선택한다.

Part 1.1 · Theory

복잡도 이론: 점근 표기와 비용 모델

실행 시간은 기계/인터프리터/캐시 상태에 따라 달라지므로, 입력 크기 n에 대한 성장률로 알고리즘을 비교합니다. 자료구조 선택의 1차 기준은 상수항이 아니라 점근적 증가율입니다.

연구 코드에서 자주 혼동되는 지점은 “빠르다”의 의미입니다. 단일 입력의 절대 시간보다, 토큰 길이·배치 크기·스텝 수가 증가할 때 비용 곡선이 어떻게 변하는지가 더 중요합니다. 따라서 자료구조 비교는 동일한 연산 시퀀스와 동일한 입력 분포를 기준으로 점근 표기와 실측치를 함께 해석해야 합니다.

$$ T(n)\in O(g(n)) \iff \exists c>0,\ \exists n_0\ \text{s.t.}\ 0\le T(n)\le c\,g(n),\ \forall n\ge n_0 $$
표기의미해석
O(g(n))점근적 상한최악/상한 관점에서 비용이 g(n)을 넘지 않음
Ω(g(n))점근적 하한최소한 g(n)만큼은 필요함
Θ(g(n))점근적 동치상한·하한이 같은 차수

예: list.pop(0)는 요소 시프트로 Θ(n), dict.get은 평균 Θ(1)입니다. 같은 Python 코드라도 자료구조가 바뀌면 차수가 바뀝니다.

Part 1.2 · Amortized Analysis

상환 분석: 왜 append는 평균적으로 O(1)인가?

동적 배열(list)의 append는 개별 연산 기준으로는 최악 O(n)입니다. 그러나 연산열(sequence) 전체를 보면 총비용이 선형이므로 연산당 평균 비용은 상수입니다. 상환 분석은 바로 이 “연산열 수준 보장”을 제공하는 도구입니다.

분석 모델을 명확히 두겠습니다. 현재 길이를 m, 용량을 c라 하면, m < c인 append 비용은 1(쓰기), m = c이면 새 배열 할당 + 기존 원소 m개 복사 + 쓰기 1로 m+1입니다. 성장 정책은 c' = 2c로 가정합니다.

$$ \text{if }c_t = 2^t,\ \sum_{j=0}^{t-1}c_j = 2^t-1 \Rightarrow \text{total copy work before capacity }2^t\text{ is }<2^t $$
분석 축핵심 결론
Worst-case per op재할당 시점의 append는 O(n)
Total cost over n appends쓰기는 n, 누적 복사는 2n 미만, 따라서 총 O(n)
Amortized cost총 O(n)/n = O(1)
시스템 관점지연 스파이크는 존재하지만 장기 처리량(throughput)은 선형적으로 안정

가정의 역할

상환 O(1)은 “증가율이 1보다 큰 상수배 성장”과 “복사 비용이 원소 수에 비례”라는 모델 위에서 성립합니다. 성장 정책이 선형 증가이면 상환 O(1)이 깨집니다.

설계 시사점

append 중심 배치 누적에는 list가 유리합니다. 반면 앞단 삭제/삽입이 주연산이면 상환 분석 대상이 달라지므로 deque로 구조를 분리해야 합니다.

Part 1.2.1 · Proof

상환 O(1)의 엄밀한 증명: Aggregate / Accounting / Potential

정리(Theorem). 빈 동적 배열에 대해, 용량이 가득 찰 때마다 2배로 확장한다고 하자. n번의 append 총비용은 O(n)이며, 따라서 상환 비용은 O(1)이다.

1) Aggregate Method

$$ T(n)=n+\sum_{j=0}^{k-1}2^j,\ \text{where }2^k \le n < 2^{k+1} $$ $$ T(n) \le n + (2^k-1) < n+n = 2n $$

여기서 첫 항 n은 각 append의 쓰기 비용, 두 번째 항은 재할당 시 누적 복사 비용입니다. 결과적으로 T(n)/n < 2이므로 상환 비용은 상수입니다.

2) Accounting Method

각 append에 크레딧 3을 부과합니다. 실제 쓰기에 1을 즉시 지불하고 2를 적립합니다. 재할당 시 복사되는 각 원소는 과거에 적립한 크레딧 1개로 복사비를 지불할 수 있습니다.

모든 원소는 다음 재할당까지 충분한 크레딧을 보유하므로 적자가 발생하지 않습니다. 따라서 상환 비용은 상수(여기서는 3)로 상계됩니다.

3) Potential Method

잠재함수를 \Phi = 2m - c로 둡니다(m: 길이, c: 용량). 항상 \Phi \ge 0 (충분히 큰 단계에서)이며, 상환비용을 \hat{c}_i = c_i + \Phi_i - \Phi_{i-1}로 둡니다.

$$ \text{non-resize: }c_i=1,\ \Delta\Phi=2 \Rightarrow \hat{c}_i=3 $$ $$ \text{resize at }m=c:\ c_i=m+1,\ \Delta\Phi=2-m \Rightarrow \hat{c}_i=3 $$ $$ \therefore\ \forall i,\ \hat{c}_i \le 3 \Rightarrow \sum_{i=1}^n c_i \le \sum_{i=1}^n \hat{c}_i = O(n) $$
정책 변화상환 결과해석
확장비율 r > 1 상수append 상환 O(1)기하급수 확장으로 재할당 횟수 로그 수준
확장비율 r = 1 + 1/k 형태(거의 선형)상환 비용 증가 가능복사 누적이 선형보다 크게 증가할 수 있음
축소 정책까지 포함thrashing 가능성 검토 필요히스테리시스(예: 1/4 이하에서 축소) 필요
Part 1.3 · Hash Table Math

해시 테이블 이론: 로드 팩터와 기대 비용

딕셔너리의 평균 O(1)은 “무조건”이 아니라 해시 함수 분산성과 적절한 리사이즈 정책에 의존합니다.

핵심은 균등 해싱 가정입니다. 키가 버킷에 균일하게 분포된다는 가정이 깨지면 탐색 길이가 길어지고, 평균 O(1) 근사도 약해집니다. 따라서 키 설계(불변성, 동등성, 해시 일관성)는 단순 구현 문제가 아니라 성능 보장의 전제 조건입니다.

$$ \alpha=\frac{n}{m},\quad \mathbb{E}[\text{lookup}] \approx O(1+\alpha) $$
상태현상영향
α가 낮음충돌 감소조회/삽입이 안정적으로 빠름
α가 높음충돌 증가탐사 길이 증가, 캐시 비효율
리사이즈 발생재해싱 비용단기 스파이크, 장기 평균은 유지

“평균 O(1)” 표현은 가정(해시 분산/적대적 입력)을 함께 명시해야 의미가 정확해집니다.

Part 1.4 · CPython Memory Model

CPython 객체 모델과 메모리 오버헤드

파이썬 컨테이너는 원시 값 배열이 아니라 객체 포인터 배열을 저장합니다. 따라서 “원소 수”뿐 아니라 “원소 객체 자체의 오버헤드”를 함께 고려해야 메모리 사용량을 정확히 예측할 수 있습니다.

메모리 해석 단위는 “값”보다 “객체 그래프”가 적합합니다. 예를 들어 같은 정수 10,000개라도 list[int], array, ndarray는 저장 모델이 달라 캐시 지역성(locality)과 순회 처리량이 크게 달라집니다. 자료구조 선택은 시간복잡도뿐 아니라 메모리 계층 구조(L1/L2/cache miss)와도 직접 연결됩니다.

$$ M_{\text{total}} \approx M_{\text{container-header}} + M_{\text{pointer-array}} + \sum_{i=1}^{n}\left(M_{\text{obj-header},i}+M_{\text{payload},i}\right) $$

list는 값을 직접 저장하지 않고 객체 참조를 저장합니다. 대규모 데이터에서는 이 계층 구조가 메모리와 캐시 효율을 결정합니다.

import sys

nums: list[int] = list(range(10_000))
tu: tuple[int, ...] = tuple(range(10_000))
st: set[int] = set(range(10_000))

print("list bytes :", sys.getsizeof(nums))
print("tuple bytes:", sys.getsizeof(tu))
print("set bytes  :", sys.getsizeof(st))
print("one int    :", sys.getsizeof(0))
구조메모리 특성설계 시사점
list연속 포인터 배열 + 여유 용량순차 접근/끝 삽입 유리
dict/set해시 슬롯 + 엔트리 오버헤드 큼속도는 빠르나 메모리 비용 큼
tuple불변, 상대적으로 컴팩트키/레코드 표현에 유리

수치 집약 루프는 Python 객체 컨테이너(list/dict/set) 안에서 직접 연산하기보다, 배열/텐서 표현으로 일찍 변환해 연산 경로를 분리하는 편이 유리합니다.

List = mutable sequence of object references

a: list[int] = [10, 20, 30, 40]
print(a[0], a[-1])
a[1] = 25
del a[2]

리스트는 순서가 있는 가변 시퀀스이므로, “위치 기반 접근”이 핵심입니다. 수학적으로는 함수 f: {0,...,n-1} → ObjectRef 로 모델링할 수 있고, 인덱스 연산은 f(i) 조회에 해당합니다.

$$ \text{update at }i:\quad f'(j)= \begin{cases} x & (j=i) \\ f(j) & (j\neq i) \end{cases} $$

리스트를 “작은 배열”로 보는 관점이 중요합니다. 앞쪽 삽입/삭제는 내부 시프트를 유발하고, 슬라이싱은 복사를 유발합니다. 즉, 리스트는 범용 컨테이너지만 모든 연산에 균등하게 좋은 자료구조는 아니며, 연산 패턴(앞/뒤/중간, 조회/갱신 비율)에 따라 선택되어야 합니다. 시퀀스 배치 구성, 길이별 버킷팅, 샘플 셔플 버퍼처럼 순서 중심 연산에서는 list가 기준 구조가 됩니다.

기본 성질

  • 음수 인덱스: 뒤에서 접근
  • 슬라이싱은 새 리스트 생성
  • append와 extend 의미 차이

자주 생기는 오류

  • [[0]*3]*4의 참조 공유 버그
  • remove 반복 중 인덱스 꼬임
  • pop(0)를 큐처럼 사용 (비효율)

Part 2.1 · Internal Representation & Cost Model

동적 배열로서의 list: 저장 모델과 비용

리스트는 C 레벨에서 동적 배열로 구현되며, 배열에는 객체 자체가 아니라 객체 참조가 저장됩니다.

$$ T_{\text{insert at }i}(n)=\Theta(n-i),\qquad T_{\text{delete at }i}(n)=\Theta(n-i) $$

이 설계 덕분에 리스트는 이질적인 타입을 한 컨테이너에 담을 수 있습니다. 반면 참조 간접(indirection)과 객체 헤더 오버헤드가 존재하므로, 대규모 수치 계산에서는 메모리 지역성(locality)과 캐시 효율이 병목이 될 수 있습니다. 따라서 수치 집약 작업에서는 배열 라이브러리와의 역할 분리가 필요합니다.

$$\text{amortized append} = O(1)\quad(\text{occasional resize} + \text{frequent constant-time writes})$$
연산복잡도설명
a[i]O(1)오프셋 계산으로 즉시 접근
append(x)Amortized O(1)오버할당으로 재할당 빈도 감소
insert(0,x)O(n)전체 시프트
pop(0)O(n)앞 원소 제거 후 당김
sort()O(n log n)Timsort(부분 정렬 데이터에 강함)

for step in range(1000):
    batch_ids: list[int] = [step * 32 + i for i in range(32)]
    batches.append(batch_ids)      # amortized O(1)

# 앞 삽입/삭제가 필요하면 deque로 분리하는 것이 안전
# (예: 실시간 샘플 큐, window popleft)
print(len(batches), batches[0][:3], batches[-1][-3:])

Part 2.2 · List API Semantics

Core Operations and In-Place Semantics

list는 단순 함수 모음이 아니라 상태 전이 시스템입니다. 각 메서드는 “상태를 바꾸는가(in-place)”, “새 객체를 반환하는가”, “비용이 어디서 발생하는가”를 함께 봐야 합니다.

x: list[int] = [3, 1, 2]
y = x.sort()          # in-place
z = sorted(x)         # new list
print(x, y, z)        # [1,2,3], None, [1,2,3]

a: list[int] = [1, 2]
a.append([3, 4])      # [1,2,[3,4]]
b: list[int] = [1, 2]
b.extend([3, 4])      # [1,2,3,4]
print(a, b)
기능군메서드반환핵심 의미/비용
구조 변경append, extend, insert, pop, remove, clear대부분 Nonein-place mutation, 위치에 따라 O(1)~O(n)
조회index, countint선형 탐색 O(n), 실패 시 ValueError 가능
복사/변환copy, list(a), a[:]new list얕은 복사(shallow), 중첩 원소 참조는 공유 가능
순서 조작reverse, sort, sortedsort/reverse=Nonesort는 stable + in-place, sorted는 새 리스트

연산마다 `state mutation 여부`, `반환 객체`, `복잡도`를 같이 표기하면 코드 리뷰에서 대부분의 list 버그를 사전에 제거할 수 있습니다.

Part 2.3 · Indexing & Slicing

Indexing & Slice Semantics: 경계조건, clipping, 상태변화

단일 인덱스와 슬라이스는 실패 동작이 다릅니다. 단일 인덱스는 범위를 벗어나면 예외가 나지만, 슬라이스는 범위를 자동 clipping해 빈 리스트를 포함한 결과를 돌려줄 수 있습니다.

arr: list[int] = [0, 1, 2, 3, 4, 5, 6]
print(arr[100:200])   # []  (slice clipping)
# print(arr[100])     # IndexError

print(arr[1:6:2])     # [1, 3, 5]
print(arr[::-1])      # 역순 복사

arr[2:5] = [20, 30]   # 길이 변경 가능
del arr[1:3]          # arr[1:3] = [] 와 동치
print(arr)
$$ a[i:j:s]\ \text{creates a new list},\quad \text{time/space}=\Theta(k)\ \text{where }k=\left\lceil\frac{j-i}{s}\right\rceil $$
주제정확한 의미버그 포인트
단일 인덱스a[i]는 하나의 원소 접근범위 밖이면 IndexError
슬라이스a[i:j]는 반열린 구간 [i, j)범위 밖이어도 clipping되어 예외가 안 날 수 있음
extended slicea[i:j:s]는 stride 선택s < 0일 때 방향 해석 실수 빈번
슬라이스 대입구간 교체로 길이 변화 가능같은 객체 alias가 있으면 외부 상태도 변함
삭제 연산del a[i:j] == a[i:j]=[]인덱스 이동으로 후속 루프 로직 깨짐

slice는 view가 아니라 새 list(outer container)입니다. 다만 내부 원소가 mutable이면 참조 공유가 남습니다.

Part 2.3.1 · Slice Rulebook

Slice semantics를 규칙으로 외우기

슬라이스는 “부분 보기”가 아니라 “규칙 기반 복사/재조합”입니다. 단일 인덱스와 슬라이스의 실패 정책을 분리해서 기억하면 경계 버그가 크게 줄어듭니다.

규칙의미실패/주의
a[i]단일 원소 접근범위를 벗어나면 IndexError
a[i:j]반열린 구간 복사 [i, j)범위를 벗어나도 clipping
a[i:j:s]stride 복사s<0이면 역방향, s=0은 오류
a[::-1]역순 새 리스트 생성원본은 그대로 유지
a[i:j] = t구간 교체길이 변화 가능
a[i:j:s] = t확장 슬라이스 교체s!=1이면 길이 일치 필요
del a[i:j]구간 삭제후속 인덱스가 당겨짐

print(arr[100:200])   # []
print(arr[5:1:-2])    # [5, 3]

arr[2:5] = [20, 30]
print(arr)            # [0, 1, 20, 30, 5, 6]

단일 인덱스는 strict, 슬라이스는 tolerant(clipping)입니다.

Part 2.4 · Aliasing, Repetition, and Shallow Copy

왜 [[0]*3]*4가 깨지는가: 복사가 아니라 참조 반복

list 의미론에서 가장 중요한 함정은 참조 공유(aliasing)입니다. b=a, a[:], list(a), deepcopy(a)는 겉보기는 비슷하지만 공유 수준이 다릅니다.

반복 생성(aliasing) 버그

good = [[0] * 3 for _ in range(4)]

print(good) # 첫 행만 변경

Part 2.9 · Deque for Queue Workloads

deque: 양끝 연산 중심 ADT 구현

deque는 queue/deque ADT를 구현할 때 list보다 의미와 비용이 모두 명확합니다. 핵심은 “앞삭제/앞삽입이 주연산인가”입니다.

핵심 연산

from collections import deque

q: deque[int] = deque([1, 2, 3])
q.appendleft(0)
q.append(4)
print(q.popleft(), q.pop())   # 0, 4

w: deque[int] = deque(maxlen=3)
for x in [10, 11, 12, 13]:
    w.append(x)
print(list(w))                # [11, 12, 13]

선택 기준

  • 앞삭제/앞삽입 빈도가 높으면 deque
  • 랜덤 인덱싱/중간 접근이 핵심이면 list
  • maxlen은 고정 크기 최근성 버퍼에 유용

Part 2.10 · Heap-Based Priority Queue

heap는 전체 정렬이 아니라 최소 원소 접근을 빠르게 하는 자료구조입니다. 우선순위 갱신/임의 삭제가 빈번하면 lazy deletion 또는 재삽입 전략이 필요합니다.

import heapq
from dataclasses import dataclass, field

@dataclass(order=True)
class Job:
    priority: int
    seq: int
    name: str = field(compare=False)

h: list[Job] = []

Part 2.11 · Ordered Array with Bisect

Bisect: search는 O(log n), insertion은 O(n)

bisect는 정렬 배열에서 경계를 빠르게 찾는 도구입니다. map 대체재가 아니라 ordered array discipline이며, 삽입 시 시프트 비용 O(n)을 항상 함께 계산해야 합니다.

import bisect

print(bisect.bisect_right(arr, 20))  # 3
bisect.insort(arr, 35)
print(arr)
# bisect의 검색/삽입 경향 시간 측정

setup_insert = "import bisect; base = list(range(100_000)); x = 54_321"

t_insert = timeit.timeit("arr = base.copy(); bisect.insort(arr, x)", setup=setup_insert, number=1_000)

딕셔너리 기본: 조회, 추가, 삭제, 안전 조회

딕셔너리는 키를 통해 접근하므로 인덱스 개념이 없습니다. 따라서 설계 관점에서 가장 중요한 질문은 “키가 없을 때 무엇이 정답인가?”입니다. 일부 파이프라인은 결측을 허용해야 하므로 기본값 정책이 필요하고, 일부 내부 프로토콜은 결측 자체를 버그로 간주해야 하므로 예외를 유지해야 합니다. 예를 들어 샘플 메타데이터, 체크포인트 상태, 옵티마이저 하이퍼파라미터는 키 누락 시 처리 정책이 서로 다릅니다.

$$ \text{lookup}(k)= \begin{cases} v & (k\in K)\\ \bot & (k\notin K,\ \text{with default/get})\\ \text{KeyError} & (k\notin K,\ d[k]\ \text{access}) \end{cases} $$
메서드의미예외/반환
d[k]직접 조회없으면 KeyError
d.get(k,default)안전 조회없으면 default
d.pop(k,default)조회 + 삭제없으면 default
k in d키 존재 검사bool

Python 3.10에서는 일반 dict도 삽입 순서를 보존합니다. OrderedDict는 재배치/양끝 pop 등 order-sensitive 연산이 필요할 때만 선택합니다.

Part 3.1 · Dict Internals

딕셔너리 내부: 해시, 충돌, 키 설계

딕셔너리는 해시 테이블입니다. 핵심은 좋은 키와 충돌 관리입니다.

평균 O(1)만 외우면 부족합니다. 충돌이 늘어나는 데이터 분포, 해시 일관성이 깨진 사용자 정의 객체, 과도한 리사이즈가 동시에 발생하면 tail latency가 크게 흔들립니다. 따라서 키 설계 단계에서 불변성·동등성·해시 일관성을 명시하고, 조회 실패 정책을 함께 문서화해야 합니다.

$$ k_1 = k_2 \Rightarrow h(k_1)=h(k_2)\quad\text{(required)} $$ $$ h(k_1)=h(k_2) \not\Rightarrow k_1=k_2\quad\text{(collision allowed)} $$

딕셔너리 성능은 키 설계 품질에 크게 의존합니다. 해시 충돌·조회 정책·키 불변성은 반드시 함께 설계해야 합니다.

# hashable 조건: 불변 + __hash__ 구현
print(hash((1, 2, 3)))
# print(hash([1, 2, 3]))  # TypeError

# 사용자 정의 키
from dataclasses import dataclass

@dataclass(frozen=True)
class Node:
    id: int
    label: str

d: dict[Node, float] = {Node(1, "A"): 3.14}
print(d[Node(1, "A")])
# 실험 캐시 키 설계 예시 (튜플 키 권장)
metrics_cache: dict[tuple[str, int, float], dict[str, float]] = {}
cfg_key: tuple[str, int, float] = ("bert-base", 32, 3e-5)
metrics_cache[cfg_key] = {"val_f1": 0.913, "val_loss": 0.224}

# 키가 list면 TypeError, 가변 구조라 재현성 있는 조회가 어렵다.
print(metrics_cache[cfg_key]["val_f1"])
리스크증상예방 방법
가변 객체 키 사용삽입 후 조회 실패/불안정키 타입은 불변 객체로 제한
__eq__/__hash__ 불일치동등 객체가 다른 슬롯에 저장동등성 계약을 테스트로 강제
적대적 충돌 분포조회 지연 급증키 정규화 + 분포 점검 벤치마크
누락 키 처리 정책 부재간헐적 KeyError 또는 silent defaultd[k]/get 사용 정책 분리
$$ \text{Key design rule: immutable key } \land \left(a=b \Rightarrow h(a)=h(b)\right) $$ \(\texttt{\_\_eq\_\_}\)와 \(\texttt{\_\_hash\_\_}\)는 항상 일관되게 설계

주의: 가변 객체를 키로 사용하거나, hash/eq 불일치가 있으면 치명적 버그가 발생합니다.

Part 3.5.2 · Missing-Key Policy

defaultdict vs setdefault

setdefault는 기본 dict에서 국소 패턴을 짧게 쓰는 도구이고, defaultdict는 자료구조 수준에서 누적/그룹화 의도를 선언하는 선택입니다.

from collections import defaultdict

pairs: list[tuple[str, int]] = [("train", 128), ("train", 256), ("val", 64)]

g1: dict[str, list[int]] = {}
for k, v in pairs:
    g1.setdefault(k, []).append(v)

g2: defaultdict[str, list[int]] = defaultdict(list)
for k, v in pairs:
    g2[k].append(v)

print(g1)
print(dict(g2))
선택지장점주의
setdefault의존성 추가 없이 기존 dict에서 즉시 사용루프가 길어지면 의도가 묻히기 쉬움
defaultdict누적/그룹화 정책이 타입 선언에 드러남키 누락을 예외로 감지하는 경로에는 부적합

정리: setdefault는 패턴, defaultdict는 구조 선택입니다.

Part 3.5.3 · Counter as Multiset

Counter는 “빈도 dict”가 아니라 multiset이다

Counter의 장점은 기본값 0과 연산 의미입니다. 단순 count dict보다 intent가 분명하고, top-k/비교/차집합 작업에서 코드가 짧고 안전해집니다.

from collections import Counter

tokens: list[str] = ["a", "b", "a", "c", "b", "a"]
cnt = Counter(tokens)
print(cnt)                 # Counter({'a': 3, 'b': 2, 'c': 1})
print(cnt.most_common(2))  # [('a', 3), ('b', 2)]
워크로드일반 dictCounter
빈도 누적초기화 분기 필요0 기본값으로 즉시 누적
top-k 빈도별도 정렬 코드 필요most_common(k) 지원
multiset 연산직접 구현 필요+, -, &, | 의미 제공

빈도 중심 workload에서는 dict보다 Counter가 설계 의도를 더 정확히 드러냅니다.

Part 3.5.4 · ChainMap for Layered Config

ChainMap: merge 결과가 아니라 우선순위가 있는 조회 경로

ChainMap은 여러 dict를 복사 병합하지 않고 겹친 조회 뷰를 만듭니다. 설정 우선순위를 코드 구조로 남길 수 있어 실험 환경 관리에 유용합니다.

from collections import ChainMap

defaults = {"batch_size": 32, "lr": 1e-3}
project  = {"lr": 5e-4}

from dataclasses import dataclass, field
from typing import Generic, TypeVar

T = TypeVar("T")

class Stack(Generic[T]):
    def __init__(self) -> None:
        self._data: list[T] = []

    def push(self, x: T) -> None:
        self._data.append(x)

    def pop(self) -> T:
        if not self._data:
            raise IndexError("pop from empty stack")
        return self._data.pop()

Part 5 · Complexity & Validation Lab

실험으로 검증하는 성능: 이론 vs 실제

같은 O-표기라도 데이터 크기와 연산 위치가 다르면 체감 속도는 크게 달라집니다. 아래 실험은 “앞쪽 삭제”라는 특정 연산이 구조 선택에 얼마나 큰 차이를 만드는지 보여줍니다.

성능 해석에서는 단일 평균값보다 입력 분포 변화에 대한 안정성을 봐야 합니다. 실험은 분포(최소/최대/분산)와 함께 보고하고, 코드 경로(연산 위치/캐시 워밍/객체 생성량)도 함께 기록해야 해석 오류를 줄일 수 있습니다.

import time
from collections import deque

N: int = 80_000
arr: list[int] = list(range(N))
q: deque[int] = deque(range(N))

start = time.perf_counter()
for _ in range(15_000):
    arr.pop(0)

절대 시간값보다 연산 패턴별 상대 비용 차이를 읽는 것이 중요합니다. 앞삭제가 지배적이면 list 대신 deque가 구조적으로 유리합니다.

$$ \text{Observed time} \approx f(n)\cdot c_{\text{impl}}\cdot c_{\text{hardware}} $$ Big-O는 증가율을 설명하고, 실제 성능은 상수항/캐시/런타임 구현체 영향을 함께 측정해야 합니다.
실험 결과해석

벤치마크는 단일 평균값 대신 입력 크기 sweep·워크로드 혼합·warm-up 분리·메모리·tail latency를 함께 보고해야 합니다. 이 구성을 사용하면 이론 복잡도와 실측 곡선의 차이를 정량적으로 해석할 수 있습니다.

import timeit
import statistics as st
from collections import deque

sizes = [2_000, 10_000, 50_000]
result: dict[int, dict[str, float]] = {}

for n in sizes:
    setup = f"from collections import deque; arr=list(range({n})); q=deque(range({n}))"
    _ = timeit.repeat("arr[-1]", setup=setup, repeat=3, number=5000)  # warm-up

    t_list = timeit.repeat("x=arr.pop(0); arr.insert(0,x)", setup=setup, repeat=12, number=1200)
    t_deque = timeit.repeat("x=q.popleft(); q.appendleft(x)", setup=setup, repeat=12, number=1200)
    ratio = st.mean(t_list) / st.mean(t_deque)
    result[n] = {"list_mean": st.mean(t_list), "deque_mean": st.mean(t_deque), "speedup": ratio}

print(result)

Part 5.2 · Complexity Curve Fitting

측정 곡선을 복잡도 가설에 맞춰 해석하기

벤치마크 결과는 표로 끝내지 않고, 성장률 가설(선형/로그선형/제곱)을 데이터에 피팅해 해석합니다. 이 과정으로 “속도 차이”를 “차수 차이”로 정량화할 수 있습니다.

$$ \log t_i = a + b\log n_i + \varepsilon_i,\qquad \hat{b}=\frac{\operatorname{Cov}(\log n,\log t)}{\operatorname{Var}(\log n)} $$ \(\hat{b}\)는 경험적 복잡도 지수(기울기)입니다.
$$ \text{if }m=\rho n:\quad T_{\text{list front-delete}}\propto n^2,\quad T_{\text{deque left-delete}}\propto n $$ 실험에서 \(\hat{b}_{list}\approx 2,\ \hat{b}_{deque}\approx 1\)이면 이론-실측이 정합적입니다.

설계 관점

  • 함수 경계에서 입력 변경 여부를 명시
  • 공유 상태가 필요하면 문서화 + 테스트 강화

Part 6.4 · Runtime Memory Management

파이썬 메모리 관리: 참조 카운트 + 사이클 GC

파이썬(CPython)은 기본적으로 참조 카운팅(reference counting)으로 객체 생명주기를 관리하고, 순환 참조는 세대별 가비지 컬렉터가 처리합니다.

메모리 점검 항목은 누수 징후, cycle 회수, 관측 포인트입니다.

이중 구조를 이해하면 메모리 누수 분석이 쉬워집니다. 단순 객체는 참조 카운트가 0이 되는 즉시 해제되지만, 순환 참조 구조는 카운트만으로 해제되지 않습니다. 따라서 장기 실행 프로세스에서는 cycle을 만들기 쉬운 객체 그래프를 점검하고, 필요 시 약한 참조 또는 명시적 해제 전략을 고려해야 합니다.

$$ \text{alive}(o)\iff \operatorname{refcount}(o)>0,\qquad \text{cycle garbage requires tracing GC} $$ 
import gc

class Node:
    def __init__(self) -> None:
        self.ref: Node | None = None

a: Node = Node()
b: Node = Node()
a.ref = b; b.ref = a   # cycle

del a; del b
print("before collect:", gc.get_count())
gc.collect()
print("after collect:", gc.get_count())
# 딥러닝 루프에서 그래프/텐서 참조 누수 방지 패턴
loss_history: list[float] = []
for step_loss in [0.92, 0.71, 0.55]:
    # 실제 torch 코드에서는 loss.item() 값만 보관하는 습관이 중요
    loss_history.append(float(step_loss))
print(loss_history)

메모리 관점에서 list/dict/set은 객체 참조를 저장하므로, 원소 수가 늘면 컨테이너 오버헤드 + 객체 오버헤드가 동시에 증가합니다. 대규모 파이프라인에서는 구조 선택뿐 아니라 데이터 표현(정수/문자열/객체)을 함께 설계해야 합니다.

Part 6.5 · Logical Correctness

논리적 정확성: 불변식(invariant)과 경계조건

논리적 정확성 검증은 불변식을 명시하고 각 연산이 불변식을 보존하는지 점검하는 과정입니다. 동작 사례뿐 아니라 보존 조건을 함께 확인해야 정확성이 유지됩니다.

예: Stack 불변식

  • size == len(data)
  • top은 마지막 삽입 원소
  • push/pop 후에도 LIFO 유지

검증 항목

  • 빈 구조에서 연산 예외 처리
  • 중복 키/누락 키 시 정책 일관성
  • 얕은 복사/깊은 복사 부작용 여부
$$ \forall t\ge 0:\ I(S_t)\ \land\ \text{op preserves }I\ \Rightarrow\ I(S_{t+1}) $$ 
def push(stack: list[int], x: int) -> None:
    before = len(stack)
    stack.append(x)
    assert len(stack) == before + 1   # invariant check

def pop_safe(stack: list[int]) -> int | None:
    if not stack:
        return None
    return stack.pop()

s: list[int] = []
push(s, 10); push(s, 20)
print(pop_safe(s), pop_safe(s), pop_safe(s))
Part 6.6 · Concurrency & Immutability

동시성과 불변 데이터: 안전한 공유 상태 설계

병렬/비동기 환경에서는 자료구조 선택이 곧 동시성 안전성입니다. 공유 상태를 줄이고 불변 구조를 우선하면 race condition 가능성이 크게 감소합니다. 데이터 로더 워커, 비동기 전처리 태스크, 추론 요청 큐를 동시에 사용할수록 이 원칙의 효과가 커집니다.

동시성 안전성은 락 하나로 끝나지 않습니다. 공유 상태를 줄이고 immutable 스냅샷/메시지 전달 중심으로 구조를 바꾸는 것이 근본 해법입니다.

자료구조 구현에서는 입력 몇 개로 끝나는 예제 테스트만으로는 불충분합니다. 랜덤/속성(property) 기반 테스트를 통해 불변식 위반 사례를 체계적으로 찾는 접근이 필요합니다. 특히 시드 고정 여부, 입력 순서 섞기, 배치 경계값(0/1/최대 길이) 검증을 함께 두어야 재현 가능한 결함 분석이 가능합니다.
